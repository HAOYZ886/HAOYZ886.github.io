相关技术栈

RCU 是多读少写的无锁机制，读写操作能并行执行，但多个写操作需要使用互斥锁等同步机制。写操作会获取一个内存空间填入新数据，来作为当前变量的新版本，并发出旧版本回收的请求。而且需要等待还在旧版本变量的 readers 执行完成后，才能释放旧版本资源。
这个等待就是宽限期（grace-period），是指若干个资源回收前的“等待期”，等待 readers 访问旧版本数据完成后，回收旧版本数据。另外，静默状态（quiescent states）是宽限期结束的关键。在常规 RCU（非抢占 RCU），当所有 CPU 都进入静默状态后，表示当前宽限期已结束，一个进程在内核层被调度后，就表示执行该进程的 CPU 进入静默状态，即该进程的上下文已被切换，隐式表示读操作的完成。其它的还有，执行用户层进程的 CPU ，处于 IDLE 的 CPU ，这些都默认进入静默状态。这是因为 RCU 仅在内核层生效，而且 IDLE 状态下的 CPU 无任何 RCU 读操作。

问题解析和排查

结合以上报错信息，表示当前正在进行宽限期，但是最上层的 qsmask ，即 root->qsmask 已经是 0x0 ，即整个系统的静默状态已结束，gp_state 仍处于 RCU_GP_WAIT_FQS ，gp_flags 为 RCU_GP_FLAG_FQS 。内核线程 rcu_preempt 等待（IDLE）当前宽限期的强制静默状态，等待前 gp_flags 不为 RCU_GP_FLAG_FQS 。而现在该线程的运行状态为 TASK_RUNNING ，但是 CPU1 执行不了它，但还在调度队列中。宽限期开始的时间戳和现在正在进行的时间戳相差 1880104 jiffies ，目前调度定时器的频率为 250Hz ，换算过来就是 1880104/250=7520.416秒，7520.416秒/60=125.34分，即该内核线程超过 2 个小时没有得到执行。而且随着时间的推移，一直堆积着旧版本的数据，使得系统的空闲内存越来越少，触发 OOM 的概率越来越大。换句话说，就是当前宽限期的阶段是在等待所有强制静默状态完成，但是对该阶段处理的内核线程一直得不到执行，就出现卡死，出现诸如以上的报错。
结合以下来自linux的文档：

kthread starved for 23807 jiffies! g7075 f0x0 RCU_GP_WAIT_FQS(5) ->state=0x1 ->cpu=5
Starving the grace-period kthreads of CPU time can of course resultin RCU CPU stall warnings even when all CPUs and tasks have passedthrough the required quiescent states. The “g” number shows the currentgrace-period sequence number, the “f” precedes the ->gp_flags commandto the grace-period kthread, the “RCU_GP_WAIT_FQS” indicates that thekthread is waiting for a short timeout, the “state” precedes value of thetask_struct ->state field, and the “cpu” indicates that the grace-periodkthread last ran on CPU 5.

目前的出现该问题的原因：

  • 中断风暴：某个硬件设备（网卡、磁盘控制器等）产生大量中断，导致 CPU 大部分时间在处理硬中断（Hard IRQ）或软中断（SoftIRQ），无暇调度普通内核线程（如 rcu_preempt）。
  • 实时任务抢占：如果有 SCHED_FIFO 或 SCHED_RR 的实时线程占用 CPU 且不释放，普通优先级的 rcu_preempt 线程（通常是 SCHED_NORMAL）将永远无法运行。
  • 自旋锁持有过长：虽然 qsmask=0，但如果某个 CPU 在持有全局自旋锁的情况下关闭了中断，或者在内核临界区停留时间过长，会阻止调度器切换上下文。

另外，应用程序还使用了 AMP rpmsg 通信，有概率出现 rx/tx buffer 有概率满，从而因获取不到空闲的 buffer 而出现报错，而且 rpmsg 通信频率很频繁。使用脚本查看整个系统的进程调度策略和优先级，发现无任何应用程序使用 FIFO 实时策略，仅有几个 FIFO 实时策略的 IRQ 中断线程，但是对应的中断触发频率较低。
根据以上原因集合，极大概率是中断风暴导致的 rcu_preempt 饥饿。现在的做法是需要密切监控系统的中断，使用 top 命令，查看 %si (softirq) 或 %hi (hardirq) 是否很高；使用 cat /proc/interrupt 查看哪些中断触发频繁。
