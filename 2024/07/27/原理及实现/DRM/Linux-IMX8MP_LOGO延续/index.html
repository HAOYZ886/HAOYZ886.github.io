Linux - IMX8MP LOGO延续

Posted on 2024-07-27 Edited on 2025-01-28 In 原理及实现 Word count in article: 28k Reading time ≈ 1:43

前言

这篇文章很意识流，想到什么就说什么，但能通过此学习时钟、电源和 Pinctrl 。

目的

MIPI-DSI 屏和 LVDS 屏的 LOGO 延续从 U-Boot 阶段延续到内核阶段，直到 Weston 桌面启动。

问题

在 U-Boot 期间，在 MIPI-DSI 屏和 LVDS 屏生成 LOGO 。从 U-Boot 切换到 Kernel 后，屏幕先出现花屏，然后被像素图像刷掉，再接着背光被关掉。而且出现花屏的情况仅在内存容量为 2G 的 IMX8MP 出现。

以下是花屏的情景：

另外，在内存容量为 4G 的 IMX8MP 中，出现 LVDS 屏的 LOGO 一部分被刷掉。然后这两个屏像素图像全部被刷掉，再接着背光消失。

问题梳理

分前后顺序梳理，把这个问题拆分多个小问题：

  1. 2G IMX8MP 的 MIPI-DSI 和 LVDS 屏都出现花屏，4G IMX8MP的 LVDS 屏的部分 LOGO 被刷走
  2. MIPI-DSI 和 LVDS 屏的像素被刷掉
  3. MIPI-DSI 和 LVDS 屏的背光被关闭

先从 DRM 下手

由于 IMX8MP 公版，即初始版本，是没有 LOGO 延续。然后发现飞凌 IMX8MP 有 LOGO 延续。由于我并没有他们的 GIT 仓库，目前只找到了在 DRM 框架内根据 LOGO 延续的修改。但是我把这些修改复制到我们公司的 IMX8MP 时，发现并没有发挥作用。
在研究 DRM 框架期间，发现 plane、CRTC、encoder 和 connector 之间的联系是相对灵活的，并没有直接用指针去指定谁是属于谁的。它们之间的联系是先通过设备树建立，而且依赖于根据设备树衍生而来的 aggregate-component 机制建立。

在 pm_runtime_barrier() ：

  • 第一步就是执行 pm_runtime_get_noresume() ，该函数功能是把当前设备 power.usage_count 加一，目的是防止有其它执行流调用 rpm_suspend() 来关闭当前设备所属的电源域。power.usage_count 大于 0 表示当前设备正在使用该电源域。在 rpm_suspend() ，会调用 rpm_check_suspend_allowed() ，它里面有一个条件判断：如果当前设备 power.usage_count 不为零，则不关闭该电源域，直接返回。
  • 第二步是处理当前设备的 power.request_pending ，如果该字段为 True ，则设为 False ，并把放在工作队列内的 request 移除。但是如果 power.request 为 RPM_REQ_RESUME ，则立即开启当前设备所属的电源域。
  • 第三步，如果当前设备的 power.runtime_status 为 RPM_SUSPENDING 或 RPM_RESUMING ，则等待这些状态转变为非“进行中”状态后，再继续执行后续的操作。
  • 第四步，调用 pm_runtime_put_noidle() ，把当前设备 power.usage_count 减一 。表示设备的电源请求和进行中的状态已被处理完成。

为什么 __driver_probe_device() 需要 pm_runtime_barrier() ？
内核允许我们何时何地去新建 struct device ，去匹配 driver ，对驱动来说更是如此。而且一个 struct device 去绑定一个电源（从软件层面）也是容易，而且也没非要让一个 struct device 去与一个 driver 绑定，可以先绑定一个电源去使用。所以说，在一个 struct device 绑定一个 driver 期间，并不详细地知道在此之前 struct device 的状态是什么。所以为了执行结果不受这些与电源操作的因素影响，必须把这些电源请求和进行中的状态处理完成。

第 23 行，pm_request_idle() ，由于我们并不清楚设备和驱动绑定之后，设备什么时候被使用。所以绑定完成后，默认都把设备的状态设置为 idle 。而且这个操作是非同步的，即可以放到工作队列去执行，一段时间后再执行此操作。因为这个操作不是强制性的，而且频繁地给设备开关电源也是件不好的事。
但在此流程中，即设备与一个电源域绑定后，也会执行类似的操作，不同的是直接通过 PM Domain 关闭电源域，而非 Runtime PM 。此流程中的 pm_runtime_barrier() 和 pm_ pm_request_idle() 是仅给那些在执行此流程前，就已经有电源域的设备使用。所以一般来说，此前就已经有电源绑定的设备，对应的设备树节点内是没有 power-domains 等类似的属性（还是有可能性的，但是很小）。设备对应的设备树节点中有这些属性的，相关模块/子系统也不应在此匹配绑定对应驱动的流程之前，就提前绑定好电源域。

解决电源

现在来归纳总结。
回头看 GPC 的电源域初始化，可知此时所有的电源域状态都为 GENPD_STATE_OFF 。
MEDIA-BLK-CTRL 与对应驱动匹配并 ->probe() 期间，会调用 dev_pm_domain_attach() 把 power-domains = <&pgc_mediamix>; power-domain-names = "bus" 绑定电源域。那么绑定期间必然会开启该电源（状态为 GENPD_STATE_ON ），在对应的驱动的 ->probe() ，即 imx8m_blk_ctrl_probe() ，该函数内有 bc->bus_power_dev = genpd_dev_pm_attach_by_name(dev, "bus"); ，表示 bc->bus_power_dev 才是 pgc_mediamix 的真实消费者。在 genpd_dev_pm_attach_by_name() 会调用 genpd_queue_power_off_work() 把该电源域的关闭操作加入到工作队列中。也就是说 MEDIA-BLK-CTRL 的 platform device 通过 platform_probe() -> dev_pm_domain_attach() 开启了该电源域，但由于 platform device 的 power.runtime_status 为 RPM_SUSPENDED，所以很大概率 bc->bus_power_dev 把当前电源域给关了（剩下的小概率是除非执行 genpd_power_off_work_fn() 前，有设备去绑定并通过 Runtime PM 开启该电源）。
而且 MEDIA-BLK-CTRL 的 platform device 的初始化后续，即返回到 really_probe() 后会调用 dev->pm_domain->sync(dev); ，也就是说也把电源域关闭操作添加到工作队列，但是通过工作队列轮到它操作时，执行 genpd_power_off() 。有一个条件判断 (!genpd_status_on(genpd) || genpd->prepared_count > 0) ，会导致关闭电源域操作失败。意思就是已关闭了就不能再进行一次关闭操作了，所以该操作是“无效施法”，因为前面已经为该电源域执行过一次了。还有，返回到 __driver_probe_device() ，会调用 pm_request_idle() ，里面调用了 rpm_check_suspend_allowed() ，里面有一个条件判断 (dev->power.disable_depth > 0) 让 media-blk 的 platform device 无权去关闭该电源域，因为对应的驱动没有为它调用 pm_runtime_enable() ，而是把权限给了 bc->bus_power_dev ，即给 bc->bus_power_dev 调用了 pm_runtime_enable() 。
而且，最主要的是 LVDS 屏和 MEDIA-BLK-CTRL 用的是同一个电源域 mediamix ，所以这些操作也是导致 LOGO 延续失败的原因之一。

一个设备通过 Runtime PM 开启和关闭电源域

当一个设备与指定的电源域绑定之后，就可以开启和关闭电源域了吗？不，还差一步，就是需要调用 pm_runtime_enable() 给设备开启“权限”，这个权限就是：通过 Runtime PM 控制电源域。设备是被驱动控制的对象，不管有没有这个机制，驱动来开启/关闭电源域不是信手拈来吗？另外，对于设备驱动来说，驱动作者可以随意使用 pm_runtime_enable() ，rpm_resume|suspend() 也能在驱动内随意调用。那么这一机制防的是谁？防的是内核的各个子系统和模块，“命根子”还是掌握在自己手里的好啊。

解决电源

由于 lcdif 驱动有两个主要源文件，一是 lcdif-common.c ，另一个是在 DRM 目录内的 lcdif-crtc.c 。在 lcdif-common.c 内新增了一个 struct device ，它用于 DRM 框架，代表 CRTC 设备，新 struct device 的驱动在 lcdif-crtc.c ，且它的 parent 是描述真实物理设备 lcdif 的 platform device 。
在设备与驱动匹配期间，会调用 __driver_probe_device() 内的 pm_runtime_get_sync(dev->parent); 和 pm_runtime_put(dev->parent);。 pm_runtime_get_sync() 的核心就是 rpm_resume() ；pm_runtime_put() 的核心就是 rpm_suspend() 。上面讲过，当前设备开启|关闭电源时，如果设备对应的驱动有 ->runtime_resume|suspend() ，则调用它。而这个回调函数一般涉及到时钟操作和硬件操作等。
这两个操作会导致 LOGO 延续失败，所以需要在 lcdif-common.c 驱动的 imx_lcdifv3_runtime_resume|suspend() 做出改动：在内核初始化期间，暂时把它们的操作屏蔽。

当 LOGO 延续到 Kernel 启动完成后，开始按序启动 IMX-DRM-CORE 和 Weston 桌面。如果在此期间执行下列语句会导致死机。目前能追溯到的原因是调用重启 MIPI-DSI 的 PHY 芯片时卡住死机，相应的调用过程是：imx8m_blk_ctrl_power_off() -> regmap_clear_bits(bc->regmap, BLK_MIPI_RESET_DIV, data->mipi_phy_rst_mask) 。
由于对 MIPI-DHPY 不了解，死机的原因还不清楚，推测是可能和硬件的开启和重启操作有关，可能是开启 PHY 后，未达到一定时间后重启而死机。这个推测的由来是当 IMX-DRM-CORE 还不是 .ko 文件去延续 LOGO 时，MIPI-DSI 先被对应驱动绑定，期间会绑定 media-ctrl-clk 电源域并开启该电源域，期间会开启 MIPI-DSI 的PHY 芯片。该 sec_mipi_dsim_bind() 被调用时，是在所有的 LCDIF 和它们各自对应的屏幕接口（如 MIPI-DSI、LVDS、HDMI）匹配完成后才会被调用。这个开启和关闭的时间间隔较长。而 IMX-DRM-CORE 是 .ko 文件去延续 LOGO 时，由于延续 LOGO 阶段把 mediablk-mipi-dsi-1 的开启和关闭屏蔽了。执行 sec_mipi_dsim_bind() 的 pm_runtime_get_sync() 时才开启它。导致开启和关闭 PHY 芯片的时间间隔短而导致死机。
