从用户层出发，浅析 poll()

函数原型

函数 poll() 的功能是在 timeout 内监视 fds ，在此时间内在 fds 捕获用户指定的行为。

参数

  • struct pollfd *fds：记录将被监控的文件，其中不乏包含 普通文件、设备文件、管道文件和套接字。
  • nfds_t nfds：需要被监控的文件数量
  • int timeout：等待事件发生的超时时间（毫秒）。
    • -1：无限期等待。
    • 0：立即返回，不阻塞。
    • >0：等待指定的毫秒数。

      后面会讲解 poll 时间机制的设计。

细节要点

以下是 struct pollfd 的声明：

  • fd：需要监视的文件对应的描述符
  • events：需要监视的事件。
  • revents：在监视期间或结束后，存储相应的结果。

监控的事件类型：

  • POLLIN：数据可读。
  • POLLOUT：写数据不会导致阻塞。
  • POLLERR：错误条件。
  • POLLHUP：挂起条件。
  • POLLNVAL：非法请求。

例子：假设要监控的文件是一个字符设备文件，该文件对应的是开发板的一个按钮， 用户设置 events 为 POLLIN 。在监视期间，如果捕获到按钮被按下，相应的驱动 poll 需要返回 POLLIN 到 revents ，表示已捕获到用户想要的事件。

看 26 ~ 29 行，回头看解析和代码可知，如果计时数为 -1 时，表示一直阻塞直到捕获到 events 。因为上层代码是根据 -1 计时数，把 expires 表示为 NULL ，所以直接进程调度，而且当当前进程的状态设置为 TASK_INTERRUPTIBLE 后，就从就绪队列中移除了。

现在，针对计时数为 >0 或 -1 的情况：

  • 在阻塞期间被唤醒了，唤醒的原因有：被信号打断、捕获到 event 。
  1. 被信号打断：已知进程在阻塞，而且状态设置为 TASK_INTERRUPTIBLE ，发起信号的进程/线程会唤醒它。
  2. 捕获到 event ：event 是否被捕获，取决于文件对应的模块或子系统，如上文说的字符设备文件，它对应的驱动模块设置了一个中断处理函数，当按钮被按下时触发执行。该函数会从它给 poll 提供的 wait queue 唤醒进程（期间会调用 wait->func() ，即 wakepoll() ）。
