以上只是在“使用”层面去说明 class 的作用，在源码层面上讲，一个 class 对应一个子系统或模块。子系统或模块内有私有的描述符（内有 struct device）去表示一个真实的设备，或是一个物理设备的具体功能。而 struct device 可以表示一个真实的物理设备（如 platform device 子系统），可以是一个物理设备的一个功能（如 input 子系统的 input 设备），也可以表示一个逻辑设备（如 input 子系统的 event 设备）。

那么为什么传递一个字符串 10 之后，屏幕的背光就会改变？
每个在 class 集合下的设备是一个目录，目录内都有一组“文件”，这些“文件”都是由 class 集合定义。就像不同的公司，员工服装和工牌的样式都不一样的。这些“文件”的作用就是记录该设备的参数，又或者说这些设备的属性。它们也像普通文本文件一样有读写权限，而且还能想普通文本文件一样使用诸如 cat、head、tail 命令去读文件；用诸如 echo 命令去写文件。但是它们的值和普通文件是不一样的，根据不同的设备有不同的限制，比如 “/sys/class/backlight/backlight/brightness” ，该文件可读写，但是写的内容只能是在对应设备树节点 backlight 的属性 brightness-levels 规定的背光等级范围。如果超出范围，或不是数字的字符串，则会报错，比如：

那么我在这些设备的属性文件输入了不该输入的值，系统又是如何知道这些值就是非法的？
前文说过，一个子系统或模块对应一个 class ，而在 class 下的设备必须有它规定的属性文件。在 backlight 模块中，它有一个名为 “backlight” 的 class 描述符，同时它也定义了一组在它模块内的属性文件。每当在它的 class 下注册设备，这些设备都会被它要求生成这些属性文件。该属性文件被抽象出来，且由以下结构体描述：

  • attr: 内有描述属性文件的权限和名字
  • show：对应文件的读操作
  • store：对应文件的写操作
