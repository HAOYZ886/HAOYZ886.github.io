开启以上的宏后，u-boot trace 基本可用，对应的源码文件是 lib/trace.c 。该文件内有两个初始化函数，分别是：

当加上 __attribute__((no_instrument_function)) ，表示当前函数不会在开始和结束位置分别加上 __cyg_profile_func_enter() 和 __cyg_profile_func_exit() 。

函数 trace_early_init 被存放在 init_sequence_f ，而 trace_init 在：

in common/board_r.c

函数 trace_init 的作用是初始化 struct trace_hdr 结构体。该结构体最主要的字段是：

  1. 字段 call_accum ：指向一个数组，该数组的元素对应 u-boot 的函数调用次数，而这些元素和函数的对应关系，则和 System.map 内的项匹配。
  2. 字段 ftrace ：指向一个元素为 struct trace_call 的数组，用字段 ftrace_size 表示该数组的大小。该数组记录函数的调用过程。

上文说了，添加参数 -finstrument-functions 编译 uboot ，编译期间会给所有函数（除了已添加关键字 __attribute__((no_instrument_function)) 的函数）的开始和结束位置添加 __cyg_profile_func_enter() 和 __cyg_profile_func_exit() 。而这两个函数也是需要自己定义的，在 lib/trace.c 中，已定义了这两个函数。

如果不添加 __attribute__((no_instrument_function)) 会导致一直递归调用，无法正常进行。

当获取到一部分的调用过程后，可以调用函数 trace_list_functions() 查看各函数的调用次数，也可以调用函数 trace_list_calls 查看目前函数的调用过程。
而这些函数也被封装为命令，在 cmd/trace.c 。在 u-boot shell 或调用 run_command() ，输入 trace 即可查看和使用相关功能。

现在是要怎么查看它们了。
在 doc/README.trace 的章节《Writing Out Trace Data》中，作者建议用 tftp 把调用过程的相关值作为一个文件输出到本机的某一个目录内，但是这个方法可能因平台的不同而无法使用。其实有一个简单粗暴的方法，就是把调用过程的值使用 printf() 打印到屏幕，把值复制粘贴到一个空文件，这算是比较通用的方法了，缺点就是有点麻烦。

由于打印出来的数值还是函数地址，需要使用 System.map 把函数名和地址匹配才行，在 doc/README.trace 中，作者建议使用：

Options:

  • -m : Specify U-Boot map file
  • -p : Specifiy profile/trace file

Commands:

  • dump-ftrace: Write a text dump of the file in Linux ftrace format to stdout
