内存屏障的使用

Posted on 2024-06-19 Edited on 2024-11-16 In 原理及实现 Word count in article: 2.9k Reading time ≈ 10 mins.

理解

文中的保序是指 CPU 之间的保序，或 CPU 与 DMA 之间的保序，保序的操作就是禁止乱序执行和内存重排序。
这个保序像 CPU 之间的同步操作，如：

当 CPU0 执行到 flag = 1; 后，CPU1 才能执行 if (flag == 1) 。比如，不能把 flag = 1; 被重排到 if (flag == 1) 的后面；变量 a 的加载不能在 if (flag == 1) 之前（分支预测搞的鬼）。
综上所知，在内存屏障之前的指令要完成，然后才能执行屏障后面的指令。

当然，这个是程序员想要的结果，并不是预料到的结果。如果在弱序系统中，即 CPU 并不关心其它 CPU ，自顾自地去执行指令（尤其是为了效率，会使用乱序执行和内存重排序机制），上面的“想要的结果”是不可能执行成功的。就需要“人为干预”，这就是添加内存屏障，防止乱序执行和内存重排序。见以下 DMB 解释：

The Data Memory Barrier (DMB) prevents the reordering of specified explicit data accesses across the barrier instruction. All explicit data load or store instructions, which are executed by the PE in program order before the DMB, are observed by all Observers within a specified Shareability domain before the data accesses after the DMB in program order.

第一句话可理解为，像 store 和 load 等可重排序的指令禁止重排序。
然后第二句话，在 DMB 之前的 store 和 load 指令，都会被 Observers 监控。但是来了一个问题，就是谁来开这个监控，谁有这个主动权？
这个是需要使用场景支撑。根据上图所知，我是必须要在 if (flag == 1) 之前，把变量 flag 和 a 都赋值完成，但是 CPU1 也有可能先加载 a 变量，可以把 smp_wmb() 放在 a = 3; 和 flag = 1; 之间。在 flag = 1; 和 if (flag == 1) 之间也需要保序，则在它们之间放一个 smp_mb() 。但是 smp_mb() 给哪个 CPU 执行？可以把它放到 CPU1 ，因为防止 a 经过内存重排和分支预测被预先加载，但要把放在 a = 3; 和 flag = 1; 之间 smp_wmb() 去掉，因为 smp_mb() 是针对 load 和 store 指令内存屏障。或者 smp_wmb() 可以留着，但是要把 smp_mb() 放在 if (flag == 1) 后面。

它们的保序是通过各CPU的观察者 observer 监督来保证的。开启保序 dmb 后，观察者 observer 就监督保序下段代码的读写内存指令，尤其监督该指令的操作物理内存地址，针对各CPU对该物理内存地址读写的操作，保证该地址的读写顺序，即保序。

先搞清楚3个point：

  1. 谁要和谁保序：
    1. 谁主动使用内存屏障？
    2. 确定操作域的范围，inner share domain（CPU内的 cores）、outer share domain（CPU对DMA） 和 full system domain（CPU对CPU）。
  2. 在哪里保序：确定保序等级，如 dmb、dsb、isb 和 ldar
  3. 朝哪个方向保序：如 load、store 和 any

在第一段代码中的 smp_mb();

  1. Core0 要和 Core1 保序，即 ish
  2. smp_mb(); 的下面两条指令分别是：store 和 load ，即 dmb
  3. smp_mb(); 的下面两条指令两个方向都有，则 any
    所以 smp_mb() 实际是 dmb(ish)

当 Core0 执行 smp_mb(); 后，Core1 执行 if (hdmi->enable_late.skip)时，Core1 的观察者发现字段 skip 和 enabled 在被 Core0 分别写和读，则等 Core0 操作完成后再执行 Core1 分别对字段 skip 和 enabled 的读和写。
注意：dmb仅对 load 和 store 操作保序，且对 core0 和 1 都会访问的物理内存地址保序，如字段 skip 和 enabled。

内存屏障就是：在 SMP 系统，在同一段时间内，对同一内存地址访问的对象之间的保序。

为什么需要 3 个 point ？

  1. 为什么需要保序范围？
    范围越小，observer监视的范围就越小，使得系统仅针对一个小的范围保序，减少性能的浪费。就比如说 CPU 内的 cores 之间保序，就不需要全局的保序范围，仅需针对该 CPU 内的范围保序即可。
  2. 为什么需要保序等级？
    这个关乎于 CPU 的乱序执行，保序等级越大 CPU 的乱序执行就越少。就算编译器没有针对 CPU 乱序执行而对代码做执行顺序上的优化，CPU也是会乱序执行的，所以这个保序等级是限制 CPU 的乱序执行。
  3. 为什么需要保序方向？
    这个也和 CPU 的乱序执行有关。当然，虚拟寄存器也是有这个关系限制的（但不知道 ARM 有没有这个特性）。

SMP & ATOMIC OPERATION

1

A

static inline void tasklet_disable_nosync(struct tasklet_struct *t)
{
atomic_inc(&t->count);
smp_mb__after_atomic();
}
和
static inline void tasklet_enable(struct tasklet_struct *t)
{
smp_mb__before_atomic();
atomic_dec(&t->count);
}
中，smp_mb__after_atomic();和smp_mb__before_atomic();有什么讲究，回答前请认真思考

Q

在Linux内核中，tasklet_disable_nosync和tasklet_enable函数是用于控制tasklet（轻量级的底半部处理机制）的启用和禁用的。这两个函数通过操作tasklet_struct中的count成员来实现对tasklet的控制。count的增减通过原子操作来完成，以确保在多核处理器环境中的正确性和同步。

在这两个函数中，smp_mb__after_atomic();和smp_mb__before_atomic();是内存屏障（Memory Barriers）的使用，它们在多处理器系统中非常关键，用于确保内存操作的顺序性和可见性。下面详细解释这两个内存屏障的作用：

smp_mb__after_atomic();

smp_mb__after_atomic();是一个在原子操作之后使用的内存屏障。它确保了在此屏障之前的所有内存操作（包括紧邻其之前的原子操作）在多核心处理器环境中对其他处理器可见之前，不会与屏障之后的任何内存操作发生重排序。换句话说，它强制保证了原子增加操作atomic_inc(&t->count);完成后，之后的内存操作不会被提前执行。

smp_mb__before_atomic();

smp_mb__before_atomic();是一个在原子操作之前使用的内存屏障。它确保了在此屏障之后的所有内存操作（包括紧邻其之后的原子操作）在多核心处理器环境中对其他处理器可见之前，不会与屏障之前的任何内存操作发生重排序。这意味着在执行原子减少操作atomic_dec(&t->count);之前，之前的所有内存写入操作都已经完成，并且这些操作的效果对所有处理器都是可见的。

总结来说，smp_mb__after_atomic();和smp_mb__before_atomic();的使用是为了确保在多核处理器环境下，与tasklet控制相关的原子操作前后的内存访问顺序和可见性，从而避免潜在的数据竞争和不一致问题。这对于内核中的并发编程是非常重要的，因为它帮助维护了代码的正确性和稳定性。

2

A

我还是不懂，我想知道什么时候可以用smp_mb__before_atomic();，什么时候可以用smp_mb__after_atomic();

B

了解何时使用smp_mb__before_atomic();和smp_mb__after_atomic();，我们需要先理解它们各自的作用以及内存屏障在并发编程中的重要性。

  1. 何时使用smp_mb__before_atomic();
    当你需要确保一系列的内存写操作在一个接下来的原子操作之前对其他处理器可见时，应该使用smp_mb__before_atomic();。这种情况通常出现在你希望在执行原子操作改变某个状态之前，确保之前对共享数据的修改已经完成并且可以被其他核心观察到。简单地说，当你关心一个原子操作前的内存操作的顺序和可见性时，就应该使用smp_mb__before_atomic();。

  2. 何时使用smp_mb__after_atomic();
    相反地，当你需要确保一个原子操作之后的内存写操作不会被重排序到原子操作之前时，应该使用smp_mb__after_atomic();。这适用于当你在改变了某个状态后，希望确保紧接着的内存操作（例如，对另一部分共享数据的更新）只有在原子操作完全完成后才执行。这样做是为了保证状态的改变对其他处理器立即可见，避免因为重排序导致的不一致状态。

  • 使用场景示例
    smp_mb__before_atomic();的使用场景：
    假设有一个标志位flag和一个计数器counter，你想在增加计数器之前，先更新标志位，并确保这个标志位的更新对其他处理器立即可见。在这种情况下，你会在更新标志位之后、执行原子操作增加计数器之前使用smp_mb__before_atomic();。

smp_mb__after_atomic();的使用场景：
假设在完成一个原子减少操作后，你需要立即通知其他处理器这一变化，通过设置一个标志位。为了确保原子操作完成后标志位的更新对所有处理器可见，并且这个更新不会被重排序到原子操作之前，你会在原子操作之后使用smp_mb__after_atomic();。

总结，smp_mb__before_atomic();和smp_mb__after_atomic();的使用取决于你希望如何控制原子操作前后内存操作的顺序和可见性。正确使用这些内存屏障能够帮助你避免在多处理器系统中常见的并发问题，如数据竞争和状态不一致。

barrier 的由来是因为 GCC 编译器（带参数 -O2）会根据 CPU 特性，如乱序执行，把代码优化为符合特性的代码。最显著的是 CPU 乱序执行，GCC会把相关性差的代码的顺序打乱，使得在执行期间能更符合 CPU 乱序执行。但是在 SMP 系统，对于共享资源，这种乱序性可能是致命的。已知 CPU 访问内存期间会把相应数据放到缓存以便后续访问，每个 CPU 内核有自己的缓存（私有的），同时多个 CPU 内核连接下层的共享缓存。私有缓存的策略一般是回写，根据 MESI 协议可知，当一个共享资源被修改时，仅暂存到对应的私有缓存中，且被标记为修改
