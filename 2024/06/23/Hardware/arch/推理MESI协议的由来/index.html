推理 MESI 协议的由来

Posted on 2024-06-23 In Hardware Word count in article: 2.2k Reading time ≈ 8 mins.

MESI 的作用

在 SMP 系统内，确保 CPU 之间对同一个内存地址访问时，数据是一致的，是最新的。
https://www.scss.tcd.ie/Jeremy.Jones/VivioJS/caches/MESIHelp.htm

初步推理

比如说现在处于下列的 SMP 系统模型（暂定一个缓存行为 1 字节）：

当 CPU0 从内存地址 a0 读取数据时，会把该地址对应的值放到 CPU0 的 Cache0：

注意：每在 Cache 使用一个 Cache Line 时，其实有一个类似于名为 Exist 标签表示当前 Cache Line 被占用的，最起码在 UP 系统是这样的。在 SMP 系统的 MESI 协议已有标签去间接表示当前 Cache Line 被占用了。所以本文章的图示都没有用 Exist 标签表示 Cache Line 被占用，但是仅把 Cache Line 中的 Address 和 Data 项被填充，Flag 项为空，来表示这个缓存行是被占用的。

然后 CPU0 往 a0 写入数据 1 ，此时 Cache Line 0 的 Flag 标记被标记为 Modified 或 Dirty ：

因为 CPU 的时钟频率总是比内存的高，即 CPU 的运行速率比内存快，所以当缓存行内的被修改时，通常来说是不会立即把最新值写回到内存中。当这个缓存行有一段时间没有被它对应的 CPU 访问时，才把它写回到内存中。而这个缓存刷新机制所使用到的算法是 LRU 。

缓存刷新机制的算法不只是 LRU ，还有很多。

此时，CPU1 也来读 a0 地址的值（此时它的 Cache1 没有该地址的值）。如果直接从内存中读，而且此前 Cache0 的 Cache Line 0 没有写回到内存，所以 CPU1 读到的数值必然是 0 。导致此时的 CPU 之间在同一内存地址的数值将会不一致，根本无法保证 SMP 系统的缓存一致性。
为了使 CPU 之间在同一内存地址的数值一致，在 BUS 额外添加一个观察者 Observer 。当 CPU1 往内存发送读请求时，Observer 发现该请求，找出该 Cache0 的 Cache Line 0 的数值是最新的。则命令 Cache0 控制器把 Cache Line 0 的值写回到内存地址 a0（并清除标记 M ），Observer 再把该 a0 的数值从内存发送到 Cache1 的 Cache Line 0 。

如果现在 CPU0 和 CPU1 任意一个去写它自己的 Cache Line 0 ，此时对应的 flag 已为 M ，然后另一个 CPU 去读它自己的 Cache Line 0。Observer 去找出哪个或哪些 CacheX 的数值是最新的，并写回，再把数值从内存发送到 Cache1 的 Cache Line 0 。
但是 Observer 的作用还是微乎其微，因为当某一个地址已在当前 CPU 的 Cache 缓存，且状态为 M 时，是直接在 Cache 读写的，缓存的内容也不会立即写回到内存，Observer 也不可能隔三岔五地主动检查哪些缓存是最新的。应该触发某种条件才能运行 Observer 主动去把最新缓存值写回到内存当中。

回到最初的起点，当下图中的所有 CPU 按次序对内存地址 a0 发送读操作，每当一个 CPU 向同一个内存地址发送此操作，Observer 都会根据此地址去检查所有的 CPU 的 Cache ，确认是否有最新的数值。

然后，当 CPU0 往 a0 写入 1 时，对应 Cache 0 的 Cache Line 0 被标记为 M ；当 CPU1 往 a0 写入 2 时，对应 Cache 1 的 Cache Line 0 被标记为 M 。

这出现了严重的问题，就是缓存一致性失效。而且以后 CPU0 或 CPU1 再次向 a0 发出读操作时，仅向它们自己的 Cache 的 Cache Line 0 读数据，Observer 如同摆设一般了。
这时就需要新增一个标签去表示哪些 Cache Line 是一致的。标记 M 已经用于表示某一个内存地址的数值在本地 Cache 是最新的，这个就不能用来表示其它作用了。现在新增一个标签，名为 Shared ，简写为 S 。表示在多个 CPU 当中，凡是在它们自己的 Cache 内有对同一个内存地址缓存的，而且数值都是一致的，则对应的 Cache Line 都被标记为 S 。

让我们好好了解 Share for Cache Line 的作用，当下图中的 CPU0 和 CPU1 按次序对内存地址 a0 发送读操作时，相应的 Cache Line 都会被标记为 S ：

这些操作，也需要 Observer 去辅助。当 CPU2 也向 a0 发出读操作时，Observer 会检查 a0 在各 Cache 的情况，如果发现它处于 S 状态时，会把 a0 的数值同步到 Cache2 的 Cache Line 0 中，并把它置为 S 。

如果上图中的任意一个 CPU 对 a0 发出写操作时，此时对应 Cache Line 0 内的数值是最新的，这时应主动通知 Observer 对其余存储 a0 数值的 Cache（Share状态）发出通知，说在 a0 Shared 状态组内的某一个 CPU 修改了 a0 的数值，a0 Shared 组就地解散。
那么解散过后，这些存储 a0 的 Cache Line 又该如何表面它们自己的身份？而且 a0 的最新数值是否需要及时写回到内存中？

假设把存储最新数值的 Cache Line 标记为 M ，注意最新数值并没有写回内存。其余的被标记为 Invalid ，失效的。

其实 CPU 上电后，缓存内的 Cache Line 都为空，可以把它们的标志位都置为 Invalid ，与上文提到的标志位 Exist 是对立的。

如果此前 CPU1 访问 a0 时，Observer 发现此行为，然后再把 CPU0 的 Cache Line 0 写回到内存中，然后再让 CPU1 访问 a0 ，更新 Cache1 的 Cache Line 0 此时它们的状态为 S 。

这是可行的，但是还能做得更好。回到所有 Cache 的 Cache Line 0 都为 Shared 状态。当 CPU0 对 a0 发出写操作时，主动通知 Observer 会更改其余 Cache 的 Cache Line 0 为 I ，这段操作应该再加上一个操作，就是把最新数值写回到内存中。然后 Observer 收到通知并更改它们的状态。但是 Cache0 的 Cache Line 0 和内存中的 a0 的数值一致了，不能用 M 标记，应该用 Exclusive 去标记此 Cache Line 。

就这样，在这种状态下，CPU1 和 CPU2 就可以直接访问内存 a0 ，不用让 Cache0 先写回到内存再从内存读出，这叫时间消耗置换。

但是，如果此时 CPU0 再次对 a0 发出写操作，对应的 Cache Line 0 的状态从 E 变换为 M 。

CPU1 或 CPU2 访问 a0 时，Observer 还是会让 Cache0 的 Cache Line 0 写回到内存，然后再让它们访问。

MESI 的状态转换

总结

  • Modified：指当前 CPU 对它自己的本地 Cache 的 Cache Line 发出修改操作。
  • Shared：表示多个 CPU 的指向同一个内存地址的 Cache Line 内的数值是一致的。而且任意一方发出修改操作时，Observer 还能根据此状态找出与它共享的 Cache Line ，并修改状态为 Invalid 。
  • Exclusive：表示仅有一个 CPU 的 Cache 记录着某一个内存地址的数值，而且和缓存中记录的数值是一致的。
  • Invalid：表明缓存行不可用，就算缓存行记录着某一个内存地址的数值，但是是旧数值，不可用。
