UBoot - 基本概念

Posted on 2024-04-01 Edited on 2024-06-15 In 原理及实现 Word count in article: 1.4k Reading time ≈ 5 mins.

基本

U-Boot 在 Linux 嵌入式当中充当重要的角色，它的全称是 universal-BootLoader 。其中，“通用” 也就表示它必须支持全部的 CPU 框架，还有厂商定制的板卡等。它被固定在任何一个外存，如 SD 卡、Flash 等，它是属于在嵌入式眼里的裸机程序。

和 GRUB 都是 BootLoader ，但是 U-Boot 用在嵌入式居多，而 GRUB 用在桌面 PC 或服务器居多。

还有 BootLoader 的本职工作就是加载并启动 Linux 内核，但是要先加载并执行 BootLoader ，而且每个 CPU 或主板的启动方式都不同，这该怎么办？

概念

  1. XIP：XIP 的全称是 Execute In Place ，表示 CPU 可直接通过地址总线执行 BootLoader 或首引导程序的代码。但这些程序的存储方式因 SOC 芯片的结构或主板的结构不同而不同。比如：

    • 第一个就是主板内置了可直接访问的 Flash 和 RAM ，可直接在 Flash 执行 U-Boot ，然后加载内核到 RAM 并执行。
    • 第二个就是 SOC 芯片内置一个 Boot-ROM 存储首引导程序，还内置一个小的 RAM ，通常是 4KB ，用于加载并执行 U-Boot 。而 U-Boot 放在 SD 卡或 Flash 。在 Boot-ROM 执行首引导程序期间，初始化一些引导必须要的硬件，如 EMMC 控制器和存储设备，然后使用 EMMC 控制器选择存储设备，从该设备读出 U-Boot 到内置的 RAM 。然后跳转到 B-boot 执行，初始化相关硬件，然后加载内核。

      imx6ull 主板的 SOC 芯片，内置一个 BROM 和 RAM ，启动方式和第二个讲的类似。

    第二个启动方式和 X86 的类似，即内置了一个 BIOS-ROM ，按次从外存读出 MBR 、BootLoader 、Kernel 。

前面说了，U-Boot 在 SD 卡或 Flash 等等。这些存储设备也是需要一个简单的驱动去初始化设备和读写设备的。所以说，如果是使用第二个方法加载 U-Boot ，在 Boot-ROM 的程序也需要对一些必要的设备进行初始化，然后再加载 U-Boot 。

  1. 驱动：在 U-Boot 启动内核的方式有很多，如用 Flash 、SD 卡等本地设备，还可以用网卡启动等，这些相关的设备也是需要对应的驱动程序来驱动才行。而这些设备用设备树来表示。

  2. 文件系统：已知内核映像存储在任意一个存储设备中，而一般来说存储设备要通过分区、初始化对应的文件系统后才能使用。一个内核映像存储在某个设备的文件系统内，就表示该文件系统是根文件系统。

主目录的 Kconfig 是默认配置，但是不同厂商或系列的主板的硬件配置都是不同的，所以需要额外的配置脚本去配置它们，如 mx6ull_14x14 就需要配置脚本 configs/mx6ull_14x14_evk_defconfig 配置。

Kconfig 的作用和语法看《Kconfig介绍》。

Kconfig 和 XXX_defconfig 是相辅相成的（但注意是它们不是一个种类的东西，一个是“固定”的，一个是“可选”的），通过执行 ./conf ，即可把 XXX_defconfig 内的配置和 Kconfig 内的配置填充到 .config 和对应的 include/config.h 文件。

include/config.h 是在编译 U-Boot 期间，根据 .config 文件和其它文件生成的。
