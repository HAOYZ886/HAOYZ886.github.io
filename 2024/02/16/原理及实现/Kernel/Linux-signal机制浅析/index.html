对于系统调用的

sys_tgkill() 和 sys_tkill() ：
上面的操作需要外，额外需要调用 handle_stop_signal() ，内有针对 SIGSTOP 、 SIGCONT 和 SIGKILL 的预先操作。尤其是 SIGSTOP 和 SIGCONT ，由于在处理 sigpending 期间，是一次执行完成的，如果 sigpending 内同时存在 SIGSTOP 和 SIGCONT ，这它们的任意一个操作来说都是无用功，即我刚处理完 SIGCONT 后，又来 SIGSTOP ？这不合理吧？相关函数 handle_stop_signal() 。

关于 blocked 与 pending 的关系

blocked 和中断控制器的屏蔽寄存器是不同的，如果 sig 与 blocked 中相应的位匹配，且被置位，表示先不执行它，而是把它放到 pending ，但 blocked 中相应的位被清除时，自然就能处理它。

关于 blocked 与 sa.sa_mask 的关系

有些信号是不能屏蔽的，比如 SIGKILL 。sa.sa_mask 的本质就是在对应信号处理期间的 blocked 。当处理某一个信号前，把 blocked 备份，然后把 sa_mask 与 blocked 或操作。blocked 备份存储在 sigframe 内。当处理完成后，会把 blocked 备份写回。

对于指定线程组发送

调用 __group_complete_signal() ：

  • 如果信号是 SIGKILL 、SIGSTOP 或 SIGCONT 等，而且没有被屏蔽，则需要让线程组内的全部线程去处理该信号。
  • 否则指定某一个空闲的线程唤醒。
    • 如果当前发送信号的线程与接收信号的线程是同一个，则由它处理
    • 如果线程组内有无信号处理的线程，则由它处理
    • 如果线程组内仅有一个线程且处于睡眠状态，什么事都不用做，直接返回。等它被唤醒后，自然就在从内核层返回用户层期间处理 sigpending
    • 如果线程组内存在刚刚处理完信号的线程，则继续把该信号交给它处理

设置信号（重载相应信号的 sigaction ）

系统调用 sys_sigaction() ，核心函数 do_sigaction()：

  • 信号 SIGKILL 和 SIGSTOP 不能修改它们的 sigaction
  • 重载相应信号的 sigaction 前，检查是否有 sigpending 为 true ，则先处理它，然后再重载。主要是怕是否有类似于 SIGKILL 信号需要处理
  • 设置信号：

    • 如果对于 sigaction 是 SIG_IGN ，或是 SIG_DFL 但默认操作仍是忽略，则这些信号从线程组的 pending 移除，反正它们都是忽略不做的。

      但这个操作好像与 sig_ignore() 有点冲突，因为发送信号期间都会调用 sig_ignore() ，直接把 SIG_IGN 忽略掉，连加入到 pending 的机会都没有。

    • 如果信号不是上述类型的，则直接重载相应的 sigaction 即可。

  • 相应的 sigaction.sa_mask 不能屏蔽 SIGKILL 和 SIGSTOP 。
