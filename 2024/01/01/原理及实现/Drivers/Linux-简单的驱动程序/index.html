Linux - 简单的驱动程序

Posted on 2024-01-01 Edited on 2024-01-03 In 原理及实现 Word count in article: 2k Reading time ≈ 7 mins.

要点

调用 register_chrdev() 是把字符设备在内核当中登记，最主要的结构体是 struct cdev ，它内有一个字段 kobject ，kobject用于表示在内核内的一个对象，用于表示设备、文件系统的 inode 或设备驱动，多个对象可组成一个线性结构（或树形结构）。在struct cdev，它充当字符设备模型的一个节点，即 cdev_map 。
以上的操作都是在内核登记字符设备的操作，现在需要根据设备号把这些字符设备通过 struct class 抽象成设备文件，供给用户使用。
struct class 相当于是一个模板，当调用 class_create() 之后，就获得一个初步初始化的模板。调用 device_create()，把当前模板、设备号和设备文件名作为参数，在文件系统内新增一个设备文件。

细节

struct char_device_struct 相当于是字符设备的内部结构体。相应的，struct cdev 是提供给内核的结构体，内有 kobject 。
还有，struct char_device_struct 像是一个还未成熟的结构体，因为与它相关的对外函数与 struct cdev 相比还是很少，struct cdev 已经是深深地嵌入内核当中了，所以说 struct cdev 是对外的结构体也不为过。
另外，在内核中发现 char device 子系统的 kdev_t 好像是自己独有的，这是看 __register_chrdev_region() 了解到的。

意思是每个抽象设备子系统都有自己的 kdev_t ？

还有，通过 __register_chrdev() 可知，虽说字符设备的设备集分别用 struct char_device_struct *chrdevs[] 和 struct kobj_map *cdev_map (struct cdev) 登记，但 major 是用 __register_chrdev_region() 得到（如果 major == 0），再根据此 major 和用户提供的 minor ，把 struct cdev 添加到 cdev_map 。可以看出这是一个过渡型的代码。

但未来的走向是什么，需要看最新版的内核源码。但在 linux 4.xx ，先把 struct char_device_struct 理解为对内，struct cdev 对外。

led_board

因开发板的型号的不同，对各硬件的操作路径在不同，如可被操作的 LED 灯，在 100ask imx6ull 开发板是在 GPIO5 的 03 引脚，而在其它型号的开发板，对 LED 灯的操作路径也大概率不同（而且说不定也没有对应的操作路径）。所以，从中可知需要把 LED 驱动分层，上层是通用层，下层是针对各个开发板操作 LED 的操作层，可新建一个 board 目录存储这些不同的开发板操作。

led_board_gpio

在回顾开发 led_board 的过程中发现，虽说在不同型号的开发板，对 LED 灯的操作路径不同，但是操作方式是一致的，如初始化阶段：使能 GPIO group 、指定某个 iomux 引脚为 GPIO mode ，指定引脚的方向等，这些操作不管在哪一个型号的开发板中都是一样的。那还不如把对 GPIO 的操作从 led_board 中分离出来，独立于一层，且在 LED 层的下面。
但这样必须要事先定义一系列的结构体，用于登记设备信息，对于 GPIO 层，需要登记某个开发板的 GPIO group 信息，一共有多少个 GPIO group 等；对于 LED 层，需要登记 LED 灯的操作路径，即在哪个 GPIO group ，在哪个 GPIO 引脚。

led_board_platform

通过 led_board_gpio 可以发现一个问题，虽然说不同型号的开发板，对 LED 灯的操作方式是一致，不同的是操作路径。我们不仅需要存储这些 LED 灯的操作路径，也需要存储 GPIO 的基本信息，因为不同型号的开发板，GPIO 的参数也不同。
但随着开发板型号的增加可以发现，这些设备信息越来越多。对于内核来说，这是一个灾难，因为这些硬件资源也从属于内核管理的。嵌入式开发者们只是注重于自己开发板的驱动模块而定义的存储设备信息的结构体，从而忽视内核也是需要管理这些信息。所以内核开发者在 struct device 的基础上新增一层 platform device/driver ，供给嵌入式开发者使用，从而解决内核无法统一管理设备资源的问题。

要点

struct platform_device/driver 分别在 platform_bus_type (struct type_bus) 用双向链表连接着。
platform device 相当于是 struct device 的子类，在 platform_device_register() 可以发现，在此调用路径中，发现 device_add() 被调用。在 device_add() 中，调用了 bus_add_device() ，把 struct device 连接到 platform_bus_type （节点字段为 device->p->knode_bus ）。
然后调用 bus_probe_device() 匹配 driver ，在此函数的调用路径中，会调用 platform_bus_type->match() ，使用当前 device 逐一与 driver 匹配。

通过上面的说明可知，platform device/driver 在 struct device/device_driver 的上层。

platform_device（或是 struct device）与抽象设备之间的关系
struct device 用于表示一个具体的物理设备，在 led_board_platform/led.c 中，事先调用了 register_chrdev() 登记为字符设备，调用 class_create() 获得模板，在 led_board_platform/board/100imx6ull_leddev.c 和 100imx6ull_leddrv.c ，分别定义了 led 的 platform device 和 platform driver 。当 device 和 driver 匹配成功时，调用 ledusr_class_create_device() ，把当前设备抽象成设备文件，而且 platform device 的 struct device 中，字段 class 用于记录当前模板。所以，可以理解为在linux内核，struct device 处于抽象设备的下层。

led_board_td

在 led_board_platform 发现问题，虽说用 platform device/driver 使内核统一管理设备信息，但这些信息在代码中一多起来，导致过于冗余、同质化了，所以现在需要引入设备树。现仅需一个 .dts 文件，即可记录设备信息。把 .dts 文件编译为 .dtb 文件并载入到系统文件系统的 /boot 目录内。在启动期间，UBoot 同时加载 .dtb 文件和内核映像，生成 device_node 节点，组成一个设备树。在 device_node 内，字段 properties 记录了当前设备的多个属性，可调用相关函数去读取。
另外，在 struct device ，字段 of_node 存储当前设备在设备树的节点。而且默认生成 struct platform 并调用 platform_device_register () 登记。

那么 driver 是如何匹配到在设备树的设备？
在 struct platform_driver 的字段 driver 中的字段 of_match_table ，它用于匹配 device 的描述表，表的字段 compatible 是匹配 device 的关键。在 platform_type_bus 的 match() ：

of_driver_match_device() 使用 drv->of_match_table 与 dev->device_node 的 compatible 、type 或 name 匹配（后两者已淘汰不用，compatible 存储在 dev->device_node->properties 的其中一个）。

copy_from/to_user 的由来

因为防止 APP 通过系统调用传输的指针是非法的，即不是用户空间内的虚拟地址。还有就是因为指针指向的内存空间是有一定的不确定性，比如说多个进程的用户空间中，有一部分物理内存空间是共享在这些进程的用户空间中；又或者一个用户空间内存在若干个线程等等。这些不确定因素内核是不可能完全控制的，为了确保在内核态期间，该指针指向的内存空间的数据一致性，必须先把数据传输到内核空间，完成后再传回用户空间。
