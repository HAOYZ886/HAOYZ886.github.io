AHCI驱动的实现思路（内核版本 2.4.22，32位x86）

Posted on 2024-01-07 In 原理及实现 Word count in article: 7.5k Reading time ≈ 27 mins.

前言

这是基于 Linux 2.4 ，自己实现的 AHCI 驱动框架。

一些概念知识

AHCI 是什么

AHCI 的英文全称是 Advanced Host Controller Interface ，中文则为高级主机控制接口。结合字面意思理解，它是一个 CPU 与 SATA 设备之间沟通的桥梁，能够让软件更方便、更规范和更高效地去访问 SATA 设备。AHCI 定义了存储器架构规范，为硬件制造商提供了系统存储器与 SATA 存储设备之间的传输数据的规范。同时，AHCI 也定义了一系列供给软件访问的端口（寄存器组），如控制寄存器组、状态寄存器组等。还有，AHCI 同步支持 SATA 的众多特性，以 AHCI v1.3.1 版本为例，支持电源管理、NCQ 、接口扩展和热插拔等。
最后，AHCI 是一个 PCI/PCIe 类设备，在系统内存总线和 SATA 设备内部逻辑之间扮演一种通用接口的角色。而且它在 PCI 配置寄存器组提供了一个 PCI BAR ，把 PCI BAR 内的地址在内存控制器（北桥）映射到总线地址空间，软件可直接通过物理地址访问到 AHCI 控制器的寄存器组。

AHCI 控制器又称 HBA (host bus adapter) ，指使用 AHCI 规范的，在系统内存和 SATA 设备之间建立沟通的适配器。

SATA 是什么

SATA 的前身是 PATA ，PATA 被 SATA 淘汰的原因是它的并行传输的物理限制，而从并行传输替换为串行传输，即使用 SATA 能够解决此问题（了解详情）。另外， SATA 还提供协议层进行通信，从下到上分别是物理层、链路层、传输层和命令层，见下图。

其中， FIS 是主机和设备之间通信的一个机制，SATA 提供了 14 种不同类型的 FIS ，每一个 FIS 的格式固定，有惟一的 ID 。详情请戳这里。
最后，SATA 命令的规范是由 ATA/ATAPI8-ACS 提供的。

AHCI 和 SATA 之间的联系

结合上文可知，SATA 是被 AHCI 封装的，使得 SATA 类似于一个黑盒一样，这是因为软件可直接访问 AHCI 提供的一系列寄存器组即可控制 SATA 设备、查看 SATA 设备的状态、发送命令给 SATA 设备。而且中断信号的发送是由 AHCI 主导的。所以，我们仅需知道 SATA 的基本概念以及一些常用的特性和命令即可，重点关注 AHCI 的使用。

本驱动是基于 Linux 2.4 编写，运行环境为 QEMU ，QEMU 的 AHCI 控制器的版本是 AHCI v1.0 ，对应的 SATA 是 v1.0，使用 ATA/ATAPI8 - ACS 。

还有实现该驱动前，需要知道该驱动的侧重点是什么，主要从 性能-生产率-通用率 考虑。
本驱动更多是注重性能，目前已支持命令队列、单命令大容量传输（最高 32 MiB）、支持 NCQ 。其次是通用率，识别不同型号的 AHCI 控制器，针对不同的 SATA 设备，如 DISK 、CDROM 等提供不同的函数接口。最后是生产率，因为同时需要兼顾性能和通用率，所以编写此驱动耗费时间很长。所以，性能-生产率-通用率 之间是相互互斥的，不可能面面俱到。还有，其他诸如可靠性、易读性、正确性本驱动也有对应的实现。
想了解更多与 性能-生产率-通用率 相关的知识点请看《深入理解并行编程》第二章。

框架展示

结构图

把驱动分层的目的是提升可扩展性、可维护性，每一层分工明确，尽可能地满足高内聚低耦合。

  • IOMEM 层：该层是把“软件访问 AHCI 控制器的寄存器组”封装，提升 I/O 访问的可靠性和可扩展性。
  • 内部层：它属于基层，也是 IOMEM 层的上层，提供开启 AHCI 模式、重启 AHCI 控制器、提供获取/释放命令、中断处理和计时处理等功能。
    • 命令层：发送命令，接受命令执行结果。
    • SATA 层：识别 SATA 设备，对命令进行解析，不同类型的 SATA 设备，解析命令的方式也不同。
    • 数据请求层：它是上层接口层与 SATA 层之间的过渡层，用于把块设备的 request 解析为对应的命令描述符。
  • 接口层：和块设备驱动接轨的层。

PS：其实可以尝试把内部层的内的层独立出来，但由于开发时间不够，暂且作罢。

.C 源码文件分布

其中，ahci_probe.c 是用于 AHCI 驱动初始化；ahci_pci.c 和 pci/ 目录下的源文件是用于识别 AHCI 控制器的型号。

.H 头文件分布

从全局去说，
ahci.h 存储的是本驱动的所使用的宏、函数等；
ahci-iomem.h 是存储 AHCI 控制器的寄存器组的分布；
ahci-io.h 是存储一些发送命令的结构体、函数等；
ahci-ops.h 则是把 IOMEM 层的函数集供给内部层使用；
ata.h 、ata-hdreg.h 、ata-cmdtype.h 是与 ATA/ATAPI8 - ACS 相关。

注意，图内的头文件都是在驱动目录内的 include 目录内，对外提供的头文件则是 ahci.h（和图中的 ahci.h 重名，但用途不一样），它定义在全局的 include/linux 内（对应到我的内核目录是 include/eleos ）。

结构体分布以及关系

先从 struct ahci_general 说起，这是每一个可用的 AHCI 控制器的描述符，用于记录 AHCI 的运行状态。ahci_gen_regions[] 是一个指针数组，被定义在 ahci.c ，当识别到一个可用的 AHCI 控制器时，会分配一个内存空间作为描述符，并根据当前 AHCI 控制器的运行信息进行初始化。
其中 AHCI_CTR_MAX 是一个宏，拟定最多有八个 AHCI 控制器（实际上一个 AHCI 控制器就够了）。
下列是 struct ahci_general 结构体：

  • ports_max ：当前 AHCI 控制器可用的端口数量。
  • ports_max_map ：可用的端口数量的位图，如端口 0 可用，在位图的 bit0 被置位。
  • active_ports_max ：当前活跃端口的最大数，如端口 0 和 2 是活跃的，则该字段为 2 。
  • active_ports_map ：当前活跃端口的位图，如端口 0 和 2 是活跃的，则位图的 bit 0 和 2 被置位。
  • cmds_max ：当前 AHCI 控制器的每个端口的命令队列支持的最大命令数量。
  • status ：当前 AHCI 控制器的运行状态，以下是运行状态。
  • cap 和 cap_ext ：分别和 AHCI 的 GHC.CAP 与 GHC.CAP2 对应。
  • dma_mask ：这需要当前架构（包括 PCI 总线）和 AHCI 的 DMA 字长相同，任意一方的字长低于另一方的则需要向下兼容。
  • invalid_ports_max 和 invalid_ports_map ：在初始化驱动时，检测发现某个端口不可用，则被记录下来，待初始化结束时释放相应资源。
  • irq ：每个 AHCI 控制器都有自己的 IRQ 号，尽管 IRQ 号有相同的可能。
  • dev ：AHCI 控制器是一个 PCI 类设备，有一个属于自己的 PCI 设备描述符。
  • ahci_status ：记录 AHCI 控制器的每个活跃端口的运行状态，对应的结构体如下： 这个结构体，需要在讲解命令的运行流程时才能清楚知道它们的作用。

轮到 struct ahci_iomem ，该结构体由 IOMEM 层独享，用于保护 AHCI IOMEM ：

  • ahci_lock ：保护 AHCI 控制器的 HBA Memory Registers 内的 Generic Host Control
  • hba_base ：HBA Memory Registers 的基址（虚拟地址），是把 HBA 的 PCI BAR 内的地址空间映射到总线地址空间，再通过内核空间的非连续区域把物理地址转换为虚拟地址得来。
  • port_lock ：保护对应端口的 HBA Memory Registers 内的 Port Registers 、Received FIS Structure 和 Command List Structure 。
  • cmdlists_base ：这是一个数组，数组元素等于端口号，记录 Command List Structure 基址（虚拟地址）。
  • rFISs_base ：这是一个数组，数组元素等于端口号，记录 Received FIS Structure 基址（虚拟地址）。

struct hba_port ，该结构体是每个活跃的端口都有一个对应描述符，用于记录端口内命令队列的运行情况：

  • lock ：保护字段 cmds_nr 、cmds_active_nr 、cmds_active_map 、qcmd_tfs 以及 active_list。
  • cmds_nr ：当前端口支持的最大命令数量。
  • cmds_active_nr ：当前端口正在运行的命令数量
  • cmds_active_map ：正在运行的命令的位图，如 cmdslot0 的命令正在运行，则对应的位置位

    cmdslot0 是 Command List Structure 内的第 0 个命令槽。

  • qcmd_tfs ：这是一个数组，一个元素对应一个命令描述符，对应的结构体较大，稍后解释。
  • active_list ：当一个命令被主机端发送时，把该命令对应的描述符加入到该链表（从尾部加入），这用于当命令队列执行中出现错误时，能够使用该链表从头到尾重新把命令发送到 SATA 设备。
  • sata ：通常来说，一个端口对应一个 SATA 设备（本驱动不支持 port multi），这是 SATA 设备的描述符，该结构体较大，稍后解释。
  • wait_cmd ：当一个任务需要发送命令时，如果无空闲的 cmdslot 或正在执行 ncq 命令，则可以尝试睡眠等待，待有空闲的 cmdslot 时或执行完 ncq 命令后唤醒。
  • wait_ncq ：和 wait_cmd 类似，不同的是该任务是想要发送 ncq 命令，但无空闲的 cmdslot 或正在执行 normal 命令时睡眠等待。

结构体 ahci_qcmd_tf 是一个命令描述符，在发送命令之前，填入命令的同时，也要填写它所需要的参数。而该命令的参数包含 Command List Structure 内的第 N 个命令槽和对应 Command Table ：

  • state ：命令当前的状态，见下列的 enum

    该驱动暂未把 AHCI_QCSTAT_SYNCWAIT 和 AHCI_QCSTAT_SYNCSLEEP 提供给接口层使用。

  • cmd_list_flags ：与 Command List Structure 的 cmdslot 的 DW0 对应
  • prdts_nr 和 prdts ：与 Command Table 的 PRDT 对应，PRDT 的全称是 Physical Region
    Descriptor Table ，当命令是 data 类型时有效，作用是记录主机端系统内存的地址以及大小。比如，使用读命令从 SATA 设备读出的数据，根据 PRDT 提供的信息把它读出到对应的地址空间。
  • taskfile ：前文说过，SATA 设备使用 ATA/ATAPI8 - ASC 命令规范，所以这个描述符是命令的本体：
  • acmd 和 acmd_szl ：如果设备是 CDROM ，那我们需要使用 atapi 类命令，使用该类型的命令需要额外的参数，这些参数用字段 acmd 存储，acmd_szl 记录其大小（以 DWord 为单位）。
  • icc ：与 NCQ 相关，本驱动也没有支持该特性，默认为 0 即可。
  • lba 和 size ：如果该命令是 data 命令，该字段有效。
  • rFIS_buf 和 rFIS_buf_sz ：state 存在 AHCI_QCSTAT_OUTPUT 时有效，获取当前命令的执行结果。
  • start 和 start_data ：state 存在 AHCI_QCSTAT_USER 时有效，在加载命令期间，回调 start() 到接口层做上层任务。
  • complete 和 complete_data ：state 存在 AHCI_QCSTAT_USER 时有效，当命令被执行完成时，回调 complete() 到接口层做上层任务。
  • waiting_time ：该字段有两种用法：
    1. 要求命令要在规定的时间内完成，否则触发超时处理；
    2. 如 state 存在 AHCI_QCSTAT_SYNCWAIT 时，该字段表示任务的忙等待时间。
  • expiry ：与字段 waiting_time 的第一个用法相关，把该命令加入到 timer_list 内，如果没在规定时间内执行完成，触发超时处理。
  • wait ：如 state 存在 AHCI_QCSTAT_SYNCSLEEP 时有效，使当前任务睡眠等待。

结构体 struct sata_device ：

  • lba_sectors ：SATA 存储器内的最大可用扇区数
  • xfer_mode ：进行数据传输时，首选哪种模式，是 PIO 、MDMA 、还是 UDMA
  • class ：当前设备是 ATA 设备还是 ATAPI 设备
  • media ：设备是 DISK 、CDROM 还是 FLOPPY ？
  • major ：记录当前设备首选的命令规范，如 number=10 ，表示 ATA/ATAPI8-ACS3 ，这在 IDENITFY DEVICE data 的 Word 80 读出得知。
  • ncqDepth ：NCQ 深度，默认 31 ，在 IDENITFY DEVICE data 的 Word 75 读出。
  • max_multi ：这对 PIO 模式有效，表示一次读写的扇区数量。
  • driver ：对应结构体：

    在讲解与 struct sata_driver 相关的思路时说明

req_to_qcmd_t，把 request 转换为命令描述符 qcmd 的过渡型结构体：

  • list ：与其它 req_to_qcmd_t 连接，形成一个链表。通常来说，该链表的数量取决于当前需要处理请求的数量。
  • rw ：与 request 的 cmd 字段对应。
  • bh_head ：记录当前 request 的未处理的 struct buffer_head 的起始点。
  • lba_base ：在 SATA 设备的起始点。
  • start ：当在内部层把请求处理到一定阶段后，可回调该函数，利用相关信息更新上层的信息。
  • complete ：当在内部层把请求处理完成期间，可回调该函数，利用相关信息更新上层的信息。

struct ahci_port ，该结构体用在接口层，用于与块设备接轨的描述符。

  • status ：该字段仅有一个 PRESENT 状态位，表示是否可用
  • queue ：这是块设备的请求队列，强制要求的。
  • req_cmds_nr ：记录正在活跃的请求个数
  • req_cmds ：数组，每个元素对应一个 cmdslot ，记录当前请求的处理状态。

各层的函数讲解（不包含初始化）

IOMEM 层

以下是 IOMEM 层的函数接口：

函数 作用
ahci_ghc_read & ahci_ghc_write 对 HBA Memory Registers 的 Generic Host Control 读写
ahci_port_read & ahci_port_write 指定 HBA Memory Registers 的其中一个 Port Register 进行读写操作
ahci_port_cmdlist_read & ahci_port_cmdlist_write 指定某一个 HBA port 的 command list 进行读写操作
ahci_port_rFIS_read 指定某一个 HBA port 读取从 device 发送给 host 的 FIS ，需要指定 FIS 的类型读取
ahci_port_cFIS_write 指定某一个 HBA port ，把 FIS 写入到它的 Command Table 的 Command FIS 区域
ahci_port_ACMD_write 指定某一个 HBA port ，把 ACMD（for ATAPI）写入到它的 Command Table 的 ATAPI Command 区域
ahci_port_prdt_read & ahci_port_prdt_write 指定某一个 HBA port ，在它的 Command Table 的 PRDT 内指定某一个表项读写数据（通常是写操作具多）

内部层

基层

以下函数是供给接口层和内部层使用：

函数 作用
ahci_enable_ahci 开启 AHCI 模式，因为有些 HBA 也支持旧式的标准，所以可以选择兼容模式还是 AHCI 模式。当然，该驱动默认就是 AHCI 模式
ahci_reset_hba 重启 HBA ，当执行命令发生错误时，如 CRC 校验错误，重启 HBA 。还有初始化 HBA 时重启，因为在此期间 BIOS 会访问 HBA 并读写某些寄存器组，重启确保后续初始化操作无误
ahci_port_power_up 对某一个 HBA port 启动 active 状态，与接口的电源管理相关
ahci_disable_hbaport 关闭一个 HBA port 端口，关闭发送命令和接受 FIS 的接口（同时关闭 DMA 引擎）
ahci_enable_hbaport 开启一个 HBA port 端口，开启发送命令和接受 FIS 的接口（同时开启 DMA 引擎）
ahci_spinUp_device 如果 HBA 和对应 SATA 设备支持 Spin Up 有效，Spin Up 的功能是冷加载。该函数的功能是使用 Spin Up 重启指定端口对应的 SATA 设备
ahci_reset_sata 重启指定端口对应的 SATA 设备
ahci_has_port 检测指定的端口，检查是否活跃，即是否已连接上 SATA 设备
ahci_restart_port 重新开启 HBA port，如果 SATA 设备出错，也会需重启动 SATA 设备

以下函数用于命令层：

函数 作用
ahci_alloc_qcmd 从 struct hba_port 获取一个空闲的 CMD 描述符，并初始化
ahci_free_qcmd 释放 CMD 描述符，释放相关资源
__ahci_free_qcmd 释放 CMD 描述符
ahci_free_qcmds_nr 查询有多少空闲的 cmdslots
ahci_tf_to_fisH2D 把 taskfile 转换为 H2D FIS
ahci_sync_waiting_cmd 当前任务对当前命令进行忙等待

以下函数是与 list_timer 相关：

函数 作用
ahci_add_timer 把当前命令描述符加入到 list_timer
ahci_del_timer 把当前命令描述符从 list_timer 移除

命令层

函数 作用
ahci_send_ata_cmd 把一个命令描述符对应的命令发送到指定的 SATA 设备
ahci_send_ata_ctl 把一个命令描述符对应的命令（控制）发送到指定的 SATA 设备，意为把控制抽象成一个命令发送
ahci_send_ata_ncqcmds 发送一个命令列表（命令描述符表）到指定的 SATA 设备，且都是具有 NCQ 特性的命令
ahci_resent_ata_cmds 把还没得到执行的命令利用它对应的命令描述符重新发送，用于中断处理
ahci_resent_ata_ncqcmds 把还没得到执行的 NCQ 命令它对应的命令描述符重新发送，用于中断处理
ahci_complete_qcmd 当命令描述符对应的命令被 SATA 设备执行完成后，把命令描述符释放
ahci_complete_qcmds 当一组的命令描述符对应的命令被 SATA 设备执行完成后，把命令描述符释放
ahci_next_cmds 被中断处理程序调用，当没有任何一个执行流处理请求队列时，该函数能使当前执行流继续处理请求队列

结构体 request_queue_t 针对 AHCI 驱动的改进

为了满足 AHCI 驱动的多命令队列、NCQ ，对 request_queue_t 新增了一些功能。展示结构体 request_queue_t 的改动：

以上字段是新增和有改动的，其中

  • nr_requests ：记录当前 request 数量，之前是记录当前队列最大的 request 数量
  • max_requests ：记录当前队列最大的 request 数量
  • handle_multi ：被置位，表示可以一次处理多个 requests ，head_active 被置位时，该字段才能允许被置位
  • active_rq ：handle_multi 被置位才能使用，见相关结构体：
    • active_reqs_nr ：记录当前队列允许有多少个 requests 能被一次处理，该字段仅有在字段 plugged 为 1 时，即还未有任务执行处理该请求队列时才能被修改。
    • active_reqs_tail ：记录将被处理的多个请求的最后一个请求，
    • reqs_high, reqs_low, sects_high 和 sects_low ：允许一次处理多个 requests 是有要求的，下面将列出这些字段的作用。这些字段仅在初始化期间被修改。
    • running_reqs_tail ：从 request_queue_t->queue_head.next 到 active_reqs_tail 内，记录正在处理的 requests 的尾部。
    • running_bhs_tail ：如果一个 req 过大，无法把 req 内的所有 bh 处理，则记录最后一个被处理的 bh 的下一个 bh ，即 b_reqnext ，一般为 NULL 。
    • unhandled_reqs ：记录还未被处理的 requests 个数。注意，这个与 request_queue_t->nr_requests 不同的是，unhandled_reqs 记录范围是才从 running_reqs_tail 到 request_queue_t->queue_head.prev ，而 request_queue_t->nr_requests 的是记录整个命令队列 request_queue_t->queue_head 的 requests 个数。即 unhandled_reqs 不包括正在处理或已处理的 requests 个数。

现在到处理请求队列期间，与 active_rq 相关的函数有：

blk_get_running_req() 用于在处理请求队列时，从请求队列的头部获取 request 。如果字段 running_bhs_tail 不为空，表示 running_reqs_tail 指向的 request ，它的 bh 仍有一部分未被处理，则直接返回 running_reqs_tail 继续处理当前 request ，否则返回 running_reqs_tail->next 。

函数 blk_update_running_req（），当一个 request 处于加载状态，就是把 request 内的数据载入到实际的物理设备上时，且准备执行对应命令时，该函数被调用。该函数的参数 bh_tail 是记录还未被加载的 bhs 的起始点（单链表），如果当前的 request 内的所有 bh 都被加载了，则 bh_tail 为 NULL 。该函数的作用是更新 running_reqs_tail 或 running_bhs_tail 到下一个准备处理的 request 或 bh 。另外，如果 running_reqs_tail 更新了，相应的 unhandled_reqs 也会同步减一（调用 blk_dec_unhandled_reqs() ），表示还未加载的 requests 减一。

blkdev_update_active_rq() ，该函数在一个 request 处理完成时被调用，准确来说是在当前 request 从请求队列删除前被调用。第一个条件判断，当 request 处理完成后需要从请求队列中移除，如果将从队列中移除的 request 等于 running_reqs_tail ，则需要把 running_reqs_tail 移到上一个节点，通常来说是 q->queue_head 。
然后，如果 a_rq->active_reqs_tail 的下一个节点是 q->queue_head ，则表示还有最后一次的多处理 requests 。
最后，如果 active_reqs_tail 的前后节点都是 q->queue_head ，表示整个请求队列都处理完成，初始化 active_rq 。
