Linux - 设备驱动模型

Posted on 2025-10-10 In 原理及实现 Word count in article: 5.4k Reading time ≈ 20 mins.

前言

需要分别从 kobject 和 struct device 进行梳理。原因是 kobject 是通用表示层，而 struct device 是设备表示层，用 kobject 表示 struct device 只是它的作用之一而已。比如说设备树的节点 of_node 描述符中存在一个 kobj ，在 /sys/firmware/devicetree/base/ 表示设备树的节点。
另外，不要把 sysfs 的节点和 kobject 搞混，sysfs 中的节点是靠 kobject 生成的，kobject 的字段 sd 用于表示在 sysfs 中的层次关系。

kobject

kobject的结构

以下是它的结构：

  • name ：kobject 的名字
  • entry ：多数与 struct kset 连接，表示它是 kset 集合的一个成员
  • parent ：用于表示 kobject 之间的上下层关系
  • kset ：用于表示它属于哪个 kset 集合
  • ktype ：用于表示一个 kobject 的 “类型” ，这是上层对象对它自己的基础对象 kobject 的描述。里面最主要定义了一个对象的生命周期的结束（ kref 为零）的释放操作。
  • sd ：这是 kobject 在文件系统层的表示。在文件系统层中，一个 kobject 对应一个目录，目录内的结构由它上层对象的定义。
  • kref ：kobject 的引用计数，这是它的生命周期，默认初始化为 1 。 kref 为 0 时表示该 kobject 生命周期的结束，会被释放。

kset的结构

  • list ：用于链接 kobject 节点，表示它们都是 kset 的成员
  • list_lock ：用于保护字段 list
  • kobj ：kset 的基础对象，kset 之间亦有上下层关系，也有主集合和子集合的关系，需要用 kobject 去表示。

总结

kobject 是表示一个通用对象的结构体，可以是 sysfs 内部机制中的节点、或在 kobj_map 内用于标识一个节点等。
kobject->sd (struct kernfs_node) 只是在 sysfs 中的一个展示给“外界”的方式。

kset 中的 kobj 表示集合内的根节点，类似树的根节点。

  • 子节点 kobject 通过字段 parent 指向它，表示 kobject 的上下层关系；
  • 用字段 entry 与 kset->list 连接集合关系；
  • 以及用字段 kset 指向 struct kset 表示从属关系。

把这些关系比喻成公司就好理解了：
子节点 kobject 通过字段 parent 指向它，表示 kobject 的上下层关系：可转化为公司内的员工的上下级关系。
用字段 kset 指向 struct kset 表示从属关系：可转换为员工属于什么部门。
用字段 entry 与 kset->list 连接集合关系：相当于一个部门的花名册。

kset的衍生作用

/sys/bus 和 /sys/class 都是集合。其中 /sys/bus 记录了整个系统的总线，每个总线里面记录了它们自己的总线设备。而 /sys/class 记录的是整个系统的设备子系统，每个子系统里面记录了它们自己的设备。
换句话说，/sys/bus 是以硬件系统总线为基础的集合；/sys/class 是以内核的子系统为基础的集合。一个基于硬件，另一个基于软件。

那么一个设备该如何同时在 /sys/bus 和 /sys/class 中表示？

struct device 的各式用法

Linux 对 struct device 的定位是“模糊”的，它像是 kobject 的子类一样，表示设备对象。它可以表示真实设备的某种功能、某个内核子系统的逻辑设备。如 /sys/bus/platform/devices/gpio-keys/ ，它是 struct device ，也是 platform device in platform layer 。它的目录有 input/input1 ，它是 input 子系统的，input1 是 struct device 。在 input1 还有一个 event1 ，它也是一个 struct device ，内有 dev_t ，表示可以直接供给用户层使用的 device ，即在 /dev/ 创建设备文件供给用户层使用。

struct class 的作用

struct class 是与 bus_type 相似的结构体，它主要的作用是集合同一个功能或类型的 struct device 。同时，通过它的中文翻译为：模板，你也可以认为是若干个 struct device 用此 class 生成并通过它的 kobject 记录在 struct class 的 kset 内。

由此可以得出一个结论，bus_type 和 class 都是对各类硬件设备分类的工具。
不同的是 bus_type 注重于真实设备生成的struct device，如通过遍历设备树生成的 platform device ，统一划分到 /sys/bus/platform 。还有一些总线设备，都需要有一个 struct device 去表示所属总线。
而 class 注重于内核中的各类子系统/模块的 struct device，如上述提到的 input 子系统，生成的 input1 和 event1 都在 input_class ，同时在 /sys/class/input 内表示。
注意这些 struct device 也有上下级之分，如 gpio-keys

通过上图可知，通过 device 与 device 之间的连接，使得各子系统分为上下层也连接在一起。但各个子系统没有 device 的连接，是不会衔接在一起的。通常来说，Linux 内核内的各个子系统都是各尽其职的。但根据某个设备，去对应的驱动的实现需求使得把所需的子系统连接在一起，比如说 input 子系统的 event device ；再比如说触摸屏（i2c 设备），需要 i2c 子系统、input 子系统、framebuffer 子系统等（其中，i2c 子系统是后两者的下层，input 子系统和 framebuffer 子系统是处于同一层）。
所以，由此可知内核的各子系统都是独立。
