Posted on 2025-10-14 In 配置 Word count in article: 5.7k Reading time ≈ 21 mins.

SDK框架说明

整体目录

Tina SDK目录结构，主要有构建工具、构建系统、配置工具、工具链、芯片配置目录、内核及 boot 目录等组成, 下面按照目录顺序做简单介绍。

Brandy（U-Boot）

brandy 目录下主要存放 boot0，uboot 等代码：

目录 u-boot-2023 和 u-boot-bsp 是我们主要的开发对象。

在新版 Tina 5.0 ，全志把他们自己开发的驱动源码文件和配置文件等，存储到目录 u-boot-bsp 。

而目录 u-boot-2023 存储的是原生 U-Boot 。

Kernel和BSP

kernel 和 BSP 是一体的，目录 bsp 内有全志自己的驱动源码和配置文件。

Kernel

kernel 目录主要存放不同版本的内核代码：

目录名前缀“linux-5.10”表示内核版本，后缀“-origin”、“-rt”和“-xenomai”等表示内核特性。

  • origin ：表示原生稳定版本的内核
  • rt ：打了实时补丁的内核（软实时，还达不到硬实时要求）
  • xenomai ：打了实时补丁的内核（接近硬实时的精度，但配置繁琐，非常依赖于原厂驱动和应用层 xenomai 的支持）

我们一般使用 linux-5.10-rt 内核，其次是 linux-5.10-origin 。

BSP

bsp 目录下存放着 Allwinner 的驱动，以及对内核的改动，保证着原生内核 (kernel/linux‑X.X) 比较干净的状态，并且能够快速移植适配到新内核中。

目录 configs 和 drivers 是我们主要的开发对象。

  • 目录 configs 存储了内核的配置单和 SoC 设备树文件；
  • 目录 drivers 存储了全志的驱动源码文件。

Buildroot和platform

Buildroot 和 platform 是一体的，目录 platform 内有全志自己的软件构建。

Buildroot

buildroot 相关的配置文件以及原生 builroot 代码。

  • buildroot-202205 是原生 buildroot
  • config 和 package 是全志的配置文件

platform

platform 目录存放着一些软件包源码，这些软件包的编译方式是通用的，分别可以用在 OpenWrt 或者 Buildroot 等不同构建系统中。这个目录的存在是为了不同构建系统共用软件包提供可能性。

RTOS

rtos 目录用于存放异构系统构建环境，不同平台 rtos 构建存在差异，rtos 构建查看《FreeRTOS_RTOS_系统_开发指南》、《{IC}_RTOS_系统_开发指南》：

device

device 目录用于存放芯片方案的配置文件，包括内核配置，env 配置， 分区表配置，sys_config.fex，board.dts 等。
这里以 t153 为例：

目录 configs 是我们的主要开发对象，目录 boot-resource 其次。

  • 目录 configs 里面存放 SoC 板卡方案的配置文件，demo 是我们常用的板卡方案。
  • 目录 boot-resource 是 U-Boot 的资源分区，U-Boot 的 LOGO 显示就是放在这个目录内。

out

out 目录用于保存编译相关的临时文件和最终镜像文件，编译后自动生成此目录，例如编译方案 t153-demo ：

其它

prebuilt

prebuilt 目录存放着一些预编译好的工具，主要是内核和 Buildroot 的交叉编译链：

可提取 rootfsbuilt/arm/gcc-arm-10.3-2021.07-x86_64-arm-none-linux-gnueabihf ，作为应用层软件的编译链。

build和tool

build 目录用于存放编译 SDK 的脚本工具。

tools 目录用于存放一些 host 端工具, 例如打包工具。

I2C配置

以下是全志的 I2C 配置，驱动源码bsp/drivers/twi/twi-sunxi-core.c。

主要关注以下属性：

  • 属性clock-frequency：配置 I2C 总线的时钟频率
  • 属性pinctrl-*：配置 I2C 总线的 IO 复用
  • 属性twi-supply：IO 的电源来源，根据核心板的硬件原理图的配置去指定使用

SPI配置

以下是全志的 SPI 配置，驱动源码bsp/drivers/spi/spi-sunxi.c。

以下是 SPI-HOST 的主要属性：

  • 属性pinctrl-*：配置 SPI 总线的 IO 复用。全志的对该总线的 IO 复用分两部分，一是时钟 IO 和数据 IO ；二是功能 IO ，如 CS 。
  • 属性sunxi,spi-bus-mode：默认为 MASTER 模式，还有SUNXI_SPI_BUS_SLAVE、SUNXI_SPI_BUS_BIT、SUNXI_SPI_BUS_DBI和SUNXI_SPI_BUS_CAMERA等，根据客户需求选择。
  • 属性sunxi,spi-cs-mode：默认由硬件操作 CS 脚，可以使用SUNXI_SPI_CS_SOFT，由软件手动控制。

以下是 SPI 从设备的主要属性：

  • 属性reg：从设备的片选地址，
  • 属性spi-max-frequency：从设备要求的 SPI 时钟频率
  • 属性spi-rx-bus-width和spi-tx-bus-width：从设备要求的 mosi 和 miso 的总线宽度（bit）。

GMAC配置

以下是全志的 GMAC 配置，驱动源码bsp/drivers/gmac。

以下是 GMAC 的主要属性：

  • 属性 phy-mode：指定 PHY 的驱动模式，这要根据硬件原理图去配置。一般来说，千兆网卡使用“rgmii”，百兆网卡使用“rmii”。
  • 属性pinctrl-*：GMAC 与 PHY 芯片连接的 IO 复用。
  • 属性aw,soc-phy25m和aw,rgmii-clk-ext：指定网卡的时钟源，一个是指定使用 SoC 引出的时钟源，另一个是指定使用外部晶振的时钟源。两者只能选其一去使用，需要根据硬件原理图去配置。如上述的 gmac0 节点，指定使用 SoC 时钟源，不用外部晶振。
  • 属性tx-delay和rx-delay：这是数据传输和接收时用到的延迟，如果在进行压力测试时，发现数据传输或接收时出现丢包情况，则稍微调整一下传输和接收的延迟。
  • 属性dwmac3v3-supply和phy3v3-supply：这两个分别指定 IO 和 PHY 的电源域，根据硬件原理图去配置。

以下是 ETH PHY 的配置，一般使用内核自带的通用 PHY 驱动即可。如果厂商有自己的驱动，则建议把这些驱动放到 bsp/drivers/net/phy，如裕太微的 PHY 驱动 motorcomm.c 。

以下是 PHY 的主要属性：

  • 属性compatible：这是与 PHY 驱动匹配的关键属性，尤其是使用了 PHY 厂商的驱动。格式为：ethernet-phy-id + ID for PHY Chip，上述的属性值包含的“4f51.e91b”就是 PHY 芯片的 ID 号，一般是“厂商ID.设备ID”。
  • 属性reset-gpios：PHY 芯片的复位脚，根据硬件原理图去配置。
  • 属性reg：PHY 芯片的地址，根据硬件原理图去配置。
  • 属性max-speed：PHY 芯片的最大速率，根据 PHY 芯片的型号确认。
  • 属性reset-assert-us和reset-deassert-us：PHY 芯片的复位延迟，一般不用关心。

LocalBus配置

以下是 LocalBus 的配置，相关源码路径为bsp/drivers/lbc。

以下是 LocalBus 的关键属性：

  • 属性clock-frequency：LocalBus 的时钟频率，需要与 LocalBus 连接的从设备确认。
  • 属性transfer_mode：为 0 时表示使用直接传输模式；为 1 表示使用间接传输模式；为 2 表示使用 DMA 传输模式。其中直接传输模式是指定一个内存区域去读写数据；间接传输模式是通过 LocalBus 主控芯片的寄存器去读写数据。
  • 属性transfer_width：为 0 表示数据宽街为 8 位，为 1 表示 16 位，为 2 表示 32 位。
  • 属性burst_mode：表示是否使用激进（burst）模式，为 0 默认不使用。
  • 属性chain_delay：这个仅有在非 DMA 模式下使用，传输数据之间的间距延迟。
  • 属性lbc-io-supply和lbc-io-vol：LocalBus 的 IO 电源域，这个根据硬件原理图确认。
  • 属性pinctrl-*：LocalBus 主控与 LocalBus 从设备之间的 IO 复用。

另外，目前对 LocalBus 的开发更多针对的是：LocalBus->FPGA->AD7616 。有些 AD7616 的功能引脚（ Convst、Reset 和 Busy ）并不能与 FPGA 连接，转而使用 CPU 去控制这些引脚：
