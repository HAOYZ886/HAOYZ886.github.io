Linux - KGDB调试（基于ARM64）

Posted on 2025-01-26 Edited on 2025-12-24 In Skill Word count in article: 12k Reading time ≈ 42 mins.

前言

此文档仅在 Linux 6.1.55 验证，在 IMX8MP 和 RK3576 验证可行，理论上 Linux 5.10 以及以上的版本均适用。

KGDB

什么是KGDB

GDB 就是调试代码的工具，而KGDB就是 Kernel GDB ，即用于调试内核的 GDB 。这个机制可以调试内核的启动过程，但是需要内核启动一段时间后才能使用，但对于调试驱动来说已经足够了。后续会详细说明它在启动到哪个阶段后，暂停并等待 Server 接入。

  1. 也可以等内核启动完成后，再单独调试驱动模块，在如何调试内核模块说明。
  2. 网上还有很多关于它的介绍以及前后今生，这里就不赘述了。

测试端配置（Linux 配置）

使能KGDB的内核配置

开启以下配置，使其支持 KGDB ：

  • CONFIG_KGDB：开启 KGDB 调试
    • CONFIG_KGDB_SERIAL_CONSOLE：使用串口把 Server 端和测试端连接起来，此外还需要使用内核参数配置来支持这个功能，后面会详细说明。
  • CONFIG_DEBUG_INFO_DWARF_TOOLCHAIN_DEFAULT：使用交叉编译链默认的 DWARF ，这个是生成 DEBUG 调试信息的格式。目前最新的是 DWARF5 ，可使用 CONFIG_DEBUG_INFO_DWARF5 来代替此配置项，但这里推荐使用交叉编译链默认的。
    • CONFIG_DEBUG_INFO：开启生成 DEBUG 信息，开启 CONFIG_DEBUG_INFO_DWARF_TOOLCHAIN_DEFAULT 等类似的配置项时，默认开启此配置，因为这些配置项依赖于它。
  • CONFIG_DEBUG_KERNEL：这是默认开启的，因为很多 DEBUG 配置项都依赖于它。

额外配置

以下额外配置都是确保，在调试内核的过程中尽可能地不能被外来因素打断。

随机地址

把内核的虚拟地址随机化关闭，虽然这是安全机制，但是在 GDB 不需要，因为可能会导致 GDB 无法正常运行。

目前已知的问题

目前已知的 KGDB 限制：

  1. 平台的不同，支持硬件断点和 watch 断点的程度也不同，可能需要用到 JTAG 才能正常使用。
  2. 平台和内核版本的不同，支持软断点的程序也不同，但至少仅支持临时软断点。有些平台（如 IMX8MP ）如果直接使用 breakpoint ，当执行到断点时，使用 continue 命令时，还是会卡在当前断点出不来。
  3. 如果运行多线程时，进入到某个断点后，会让所有线程停止，并不支持指定线程卡在断点，其它线程继续运行。

移除优化参数编译内核

此章节的目的是不使用 -O2 ，而是用 -O0 或 -Og 来编译内核，尽可能地保留更多的调试信息，以及减少编译警告。
使用 -O0 或 -Og 来编译内核会遇到很多困难，这是因为内核开发者使用 -O2 的特性来开发内核。比如说带有 BUILD_* 的宏，比如 BUILD_BUG_ON() ，这些宏是依赖于 -O2 的常量传播和死代码消除实现。

回顾刚刚的报错信息，可知是 __bad_mask() 引起的，这是为什么？
回顾《解决来自alternative-macros.h报错》，使用优化参数编译代码，会把内联函数在调用处展开，而且我发现调用 u8_get_bits() 的参数 field 必须是常量，则这意味着编译器会记录该数值，并且试着预先执行 if ((field | (field - 1)) & ((field | (field - 1)) + 1)) ，如果为真，则执行 __bad_mask(); ，从而在编译期间发出 “bad bitfield mask” 。所以，从中得到的信息是，__bad_mask() 就是一个“假函数”，真正作用是在编译期间发出的 __compiletime_error("bad bitfield mask") 。
但使用 -O0 时，编译器不会展开内联函数，也不会预处理 if ((field | (field - 1)) & ((field | (field - 1)) + 1)) ，仅仅是把内联函数和 __bad_mask() 当作普通函数一般。所以处理 __bad_mask() 时，直接认为调用这个函数是错误的，直接发出 “bad bitfield mask” 。

点击这里，查看 compiletime_error 的作用。

以下是具体的解决方法：

把 __bad_mask(); 替换为 panic("bad bitfield mask"); ，既然用了 -O0 没办法让编译器预先执行，就改用为运行时再检查 field 是否正确。虽然这带来了一定的性能损耗，但起码也能在运行期间清晰地知道内核做了什么。

解决来自nvme/host/pci.c的报错

以下是出现报错的位置：

为什么会报错？先看函数 nvme_pci_npages_prp() 的实现：

如果使用 -O2 编译，会把该函数优化为一个常量，因为该函数内的值都是常量。再延续到 DIV_ROUND_UP(nvme_pci_npages_prp(), NVME_CTRL_PAGE_SIZE) ，自然也是常量。

这是 -O2 的优化方向：常量传播和死代码消除。积极使用常量“预先”运行代码（常量传播），减少代码执行流的条件分支（死代码消除）。

但如果用 -Og 编译，为了保证调用栈完整，不会把函数内联和移除，所以函数 nvme_pci_npages_prp() 不会被优化为常量，而是继续保持与源代码中表示的一样。
对于 BUILD_BUG_ON(condition) ，在 -Og 环境中是生效的。但是函数 nvme_pci_npages_prp() 没有被“展开” ，无法在编译期间参与 DIV_ROUND_UP() 计算，则 GCC 编译器只能认定这个 condition 为 True ，所以中断编译并发出报错信息。
