指令重排序

特性

  • 特点：是指通过编译器 -O2 或 -O3 优化，使得让访问内存（变量）的次数减少，把变量值存储在寄存器内，读写操作在寄存器进行。
  • 缺点：对于 SMP 系统，在多核共享同一个变量的情况下，如果变量值一直存储在寄存器内，导致内存一致性无法得到保证。
  • 解决：用关键字 volatile 修饰变量；或在关键代码处（需要查看经过优化后的汇编代码）添加 asm volatile("" ::: "memory") ，使得编译器不对该处代码优化。
  • 新的问题：这样已确保变量值都写到内存，但无法保证内存访问的顺序一致性。

寄存器布局：

  • 寄存器 r12 存储着数组 flags 的基地址；
  • 寄存器 rdi 存储着函数 thread_f() 的实参 rank ；
  • 寄存器 rbp 是变量 rankB ；
  • 寄存器 rbx 是变量 rankA 。

在以标签 .L3 起始的代码段，

  • 把 rbp 赋值为变量 turn ；
  • 把立即数 1 赋值到 flags[rankA]，对应的内存地址为（%r12,%rbx,8）；
  • 把 flags[rankB]（%r12,%rbp,8）读取到 rax

此时，如果 rax 不为 0 ，则跳转到标签 .L2 。
这时会意识到，条件判断 turn == rankB 对应的汇编代码哪里去了？细看一遍汇编代码，发现竟然被优化掉了，汇编代码内仅有条件判断 flags[rankB] ，当双方的 flags[rankA] 都设为 1 时，导致发生死锁，所以就显示仅有 Thread 0 进入了临界区。
为什么会“优化”条件判断 turn == rankB ？其实原因很简单，编译器先看到了 turn = rankB ，也看到 turn == rankB ，会自然地认为 turn == rankB 是“没用的”，就把它优化掉了。因为在它眼里看来，turn 永远都是 rankB ，也从未考虑到还有其它的执行流（编译器还没那么智能吧？）。

而且上文也讲过，-O2 和 -O3 的最大优化点就是减少内存访问操作，转而用大量的寄存器去操作。自然它也会把它自认为“冗余”的访问内存操作代码优化掉，这也导致并行编程中的共享内存的数据不一致。

如果你使用过 GDB 去 debug 一个用 -O2 优化的程序时深有体会，想查看函数内的某个局部变量时却被提示“该变量已被优化”。这也是 -O2 的“功劳”，它直接把这变量放到寄存器了。也从中体会到想调试 linux 内核最好还是先把它的 -O2 优化移除。

以下有两种解法：

  1. 给变量 turn 加上关键字 volatile ，告知编译器该变量是易变的，不能每次的读写操作都要在内存进行。

其中，asm 表示内联汇编，关键字 volatile 修饰该内联汇编不能被优化，因为汇编模板内无任何汇编代码，编译器会忽略它。
而 “memory” 表示当前内联汇编会涉及到内存修改，告知编译器不能对内存访问操作做优化。
