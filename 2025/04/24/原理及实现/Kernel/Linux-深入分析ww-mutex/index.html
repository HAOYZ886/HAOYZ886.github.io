Linux - 深入分析ww-mutex（内核版本 6.1.55）

Posted on 2025-04-24 Edited on 2025-05-04 In 原理及实现 Word count in article: 10k Reading time ≈ 37 mins.

参考

Documentation/locking/ww-mutex-design.rst

发现

在阅读 DRM 源代码时，发现使用了一个特别的锁，名为 ww-mutex ，该锁是 mutex 的“升级版”。ww-mutex 的全称是 wound-wait mutex。
在 DRM 中，如 drm_helper_probe_single_connector_modes() 和 ATOMIC COMMIT 操作等，调用 drm_modeset_lock() 和 drm_modeset_drop_locks() 去获取和释放 drm_modeset_lock 。drm_modeset_lock 的基础对象就是 ww-mutex 。

为什么需要ww-mutex

其实 ww-mutex 的由来很简单，就是预防死锁。出现死锁的场景有很多，就以互斥锁 mutex 来说，最典型的就是 AB-BA 死锁场景：

执行流程：

  1. Task1 获取了 MutexA
  2. Task2 获取了 MutexB
  3. Task1 尝试获取 MutexB，但已被 Task2 获取，等待中
  4. Task2 尝试获取 MutexA，但已被 Task1 获取，等待中

由此可知，Task1 和 Task2 都在获取对方已获取的锁，从而导致死锁。
如果使用了 ww-mutex ，以上的死锁场景就能解决，在《ww-mutex的原理》说明它的原理。

哪些实际的使用场景会出现以上死锁场景？

  1. 假设 GPU 有多个 buffers ，每个 buffer 对应一个互斥锁来保护，都能被 Tasks 使用。但是各 Task 之间访问 buffers 的顺序可能是不一样的，从而导致死锁。
  2. 假设两个设备（如 GPU 和 Camera）需要两个及以上的共享 buffers (基于 dma-buf，有锁保护) ，如果各设备驱动之间访问 buffers 的顺序不一样，也会产生死锁。
  3. 在 DRM 中，常见的 ww-mutex 有 plane 的 mutex 、CRTC 的 mutex ， bridge 的 mutex ，和 connection_mutex 等。在一次 ATOMIC COMMIT ，都会调用 drm_atomic_get_xxx_state() 去获取组件的状态，当第一次获取时，会复制当前的状态，延续当前状态作为新的状态，期间会获取对应的锁。如果期间有其它进程调用 drm_helper_probe_detect() -> connector->helper_private->detect_ctx() ，如果 hang 函数 detect_ctx() 有涉及到获取其它组件锁的操作，这也有死锁的风险。

ww-mutex的原理

概念

特性

根据 ww-mutex 的原理，先讲一两个关键特性：

  • 当有一个锁被获取时，有两个及以上的 Tasks 在等待同一个锁时，应让最先等待的 Task 优先获取。
  • 当双方 Tasks 处于死锁状态时，让两者之中最年轻的 Task 先“死”（返回错误码 -DEADLOCK ），并释放自身所有已持有的锁，然后再一直尝试获取（等待）此前还未获取的锁。

ww-mutex 的关键点，就是任务之间的生命周期（包含出生日期），以及竞争锁期间的顺序必须是 FIFO 。
系统要管理多个任务的资源以及生命周期，一个任务应当在合理的时间范围内完成（这个时间范围由开发者定义），特别是需要持有多个锁的任务。
在竞争锁的过程中，是否能有一个简单的确定性？就是哪个 Task 先竞争，就先让它先获取到锁？这是必要的。如果没有，当锁释放后就“乱成一窝粥”，则使得先竞争的任务可能得不到更快的执行，还有老的任务无法得到执行。
另外，出现死锁时，两者之间，应该让年轻任务退出，让老任务先行。因为老任务因为占用系统资源比年轻任务久，需要尽快释放资源，这样才能“循环利用”。
还有，通常来说，老任务持有锁的个数比年轻任务多，也比年轻任务早持有，老任务也应当尽快完成（也可选择改用 Wait-Die 算法）。

ww-mutex 与 mutex 相比，新增了一些功能。

  • acquire_context：这是防死锁的关键特性，用于记录当前 Task 获取/释放锁的上下文（过程）。它对应的结构体是 struct ww_acquire_ctx 。结构体内有两个关键的字段，第一个字段 stamp ，用于标识当前 Task 的“出生日期”，数值越大表示越“年轻”；第二个字段是 acquire，表示在当前执行流内获取了多少个锁（该字段的数值区别只有 0 和 >0 ）。
    当一个 Task在获取一系列的锁之前，都会把它初始化。初始化期间初始化字段 stamp ，记录当前 Task 的“出生日期”，还有把字段 acquire 初始化为 0 。
  • class：用于标识当前 acquire_context 防死锁的算法是什么。一般分为两种，一是 Wound-Wait ，二是 Wait-Die ，两者只能选一种。在初始化 acquire_context 时，就要指定用哪个算法。本文更多关注 Wound-Wait ，理解 Wound-Wait 后，自然也会快速理解 Wait-Die 的原理。

准确来说，acquire_context 的 stamp 其实是指当前 acquire_context 的 stamp_id ，通过调用 atomic_long_inc_return_relaxed(&ww_class->stamp); 获取。本文中提到的老 Task 和年轻 Task 是为了方便理解和说明，但意思是不变的，只是范围缩小了，是 acquire_context 的“出生日期”，范围仅在 acquire_context 所属的 ww_class 内。

以下是这两个算法的解释（from ww-mutex-design.rst）
Wound-Wait ：

  • If the transaction holding the lock is younger, the locking transaction wounds the transaction holding the lock, requesting it to die. If the transaction holding the lock is older, it waits for the other transaction. Hence Wound-Wait.
    Wait-Die ：
  • If the transaction holding the lock is younger, the locking transaction waits. If the transaction holding the lock is older, the locking transaction backs off and dies. Hence Wait-Die.

这两个算法都是相互反着来的，但是本质都是，发生死锁时，让老的 Task 先行，让年轻的 Task 重新再来。对于 Wound-Wait ，是老的 Task 提前通知它：你一到 waiting 就 go die ；对于 Wait-Die ，是年轻的 Task 意识到自己“大限将至”，主动 go die 。
这两个算法最大的区别是,当 Task 获取不了锁等待时：

  • Wound-Wait ：加入该锁对应的等待队列，该队列不分老小，按 FIFO 。获取锁时也是按哪个 Task 先竞争这个锁，就谁获取。
  • Wait-Die ：加入该锁对应的等待队列，但是如果当前 Task 的 ctx->stamp 比队列中任何一个 task 大，则直接返回 -DEADLOCK 。

还有，ww-mutex 与 mutex 相比，在获取锁之前和释放所有锁后，还有额外的操作。

  • 在获取锁之前，初始化 acquire_context ，调用 ww_acquire_init() ，表示当前持有 ww-mutex 的执行流开始。
  • 释放所有锁后，调用 ww_acquire_fini() ，表示当前持有 ww-mutex 的执行流结束。

用例子来理解概念(获取锁阶段)

下面用两个实例初步理解 ww-mutex 的特性。
为了方便理解，假设防死锁的算法是 Wound-Wait ，实例图中的 Task1 的运行时间比 Task 早，而且 Task1 比 Task2 先获取锁。
则目前的 class 指定的是 Wound-Wait ，Task1 和 Task2 的 acquire_context 用该 class 初始化。而且 Task1 的 acquire_context->stamp 比 Task2 的小，即 Task1 的“出生日期”比 Task2 的早（假设 Task1 的 stamp 为 1 ，Task2 的 stamp 为 2）。它们两的 acquire 初始化都初始化为 0 。

后面会把 acquire_context 简写为 ctx 。

实例一

流程 1 ：

  1. Task1 获取到 MutexA ，此时它对应的 ctx->acquire 为 1 。
  2. Task2 获取到 MutexB ，此时它对应的 ctx->acquire 为 1 。

流程 2 ：
Task1 尝试获取 MutexB ，但是它被 Task2 获取了，只能等待。
接下来会有两种情况：

  1. Task2 释放 MutexB ，Task1 就能获取到 MutexB 。
  2. Task2 获取 MutexA ，但它已被 Task1 获取，只能等待。

针对第二种情况，这已经是死锁了，这时需要根据 ww-mutex 的特性进行解开死锁。
已知 Task1 的 ctx->stamp 为 1 ，Task 的 ctx->stamp 为 2 。另外，此时 Task1 的 ctx->acquire 等于 1 。如果 ctx->acquire 大于 0 ，表示当前 Task 已持有一到多个锁，如果有其它 Task 也尝试获取这些锁时，必然会等待。那么如果它也获取不了锁也陷入等待，这就造成了死锁。但是当前 Task 并不知道谁在等待它持有的锁，而且等待锁的其它 Task 也未必也持有了其他锁。
根据 Wound-Wait 算法可知，Task1 能做到的就是“主动出击”，先提前预判持有 MutexB 的 Task2（比它年轻）是否会也陷入等待其它锁（如 Task1 持有的 MutexA ），先打个“预防针”，意思是把 Task2 的 ctx 标记为 Wound 。

实例二

流程 1 ：

  1. Task1 获取到 MutexA ，此时它对应的 ctx->acquire 为 1 。
  2. Task2 获取到 MutexB ，此时它对应的 ctx->acquire 为 1 。

流程 2 ：
Task2 尝试获取 MutexA ，但是它被 Task1 获取了，只能等待。
接下来会有两种情况：

  1. Task1 释放 MutexA ，Task2 就能获取到 MutexA 。
  2. Task1 获取 MutexB ，但它已被 Task2 获取，只能等待。

第二种情况是死锁，和实例一的解法一样。已知 Task1 的 ctx->stamp 为 1 ，Task 的 ctx->stamp 为 2 ，Task1 的 ctx->acquire 等于 1 。
根据 Wound-Wait 算法可知，解死锁是由老的 Task 来主导，把 Task2 的 ctx 标记为 Wound 。此时 Task2 在等待 MutexA ，期间也在等待它的“死期” ，即每一次的循环等待，都会去读自己的 ctx->wound 是否为 true ，是则返回 -DEADLOCK ，见下面的流程 3 。

对于 Wait-Die 算法，当 Task2 获取 MutexA 失败并等待时，发现自己的 ctx->acquire 大于 0 ，而且 ctx->stamp 比持有此锁的 Task 的大时，会直接返回 -DEADLOCK 。

以上是第二种情况的流程 3 ：
Task1 给 Task2 的 ctx 被标记为 Wound 后，当它等待 MutexA 时，发现自己的 ctx 被标记为 Wound ，则不等待锁，直接返回错误码 -DEADLOCK 。然后会释放所有已持有的锁，如 MutexB ，再尝试获取锁 MutexA 并等待…
当 Task2 释放 MutexB 后，Task1 会获取到 MutexB 。
当 Task1 按顺序释放 MutexB 和 MutexA ，Task2 就能获取 MutexA 和 MutexB 。

看完以上两个实例后，或许会发现一个细节，就是当 Task 返回 -DEADLOCK 后的获取锁的顺序和一开始的顺序不一样。当某个 Task 因获取某个 Mutex 失败返回 -DEADLOCK 时，释放所有的已持有的锁，再尝试获取刚刚获取失败的 Mutex ，获取不了就加入到等待队列，遵循先进入出的原则获取锁。
另外，如果获取到锁了，就继续按之前的顺序再次获取多个锁。如果到先前已获取到的锁时，会返回 0 。

为什么需要标记为 Wound ？
因为你“侥幸”夺走了锁，这个锁可是等待队列中的“老东西”所心心念念的，如果你把握不住（发生死锁）就把锁给回“老东西”吧，还要“散财”（释放已持有的所有锁）以示诚意。
那么为什么能够“侥幸”获取到锁？这需要在看 slowpath 才能知晓。

  1. 当前 Task 被标记为 NEED_RESCHED ，即稍后将被调度了（一般在内核层切回用户层期间，或在中断期间）。
  2. 持有该锁的 Task 不在任何一个 CPU 运行中，或被标记为 NEED_RESCHED 。
  3. 当前 Task 的 ctx->acquire > 0 ，与持有该锁的 Task 之间可能出现死锁。
  4. 如果该参数 waiter 为 NULL ，但该锁有其它的 waiter(s) ，则立即跳出循环，因为应优先让在等待队列内的 Task 获取。
  5. 如果该参数 waiter 不为 NULL ，但不是第一个竞争的 waiter ，则立即跳出循环，因为等待队列的 FIFO 原则。

而且需要注意，在死循环中的最后一步是调用了 cpu_relax(); ，目的是减少 CPU 功耗，减缓死循环的运行。

总结（基于Wound-Wait）

总得来说，以上的讲解更多围绕竞争锁，可见从竞争中获取一个锁是很不容易的。但是相比杂乱无序，有了一个很明确的竞争规律：

  1. 优先给等待队列中的第一个 waiter 获取到锁
  2. 其次可以是比第一个 waiter 年纪大的 Task 获取到锁，
  3. 可以“侥幸”获取到锁，但如果比等待队列中任何一个的 task 年纪小，就标记为 wound 。

对于死锁：
让老 Task 通过对年轻 Task 的 Wound 标记，来解开死锁。而且必要时，还要唤醒该 Task 来尽快处理此标志，释放锁，给老 Task 用。

还有一些疑问：
为什么获取锁分两路走？fastpath 和 slowpath ？
看了 fastpath 和 slowpath 的实现可知，fastpath 路径更多是针对没有竞争锁的情景下使用；slowpath 则是针对竞争锁的情景下使用。

那么为什么能够“侥幸”获取到锁？
如果 Task1 运行在 slowpath ，Task2 运行在 fastpath ，Task3 持有该锁，当 Task3 释放后，Task2 获取到锁，而 Task1 获取不了。看 slowpath 的实现可知，有一部分代码实现不是“获取锁”操作，从而被处于 fastpath 的 Task2 钻空子，获取到锁。而且在尝试获取锁的死循环中，用到了 cpu_relax(); ，使得 CPU 功耗降低，循环减缓，导致处于 fastpath 的 Task 有了可乘之机。

ww-mutex的实际运用（DRM）

在 DRM 中，drm_modeset_lock 是基于 ww-mutex 实现。这是因为 DRM 的组件，plane 、CRTC 、bridge 和 connector 等，都有各自的锁。而且每一次操作未必就是每次都按 connector->bridge->encoder->crtc->plane 的顺序操作，还有针对个别组件，如 plane 和 CRTC 去操作，驱动中也有一些操作不按套路出牌。
例如，在一次 ATOMIC COMMIT ，都会调用 drm_atomic_get_xxx_state() 去获取组件的状态，当第一次获取时，会复制当前的状态，延续当前状态作为新的状态，期间会获取对应的锁。如果期间有其它进程调用 drm_helper_probe_detect() -> connector->helper_private->detect_ctx() ，如果 hang 函数 detect_ctx() 有涉及到获取其它组件锁的操作，这有了死锁的风险。
