相关技术的简述

Linux的调度策略

先了解一下 Linux 的调度策略：

  • SCHED_DEADLINE ：优先级最高的调度策略，没有时间片机制，仅用在内核线程。
  • SCHED_FIFO ：先进先出调度策略，没有时间片机制，高优先级的任务优先执行，低优先级的任务会被高优先级的任务抢占，当睡眠或 IO 操作才会主动让出 CPU 。
  • SCHED_RR ：优先级实时调度，有时间片机制，当时间片为 0 后让出 CPU ，但只有在相同优先级下进行时间片调度。
  • SCHED_NORMAL 和 SCHED_BATCH ：Fair 调度策略，有时间片机制，当时间片为 0 后让出 CPU ，通常用于批处理任务，可以使用 nice 值调整优先级
  • SCHED_IDLE ：空闲调度策略，当 CPU 无任何任务调度后，调度 IDLE 进程，进入低功耗模式。

其中：

  • SCHED_FIFO 和 SCHED_RR 的实时优先级的范围是 1~99 ，数值越大表示优先级越高。
  • SCHED_NORMAL 和 SCHED_BATCH 的优先级通过 nice 值调整，nice 值的范围是 -20~19 ，数字越小表示优先级越大。

另外，在这些调度策略中，有一个机制来表示这些调度策略之间的优先级，它就是 PRIO ，它的范围是 -1~139 ，数字越小表示优先级越高。其中：

  • -1~0 代表 SCHED_DEADLINE ；
  • 1~99 代表 SCHED_FIFO 和 SCHED_RR ；
  • 100~139 代表 SCHED_NORMAL 和 SCHED_BATCH 。

这些优先级范围的定义和计算在：include/linux/sched/prio.h 和 include/linux/sched/deadline.h

RT节流（RT throttling）

参考以下文档,了解 RT throttling 机制：

  • https://docs.redhat.com/en/documentation/red_hat_enterprise_linux_for_real_time/7/html/tuning_guide/real_time_throttling
  • https://developers.redhat.com/articles/2025/03/26/rhel-real-time-cpu-throttling-and-risks#diagnosing_the_latency_problem
  • https://pwl999.github.io/2017/12/16/linux_scheduler/

变量 sched_rt_runtime_us 和 sched_rt_period_us 都是属于 RT throttling 机制的一部分。
其中，变量 sched_rt_period_us 定义了实时任务的时间周期，变量 sched_rt_runtime_us 定义了实时任务在 sched_rt_period_us 内的可运行时间周期。
一般来说，变量 sched_rt_runtime_us 为 950000，变量 sched_rt_period_us 为 1000000 。意思是在一秒的时间，有 0.95 秒的时间给实时任务执行，剩余的 0.05 秒给非实时任务。

看门狗的喂狗流程

在 watchdog 框架的初始化阶段，调用 watchdog_dev_init() ，创建名为 watchdogd 的内核工作线程，调度策略为 SCHED_FIFO ，实时优先级为 50 。
在 watchdog 驱动的初始化阶段，必须为它的设备描述符 watchdog_device 的字段 max_hw_heartbeat_ms 设置数值，表示喂狗的间隙。如果开启了配置项 CONFIG_WATCHDOG_HANDLE_BOOT_ENABLED 来指定初始化后就需要喂狗，立即使用 hrtimer 准备喂狗，要求在中断上下文调用回调函数（模式为：HRTIMER_MODE_REL_HARD），其回调函数是 watchdog_timer_expired() ，以下是它的实现：

尝试解决

把sched_rt_runtime_us设为-1

然后应用正常启动，外部看门狗正常启动，测试一天，未发现重启，而且没有出现 “RT throttling” 报错。

但如果把 sched_rt_runtime_us 设为 -1 ，则表示 FIFO 任务则可以永远占用 CPU ，不给其它策略的进程占用。这会导致一个问题，如果 FIFO 任务因为某些原因处于死循环阶段，则出现一直占用着 CPU ，不让出 CPU 给其它任务执行，最终可能会导致系统崩溃。

而且这里出现一个疑问，为什么把 sched_rt_runtime_us 设为 -1 就能让 kwatchdog 正常执行？

隔离一个核

把 sched_rt_runtime_us 还原，现在怀疑是因为 RT 节流导致 kwatchdog 线程无法得到执行，而且它的实时优先级比应用程序的高。现在把它的 CPU 亲和性设置为 CPU3 ，即仅在 CPU3 运行。
但经过测试发现还是会重启。

应用手动喂狗

让客户在应用层新建一个线程用于喂狗，并放在隔离核上。发现还是会重启，但这是不应该的，大概率是客户的应用程序没有写好。

把sched_rt_runtime_us改为990000

把 sched_rt_runtime_us ，从 950000 改为 990000 ，经过测试发现也会重启。

默认喂狗的间隔为 500ms ，则把间隔改为 300ms ，但经过测试也会重启。

最终的解决方法

把应用程序的FIFO调度策略改回Normal调度策略

客户考虑实时性没有考虑采用。

把sched_rt_runtime_us设为-1（要求应用健壮性高）

经过测试是可以解决这个问题，但我们无法了解客户应用程序的全貌，真的无法查明问题所在。而且由于时间关系，把它设为-1，仅经过一天时间去测试，还未必就是可以的，起码要 7x24 小时测试。

额外定时器去喂狗（最终选择）

这个方法只是绕过任务调度的问题，因为客户不会提供他们的应用程序。
因为 SoC 有额外的定时器，则可以在对应的驱动中，实现直接在中断处理相关任务，不分上下两部分，当时间一到就执行喂狗操作。
