Linux - 任务调度（内核版本 2.4.22，32位x86）

Posted on 2022-12-30 In 原理及实现 Word count in article: 4k Reading time ≈ 15 mins.

任务调度的对象

任务调度中的任务，按类型大致分为三种：

  1. 交互式任务，该类型的任务有一定的实时要求，起码不能让操作该任务的用户感觉到有延迟，且对执行时间的长度要求不高。最典型的例子就是 bash（见下图）。
  1. 后台任务，又称为批处理应用，该类型的应用是没有严格的实时要求，但是它需要相对较长的时间去执行。详细地说，就是这种类型的应用的时间片要求是比较大的，而且它要在“理想”的时间间隔内，得到 CPU 的使用权。所以这种任务有一个指标：平均速度，而平均速度包含：在调度队列内多久才轮到它执行？以及从调度后的执行时长又是多久？
  2. 实时任务，该类型的任务有强烈的实时要求，而且也要考虑到它执行的时间是否很快？是否达到一个“平均速度”？它程序的本身是否很庞大？最典型的例子就是，当你在 bash 输入一个命令并回车，而你输入的命令对应的程序就是一个实时任务，如上图已输入的命令 ls 。

根据这些类型的任务，新增加一个功能：根据不同类型的任务赋予不同的调度策略。调度策略有：SCHED_RR 、SCHED_FIFO 、SCHED_OTHER 。

from sched.h

  • SCHED_OTHER ：其他类型的任务，如：交互式任务、idle、内核线程 init 。
  • SCHED_FIFO ：要求响应时间快，且执行时间短的任务（实时任务）。
  • SCHED_RR ：执行时间较长的任务（后台任务）。

调度方式

任务调度的方式分为两种，一是主动调度，二是被动调度。
主动调度就是任务在执行它自己的主执行流时（指的是无意外情况），执行流内存在任务调度函数 schedule() 。
被动调度是有条件被动调度，当满足某一条件时，内核会强制调用 schedule() 把当前任务换下，把 CPU 的执行权给下一个任务。

被动调度设置在哪里？有一个被动调度设置在用户任务通过中断/异常进入到内核层后，再从内核层回到用户层时，检查当前任务是否可调度。前面刚刚说了被动调度是有条件的，最典型的例子，就是实时中断。当实时时钟的计数器0计数完成后发出一次中断，发出的间隔为 10ms 。每个任务都有一个时间片，实时中断一次，时间片减一次。时间片为 0 时，设置该任务的状态为可调度，该任务从内核层返回用户层前夕，检查此标志位，该标志位被置1表示要调度。先列出该被动调度的代码：

那么主动调度和被动调度有哪些？
主动调度：

  • 睡眠，指的是主动调用有关 sleep_*() 的函数，可设置睡眠时间。
  • 让出，指的是调用函数 yield() ，让出 CPU 使用权给其他任务。

被动调度：

  • 当任务的时间片被时钟中断减为0，则调度。
  • 竞争互斥锁失败时睡眠等待，任务获取不了锁（这是条件）只能被迫睡眠。
  • 信号，当然，这不是泛指全部信号，而是指某一个类型的信号。举个例子，当我们在命令行终端输入一个命令执行程序，在执行过程中，我们可以输入快捷键“ctrl+z”，把当前进程挂起，我认为这也是主动调度，因为我“手动”干预了，也是有条件的被动调度。
  • 唤醒一个任务，把当前任务和刚唤醒的任务对比权值，当前任务的比它的小（条件）则调度。
  • 设置任务的调度策略（后面会知道策略是什么），当设置的任务调度策略比当前任务的高（条件），则调度。

另外，主动调度和被动调度是和调度策略挂钩的。调度策略为 SCHED_FIFO 的任务是不会被强制调度的，因为它的实时要求高，且执行时间短^1。调度策略为 SCHED_RR 或 SCHED_OTHER 的任务即可主动调度，也可被动调度。

任务调度和任务运行状态之间的联系

调度队列内的任务都是 TASK_RUNNING 状态，当我们对某一个任务设置睡眠或追踪时，该任务的运行状态就要设置为非 TASK_RUNNING ，如 TASK_INTERRUPTIBLE 、TASK_STOPPED 等等。并且要从调度队列移除放到其他队列、或者其他存储空间存储该任务对应的 PCB 地址。
比如，我们要对当前任务进行浅度睡眠（可被信号打断），调用函数 interruptible_sleep_on() 。执行该函数的过程中，首先对该任务的运行状态设置为 TASK_INTERRUPTIBLE ，然后把该任务的 PCB 地址保存（一般是用一个队列保存的），接着调用 schedule() ，期间会把该任务从调度队列移除，然后切换下一个权值高的任务执行。

检查该函数是否在执行中断期间执行，是否打出 BUG 信息。因为该函数用于切换任务的，换句话说是切换任务的执行流。虽然当前任务因中断切换到了中断处理程序这个执行流，但从内核的角度来说，当前执行的中断处理程序也是属于当前任务的执行流。所以切换新任务后，该中断处理程序就不能继续执行了，只能等再次切换到此任务后再次执行。这是一个非常致命的错误，因为在中断期间所处理的数据是必需随着中断处理程序的实时启动而实时更新的，简单来说就是有很强烈的时间要求，不能怠慢。在中断期间调用任务调度，是很不合理的行为。

首先对寄存器 esi 、edi 、ebp 压栈，原因是 esi 和 edi 是数据流指令的指定寄存器；ebp 是作为在栈内的函数实参和局部变量的基地址。而其他寄存器，如 eax 、ebx 、ecx 和 edx，会根据实际情况压栈保存。

如何根据实际情况存储的呢？
以 gcc 编译器把 C 源码编译为汇编指令为方向，以用 C 源码定义的函数出发。
已知在 i386 平台内， eax 是基础寄存器（访问内存最为频繁的）；ebx 是基地址寄存器；ecx 是计数寄存器；edx是给 eax 辅助运算的寄存器；ebp 和 esp 是栈操作寄存器；esi 和 edi 是数据流寄存器（分别是源地址寄存器和目的地址寄存器）。gcc 也是支持 i386 的寄存器使用标准。C 源码的函数的局部变量以及实参都在栈内（除非用寄存器传参），且用 ebp 作为在函数内、在栈内局部变量以及实参的基地址。C 源码通过 gcc 编译为汇编指令后，可以发现，eax 是使用最频繁的寄存器，而且是访问内存最为频繁的寄存器（更多的像 CPU 访问内存的中转站一样）。 gcc 还会根据函数与函数之间寄存器的使用情况，把寄存器压栈保护。比如函数 A 和函数 B 都使用 ecx ，函数 A 调用函数 B ，但函数 A 调用前后还需要 ecx ，所以执行函数 B 时，会把 ecx 压栈，返回前出栈。
另外再说一句，如果使用内联汇编，内联汇编所用到的寄存器会在执行内联汇编的指令前压栈保护。

__switch_to 是一个函数，但目前作用仅有一个，就是作为一个跳板。用该函数返回后就是下一个任务的执行流了。
