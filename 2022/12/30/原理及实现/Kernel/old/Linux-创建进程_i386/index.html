Linux - 创建进程（内核版本 2.4.22，32位x86）

Posted on 2022-12-30 In 原理及实现 Word count in article: 3.9k Reading time ≈ 14 mins.

进程是什么？

能被 CPU 执行的一个执行域，它有自己的私人内存空间，即它有自己的代码段、数据段、堆栈等等。而且它还有一个能被操作系统调度切换的 PCB ，以及拥有自己内核栈。

进程由什么组成？

进程由四个部分组成

  1. 拥有能被 CPU 执行的执行域，该执行域可私用，也可共享。
  2. 拥有能被操作系统识别的调度任务的单位 PCB ，该 PCB 还存储该任务的状态。
  3. 拥有自己的内核栈（8KB），用于在系统调用时、以及任务切换时使用。
  4. 拥有自己的内存空间，即页目录表的 0～3GB 的空间是用户进程使用的。

线程由什么组成

满足上面的前三条的是线程，而且线程分为内核线程和用户线程。

  • 内核线程是指只在内核空间运行的任务，其执行域以及数据域都在内核空间。
  • 用户线程是指在用户空间运行的任务（也会调用系统调用来到内核空间），一般在用户空间拥有属于它自己的用户栈。

线程、进程和任务这些名词有什么区别？

经过以上说明，已知线程和进程之间的区别是一个是没有用户空间，另外一个是拥有自己的用户空间。当他们之间都有一个性质，就是都有执行域。在 Linux ，我认为，线程和进程都被通称为任务，把他们两一视同仁，因为都是在内核内可接受的调度单位。我想，为了统一名称，不能混淆调度单位是线程还是进程，都把它们通称为任务，因为它们都拥有需要继续执行的执行域。所以对于“进程调度”这一名词，我倾向于“任务调度”这一词。

从字段 state 开始，
字段 state ：记录当前任务的运行状态，运行状态的集合如下：

  • TASK_RUNNING：它的意思并不是表示正在运行中，而是表示该任务可以允许加入到任务调度队列，等待被调度切换去执行。
  • TASK_INTERRUPTIBLE：表示任务处于睡眠状态，但是能被信号唤醒，即中断睡眠。
  • TASK_UNINTERRUPTIBLE：表示任务处于睡眠状态，但是不能被信号唤醒。
  • TASK_ZOMBIE：表示当前任务已“死”，即已调用 exit() ，但是 PCB 还存在，这个得让它所属的父进程处理。
  • TASK_STOPPED：该状态用于调试用的。

字段 flags ：记录当前任务的管理状态，比如该任务的内存、I/O以及信号等使用情况。

注意，该字段并不存储运行状态

字段 sigpending ：当其他任务对当前任务发出信号后，该任务对应的 PCB 的字段 sigpending 置 1 。最典型的例子是，该进程的 state 为 TASK_INTERRUPTIBLE 时，其他任务发出信号对该进程唤醒，该进程的 PCB 的 sigpending 被置 1 。

字段 addr_limit ：表示当前任务的可用内存空间的范围。

字段 need_resched ：当任务被认定为需要被调度时，该字段置 1 。

该字段将在 任务调度 详细说明。

字段 lock_depth ：现在还没用到，暂且忽略。

字段 counter、nice、policy、run_list、rt_priority、thread ：将在 任务调度 详细说明。

字段 mm、active_mm ：仅在 任务调度 说明一点，待实现用户进程后详细说明。

字段 pid ：每个任务都有它们自己的 ID 。

字段 next_task、prev_task ：该字段的作用是把所有的 PCB 链接在同一个链表 task_list 当中。

字段 local_pages、allocation_order、nr_local_pages：字段 local_pages 用于链接 struct page ，当系统处于内存紧缺状态，任务也需要内存，则它以同步的方式去获取内存（字段 allocation_order 记录任务所需的内存页面对应的 order ），即先执行回收不活跃页面，把空闲页对应的 struct page 链接到该字段（字段 nr_local_pages 记录 struct page 在 local_pages 的个数），而不是伙伴位图分配器的空闲链表，然后在这个字段尝试获取符合要求的页面。

详细请看 伙伴位图分配器 和 page_alloc.c

字段 start_time ：当创建任务之后，该字段记录创建的时间。

字段 pidhash_next 和 pidhash_pprev ：当我们需要根据 pid 查找任务的 PCB 时，发现根据 pid 在 task_list 一个一个的查找是不行的。所以使用 HASH 表，对 PID 散列作为 HASH 值，以空间换取时间来解决。

我们先了解一下创建线程/进程的步骤，根据上一点的进程由什么组成得知，创建进程的大致步骤为：

  1. 在内核空间获取到2个连续的内存物理页面作为它的内核栈
  2. 在内核栈的底部空出大约 1KB 的内存空间，存储 PCB ，对应的结构体为 struct task_struct 。然后对该 PCB 初始化。
  3. 在内核空间获取若干个物理内存页面（可连续也可不连续）分别作为一个页目录表以及若干个页表，概括起来就是创建自己的用户空间。
  4. 读取可执行文件，把代码段、数据段等载入到内存当中，根据可执行文件头的信息，把开始执行该进程的代码虚拟内存地址载入到它的内核栈当中，待任务调度切换后执行（创建进程是系统调用，所以要从内核层返回到用户层）。

而线程的创建步骤则是仅有第一条和第二条，待任务调度切换后，执行创建该线程的任务所指定的函数。

上面的伪代码是比较简单的，其实还有很多额外的要求。
比如，已知用户线程是“寄生”于进程的用户空间（内核线程则是内核空间），那么进程可以用的，它也可以用。比如 在进程正在打开的文件 ，某些线程需要把进程已打开文件的状态复制过来，自己单干，不用和进程“同一个饭碗”（一分两了，像 fork()）。所以有必要在函数 create_uthread() 和 create_kthread() 新添加一个形参：存储标志位的变量 。某一位置位，就表示进程的某一资源另外复制给新线程。

话都说到这里了，好像这个越来越像函数 fork() 。我们知道，fork() 的功能就是复制出一个子进程。那么 fork() 的执行步骤是什么？

  1. 在内核空间获取到2个连续的内存物理页面作为它的内核栈
  2. 在内核栈的底部空出大约 1KB 的内存空间，存储 PCB ，对应的结构体为 struct task_struct 。然后对该 PCB 初始化。
  3. 在内核空间获取若干个物理内存页面（可连续也可不连续）分别作为一个页目录表以及若干个页表，概括起来就是创建子进程的用户空间。

另外，fork() 复制一个子进程是不需要复制父进程的代码段、数据段等数据复制到子进程的用户空间，这样很浪费时间。为了能够很好的利用虚拟内存这一机制，直接把父进程的指向用户空间的页表的页表项（除了指向栈的物理地址），复制到子进程的页表当中，且都设置为仅可读。当子进程写操作时，发出内存异常，则新获取一个页，把发生异常的页的数据复制到新页，新页的物理地址写在子进程的页表内发生异常的页表项，并更新为可读可写。从异常返回后，再重新进行写操作。这就是 read-copy-write 机制。

现在我们可知，创建线程/进程的有些步骤是和函数 fork() 类似的，那么我们是否可以“求大同存小异”。“小异”取决于“标签位”，像给我在上上段那样 “create_uthread() 和 create_kthread() 新添加一个形参来存储标签位”，根据自己的需求创建/克隆任务。

创建用户进程是先用 fork() 出一个子进程，调度到子进程后，调用 execve() 。

用 fork() 创建/克隆任务的话，优点有这些：代码高复用性（这也从缓存中体现）、利于扩展、代码相对简洁易读和代码低重复率。

fork() 和 clone()

fork() 和 clone() 对应的系统调用分别是 sys_fork() 以及 sys_clone() 。其中 sys_fork() 的功能是完美无瑕地把父进程复制出子进程；sys_clone() 的功能是可以选择性的复制父进程的数据，即你设置哪个标志位，标志位对应的数据就不从父进程复制过来了，而是用指针指向它。我们可以通过 sys_clone() 来创建线程。
另外，严格地说，sys_fork() 和 sys_clone() 只能算是接口函数，因为工作的核心部分是在核心函数 do_fork() 。
