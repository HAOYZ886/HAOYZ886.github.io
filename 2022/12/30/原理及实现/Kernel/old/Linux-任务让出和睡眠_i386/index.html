Linux - 任务让出和睡眠（内核版本 2.4.22，32位x86）

Posted on 2022-12-30 In 原理及实现 Word count in article: 3.5k Reading time ≈ 13 mins.

任务让出

任务让出就是指主动让出 CPU 使用权给其他在调度队列内权值最高的任务，但注意的是执行任务让出操作的任务是还在调度队列的，但是在该队列的尾部。而且任务策略的不同，做出的让出操作也是不同的。
现在来看看在内核内的接口函数 yield() ：

最主要的操作是 sys_sched_yield()

from sched.c

通过该函数可知，如果调度队列内仅有一个任务，让出操作是无效的。如果任务策略为 SCHED_OTHER ，则该任务的策略需要被附加上 SCHED_YIELD ，这个策略标志位需要和函数 schedule() 结合说明。然后，当前任务被设置为需要调度，并加入到调度队列的尾部。

然后要看看函数 schedule() 和 yield_*() 之间的联系。

  • 假设策略为 SCHED_OTHER 的任务执行 yield 操作，并调用 schedule() 。最主要的是调度时的任务权值比较，看过函数 goodness() 知道，运行状态存在 SCHED_YIELD 的任务，对应的权值是 -1 。而一般来说任务的权值一般是大于等于 0 的，所以权值为 -1 的任务被调度的机率很小（指全调度队列仅有一个任务，且该任务的策略是 SCHED_OTHER|SCHED_YIELD ，这种任务才会被调度）。经过 switch_to 切换任务后，CPU 的执行权就交给权值最高的任务了。待切换回来后，还在函数 schedule() ，执行 __schedule_tail() ，把策略标志位 SCHED_YIELD 清除。
  • 假设策略为 SCHED_RR 或 SCHED_FIFO 的任务执行 yield 操作，并调用 schedule() 。这些任务是没有策略标志位 SCHED_YIELD 的，仅找出权值最高的任务切换即可。因为这种类型的任务时间要求高。

任务睡眠

非计时睡眠

非计时睡眠，就是某一任务同步等待某一个事件完成而陷入睡眠，当事件完成后唤醒。比如竞争互斥锁，锁被其他任务占用，则只能睡眠等待。
通用的非计时睡眠函数有：interruptible_sleep_on() 、sleep_on()

from sched.c

总的来说，因为将要睡眠的任务需要被移出调度队列，所以要事先记录该任务的 PCB 地址，即加入某个等待队列内。然后调用函数 schedule() 调度切换任务。然后通过 wake_up_*() 或 wake_up_interruptible_*() 唤醒在该队列的任务（加入到调度队列），最后返回到从等待队列移除。

计时睡眠

计时睡眠，通常用于等待某一个功能的同步，对于不同调度策略的任务，计时睡眠方式也有点不同。计时睡眠的函数有：sys_nanosleep() 、sleep_on_timeout() 、interruptible_sleep_on_timeout() 。

  • 如果计时睡眠的时间小于等于 2ms ，而且执行计时睡眠的任务的策略还是 SCHED_FIFO 或 SCHED_RR ，则执行忙等待操作。对应的函数/宏是 udelay() ，有关 udelay() 的内容在 时间延迟 。因为 SCHED_FIFO 和 SCHED_RR 的任务是有时间要求的，设置这么小的计数去睡眠，而且睡眠的计数是根据时钟中断执行的，发送一次时钟中断的间隔为 10ms ，所以让仅睡眠小于等于 2ms 的，这种类型的任务去等待每隔 10ms 发送一次时钟中断发出是不可取的。

  • 把时间计数转换为跳数的 timeout 加上当前的 jiffies ，赋值给 expire ，这个 expire 表示计时结束的位置。
  • 这里引出一个新的结构体 先暂且不了解该结构体的作用，我们先引出一个问题，就是计时睡眠是如何倒计时的，假设有很多任务进行计时睡眠，那么如何存储睡眠中的任务，倒计时结束后如何唤醒对应的任务？
    根据前文所知，我们根据时钟中断的跳数进行倒计时，即 jiffies 达到 expire 就结束。但多数任务的 expire 是不同的，所以每个任务都要存储自己的 expire 。然后呢，我们该如何存储计时睡眠的任务？如果用 HASH 表存储（ HASH 冲突用链地址法），但是不同计时的任务放在同一个 HASH 表项内的链表也不行，因为每一次时钟中断，都要去查看哪些任务计时结束，而去各个表项的链表查找哪些任务计时结束是很费时间的。如果用一个很长的循序表，一个元素对应的计时，如元素 10 对应计时睡眠为 10 的任务，这样也不行，太浪费内存空间。

存储计时睡眠任务的介质搞定了，但要如何存储？
我们根据睡眠时间转换而来的 timeout 确定在哪个数组存储，根据当前 jiffies 加上 timeout 确定任务存储在该数组内的哪个元素。

接下来讲解函数 internal_add_timer() 会明白。

看 struct timer_vec 和 struct timer_vec_root 的声明，index 的作用是什么？
已知 jiffies 是与时俱进的，index 也是一样。比如 tv1.index ，它的范围是 0～255 ，当它被累加到 256 时复位，tv2.index 加 1 ，当 tv2.index 被累加到 64 时复位，tv3.index 加 1 ，以此类推。所以字段 index 作用是在每次的时钟中断后，根据 index 在它所属的数组找出计数睡眠的任务。

先不管条件判断 if (!tv1.index) 以及在内的代码块，先看标签 repeat 以下的代码块。通过代码可知，对计时睡眠任务的处理，主要的是在 tv1 数组内。执行流程大概为：根据 tv1.index 在 tv1 数组内找出对应的元素（链表头），如果该链表存在节点，表示在该链表的任务计时睡眠已结束，则把它从链表移除，执行函数 fn(data) （实际是执行函数 process_timeout() 唤醒该任务，加入到调度队列）。从该链表遍历完成后，timer_jiffies 加 1 ，tv1.index 也加 1 。
然后看这个条件判断 if (!tv1.index) ，最典型的例子就是当内核初始化期间开启中断，jiffies 和 timer_jiffies 都为 0 。当 CPU 接收第一个时钟中断时，jiffies 加 1 。然后执行该 BH 的处理程序 run_timer_list() 时，tv1.index = 0，期间调用函数 cascade_timers()，使其 tv2～5 的 index 都为 1 ，因为 index=0 内的元素为空，结合上面给出图和解释去思考就明白了。

例子1：假设 timer->expires 为 260 ，timer_jiffies 为 20 ，那么 idx = 240 ，则把该任务加入到数组 tv1 的第 timer->expires & TVR_MASK 个元素（第四个元素）。当 tv1.index 从 20 累加到 256 复位时，期间已经经过 236 次 jiffies 了，当 index 被累加到 4 时，该任务的计时睡眠结束。

例子2：假设 timer->expires 为 300 ，timer_jiffies 为 20 ，那么 idx = 280 ，则把该任务加入到数组 tv2 的第 (expires >> TVR_BITS) & TVN_MASK 个元素（第一个元素）。当 tv2.index 为 1 时，timer_jiffies 为 256 ，而且 tv1.index 为 0 ，表示期间已经经过 236 次 jiffies 了，剩余的计时 44 ，则把它作为 tv1 的索引值，把睡眠中的任务移动到该索引值对应的元素。当 tv1.index 为 44 时，该任务的计时睡眠结束。
