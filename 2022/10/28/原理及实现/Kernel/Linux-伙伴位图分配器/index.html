伙伴位图分配器

什么是伙伴位图分配器

伙伴位图分配器是 Linux 内核从位图分配器过渡的第二款物理页面分配器。“伙伴”是指它把 2^order 个连续且相邻的内存页面作为一个块（ order >=0 & order < MAXORDER ），且把两个相邻的，页面数相同的块互相作为伙伴（一个新的整体）。而“位图”指的是一个位代表着一个伙伴（两个块）。当伙伴全为空闲或被分配时，它对应的位就被清除；当伙伴当中的其中一个块被分配，它对应的位就被置位。把 order 当做层数，每一层都代表着块是由 2^当前层层数 个连续内存页面组成。比如 order=0 ，表示第0层，表示一个块对应的连续页面数是 2^0=1 ；比如 order=1，表示第一层，表示一个块对应的连续页面数是 2^1=2 ；以此类推，直到 order=MAXORDER-1 停止。每一层都有它们自己的空闲链表还有伙伴位图，其中，空闲链表存储的是一个块，它的另一伴已被分配而“苦苦地等待它回来”的块（该块对应的位被置1）。当伙伴里面块都是空闲时，可以“往上一层楼”。意思是指把两个相邻且连续的块组成一个块，作为在上一层与它连续相邻的块作为伙伴。当我们需要在某一层获取一个块时，是优先从空闲链表获取。如果在当前层的空闲链表为空，我们就请求从上层的空闲链表获取更大的块，在高层获取到更大的块之后，把这个块按下层对应连续页面数拆分两个块，把两个块当中的右边块往下传给下一层（对应的伙伴位被置位），左边放在所处层的空闲链表，按此方式一直到在我们想要获取块的层的层数。把从上一层获取到的块被拆分成两个小块，一块放在所处层的空闲链表，另一个给需要获取该层的块的进程（对应的伙伴位被置位）。

This is an allocation scheme that combines a normal power-of-two allocator with free buffer coalescing, and the basic concept behind it is quite simple. Memory is broken up into large blocks of pages where each block is a power of two number of pages. If a block of the desired size is not available, a large block is broken up in half, and the two blocks are buddies to each other. One half is used for the allocation, and the other is free. The blocks are continuously halved as necessary until a block of the desired size is available. When a block is later freed, the buddy is examined, and the two are coalesced if it is free.

伙伴位图分配器描述符

根据上文，对应的伙伴位图分配器描述符是：

  • free_list ：连接空闲块的链表
  • map ：记录伙伴状态的位图

该描述符存储在每一个内存管理区当中。

前面提过，位图内的一位就代表着两个块（一个伙伴），那么块的起始地址（PFN）和 order 组合成什么公式才能在每一层中的位图内定位到对应的位？

分配页面

概论

分配页面之前，我们需要知道要在哪个管理区获取页面。在 UMA 系统，由于仅有一个内存节点 contig_page_data ，所以仅针对它里面的管理区。根据内存节点描述符的字段 node_zonelists[] 以及函数 build_zonelists() 可知，我们需要参数指定先从哪个管理区获取内存页面，所指定的管理区被称为 classzone ，如果 classzone 无多余的空闲页面，则退回到下一个管理区尝试获取。比如 classzone 是 ZONE_HIGHMEM，如果无多余的空闲页面，则退回到 ZONE_NORMAL ，如果还是没有，就退回到 ZONE_DMA 。在 NUMA 系统，有多个内存节点，当有一个进程申请获取内存，根据本地分配策略（解析E820图有说明），优先从执行该进程的 CPU 内核对应的内存节点，指定管理区获取内存（虽然 NUMA 也支持 Remote-Access ，但暂未详细了解该概念）。
另外，执行分配内存页面操作前，一般来说，你还需设置影响它操作的标志 GFP_* 。比如 GFP_ATOMIC ，表示执行该函数的进程：在此期间是不允许睡眠、不可被重新调度的。有关于标志 GFP_* 的详细内容在 eleos/mm.h 有详细说明，这里就不赘述了。
指定好从哪个管理区获取内存页面后，还需要执行内存页面的数量，这里一般是指定参数 order ，即指定从哪一层获取内存块。请根据小节 什么是伙伴位图分配器 以及以下的图例理解。

核心函数

函数 alloc_pages()

函数 alloc_pages() 仅做一个简单检查：查看参数 order 是否大于 MAX_ORDER 。是则返回 NULL ，否则到下一层函数 _alloc_pages() 。

函数 _alloc_pages()

函数分为两个版本，一个是 UMA 的，另一个是 NUMA 的。它们之间的区别就是，UMA 直接使用内存节点 contig_page_data 内的管理区分配内存块；NUMA 的则是获取执行该进程的 CPU 内核对应内存节点，在该内存节点内的管理区获取内存块。
由于 EleOS 内核目前仅支持 UMA ，所以仅说明 UMA 的 _alloc_pages()

函数 __alloc_pages()

函数 __alloc_pages() 是核心函数，该函数有很多详细的注释，可直接在源码页查看，这里没有必要再赘述了。

释放页面

概论

释放页面的核心函数是 __free_pages_ok() ，我们先从该函数讲起为好。当我们释放页面前，我们首先要知道它的 struct page * 或页面的起始虚拟地址、和 order 。注意 order ，该值必须和分配页面时 order 一致，否则会有大麻烦！
在对该页面释放期间，我们不会立刻把它放到 order 对应的空闲链表，而是检查它在该层数的位图的位。如果该位被清除，表示它的另一伴正在处于被使用的状态，则我们把该位置位（表示伙伴内的其中一块处于空闲状态），并把它放到空闲链表当中（头插法）；如果该位被置位，表示它的另一半处于空闲状态，则我们把该位清除，并把空闲块从所处层的空闲链表移出，它和另一半组合在一起，跳转到上一层。然后在上一层的操作和刚刚讲述的是一致的，直到另一伴还处于使用状态或达到 MAX_ORDER-1 层。

注意：MAX_ORDER-1 层仅有空闲链表，无 map 。

从位图分配器过渡到伙伴位图分配器

为什么不继续使用位图分配器？与位图分配器相比，伙伴位图分配器有哪些优势

不继续使用位图分配器，主要原因是位图分配器的分配页面机制。假设继续使用位图分配器，如果想要获取一个相对较大的连续页面，从头去遍历找出这类型的页面是需要花费很多时间的，而且还有可能获取不了，到头来一场空。毕竟执行位图分配器期间，会产生较多的内存碎片。且后续较难更新，上限较低。
伙伴位图分配器和它相比，优势有：内存页面根据 order 被组成多个块，分层管理不同大小的块，使得分配内存时效率快速。每个块有它自己的伙伴，使用位图记录每个伙伴的状态，能够组成更大的块供给上层，尽可能地使需要获取大块内存的进程获取到。

虽然伙伴位图分配器在初始化阶段较为耗时（看下节源码），但是后续使用它分配/释放页面的效率快。这是属于置换时间耗费吧。

伙伴位图分配器的缺点（内存碎片）

执行伙伴位图分配器期间还是会有内存碎片的。为了减少内存碎片，还是需要另外的操作。内存碎片被分为内部碎片和外部碎片。
内部碎片是在直接映射到内核空间的 0～896MB 内产生的内存碎片。由于是直接映射的，那么如果想要获取较大的连续页会比较困难。但这里新增了 slab 分配器来缓解内部碎片（ slab 分配器下节讲解）。
外部碎片是指在 896MB 以外产生的内存碎片。内核空间的 896MB～1GB 不是直接映射区域，可使用非连续内存分配器解决。

One important problem that must be addressed with any allocator is the problem of internal and external fragmentation. External fragmentation is the inability to service a request because the available memory exists only in small blocks. Internal fragmentation is deﬁned as the wasted space where a large block had to be assigned to service a small request. In Linux, external fragmentation is not a serious problem because large requests for contiguous pages are rare, and usually vmalloc() is suﬃcient to service the request. The lists of free blocks ensure that large blocks do not have to be split unnecessarily.
Internal fragmentation is the single-most serious failing of the binary buddy system. Although fragmentation is expected to be in the region of 28 per-cent, it has been shown that it can be in the region of 60 percent, in comparison to just 1 percent with the ﬁrst ﬁt allocator. It has also been shown that using variations of the buddy system will not help the situation signiﬁ-cantly. To address this problem, Linux uses a slab allocator to carve up pages into small blocks of memory for allocation. With this combination of allocators, the kernel can ensure that the amount of memory wasted due to internal fragmentation is kept to a minimum.
–截自《内存管理》第六章第六节
