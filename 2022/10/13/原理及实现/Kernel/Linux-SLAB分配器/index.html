Linux - SLAB分配器（内核版本 2.4.22，32位x86）

Posted on 2022-10-13 Edited on 2022-10-28 In 原理及实现 Word count in article: 9.1k Reading time ≈ 33 mins.

SLAB 分配器为什么存在

由于内核有很多模块，每个模块都有它们自己的描述符（结构体），那么怎么存储它们的描述符是一个问题。为了能够很好的利用内存，我想应该把这些描述符都集中存储到一个页面或者是多个连续的页面，每个模块都有它们自己的内存页面去分配/释放描述符。但是随着模块的增加，有了很多缺点：

  1. 过于分散的管理，即让每一个模块管理自己的分配/释放的描述符内存页面，不利于以后的更新扩展。
  2. 由于每个模块都有基于自己描述符的内存管理器，内存回收机制需要到每个模块的页面去回收，影响效率。
  3. 代码重合度高，过于臃肿。

应把它们集中管理，新增一个分配器去分配以及管理这些页面，所以， SLAB 分配器诞生了。
有了 SLAB 分配器之后，每一个模块都有自己的 CACHE ，这些 CACHE 都被连接到一个双向循环链表，这个链表主要是辅助内存回收机制回收某个 CACHE 的空闲页面，以及便于查看所有 CACHE 的状态信息。然后呢，每一个 CACHE 都有三个队列用于存储分配描述符（以后通称 OBJECT ）的页面块，分别是 slabs_full 、slabs_partail 、slabs_free 。其中，slabs_full 存储已经分配完 OBJECTS 的页面块；slabs_partail 存储正在分配 OBJECTS 的页面块；slabs_free 存储处于空闲状态的页面块。为什么把多个内存页面块分成三部分存储？其实最主要的原因是在执行回收不活跃页面时，我们也要针对某个 CACHE 回收在 slabs_free 内一部分的空闲页面。而设计 slabs_full 、slabs_partail 的目的是便于分辨，哪些页面已经被使用完了，或正在使用等等。下图是 SLAB 的结构图。

通过上图发现，页面块里面的布局分为两种，一种是仅划分 OBJ ，不存储 SLAB 描述符的；一种是即划分 OBJ ，也存储 SLAB 描述符的。有关于如何存储 SLAB 描述符的内容将在 如何存储SLAB描述符 说明。

SLAB 分配器还有哪些特性

有了 SLAB 分配器以后，我们还需要考虑一些有关于使用以及执行效率的问题：

  • 我们需要通用 OBJ 的 SLAB 分配器，目的是：有些模块有它自己的描述符，但是用的相对较少，新增一个 CACHE 反而会有浪费；或者其他需要动态分配内存的对象，比如字符串之类的；还有针对内部空间的内存碎片，当我们需要较大的连续页面时，可能因为内部空间内存碎片而导致无法很好地获取到它。针对这些问题，我们创建一系列新的 CACHE ，命名为 size-N 以及 size-N(DMA) ，其中第二个是专门给 DMA 设备的使用的。size-N 和 size-N(DMA) 当中的 “N” 的范围是 2^5（32）~ 2^32（131072）。
  • 已知一个 OBJ 的生命周期是：分配-初始化-使用-释放（见下图1）。其中“分配”和“初始化”的耗费时间是一致的，或者是后者多于前者的。所以我想，是否可以尽可能地把初始化划分为多个部分，一部分放在分配到页面统一初始化 OBJ 时执行，另一部分在某一进程分配到 OBJ 后执行，能更快地使用到 OBJ 。当然，把初始化分成若干个部分还需根据使用它本身的性质决定的。比如，OBJ 是一个描述符，它有一个字段是 struct list_head 类型的，这个可以在页面统一初始化 OBJ 时设置。还有其他的字段是当前进程执行分配 OBJ 操作后，再根据当前的参数设置（见下图2）。

  • 当已有足够的 obj 填充 slab ，但由于 obj 的大小可能会导致 slab 内存存在多余的空间。为了让剩余空间挥发余热，利用它来更好地使用 CPU 缓存行。
    怎么使用？把剩余空间划分成几个 COLOUR SPACE ，大小一般是 32 。COLOUR SPACE 的数量被 CACHE 的字段 color 存储；其大小被 CACHE 的字段 color_off 存储。在 SLAB 内，利用这些字段偏移第一个 OBJ 的起始地址。在详细说明之前，需要说明这样做的原因是什么。已知以内存地址（物理地址/虚拟地址）为基础，有很多种方式去寻找 CPU 缓存行。内存地址被分成这几部分来寻找 CPU 缓存行：tag entry 、index 、offset 。

    1. index 是作为缓存行在 CPU 缓存内的索引，如上例子的 index 为 5 ，意思是我们需要在 CPU 缓存内找出 index=5 的缓存行。
    2. 而 offset 是缓存行内偏移量，如上例子的 offset，可知单个缓存行的字节大小为 32 ，即 offset 的范围是 0～31 ，所以以上例子的 offset=4 。结合第一条可知我们需要访问 index=5 的缓存行内的第 4 个数据（字长未知，先不管）。
    3. 以上两点还不够，因为内存地址很大，而 CPU 缓存很小，内存地址的前几个位会重复的（指 index 和 offset 会重复），所以使用 tag entry 用于辨别。当使用上面两点找出缓存行之后，使用上列例子的 tag entry 对比缓存行对应的 tag entry ，如果相等则表示该 CPU 使用该内存地址命中了缓存行。
      前面刚说了，内存很大，而 CPU 缓存很小。如果单单是扩大 index ，缩小 tag entry 是不行的，所以，N 路相联缓存行出现了。

    N 路相联是什么？
    先从“ N 路”开始，假设 N=1 ，则表示 CPU 缓存内仅有一列连续的缓存行；假设 N=2，则表示内有两列连续的缓存行，以此类推。而“相联”指的是，CPU 缓存内的每一列内的每一行缓存行用 offset 相联着。比如 2路相联 ，还是以上面的例子为例，CPU 使用 index=5 ，tag entry=1011110B 在第一路命中缓存行。如果 CPU 使用和上例地址的 offset 相同的，但 tag entry 不相同的内存地址去寻找缓存行，发现第一路的已被占有了，就来到第二路，即第二列缓存行，发现 offset=5 的缓存行还是空闲的，则 CPU 使用该地址从内存读取写到该缓存行（ tag entry 也写到该缓存行对应的）。
    话说到这里了，那么 colour 和 CPU 缓存行之间有什么联系？
    当一个 slab 填充足够的 obj ，发现剩余空间后，把这个空间分成若干个小块（通常是32字节大小的，刚好对上 L1 缓存行的大小）。小块的数量被字段 colour 保存（准确来说是偏移量，因为也包含 0 ）；小块的大小被字段 colour_off 保存；小块的起始偏移量被字段 colour_next 保存。
    这里有个很好的例子：

    Let us say that s_mem (the address of the ﬁrst object) on the slab is 0 for convenience, that 96 bytes are wasted on the slab and alignment is to be at 32 bytes to the L1 Hardware Cache on a Pentium II. In this scenario, the ﬁrst slab created will have its objects start at 0. The second will start at 32, the third at 64, and the fourth at 96, and the ﬁfth will start back at 0. With this, objects from each of the slabs will not hit the same hardware cache line on the CPU. The value of colour is 3 and colour off is 32.

    这样，该功能和 CPU 缓存的 N 路相联配合。虽然效果微弱，单靠移动 slab 内第一个 obj 的起始位置是不行的。其实最主要的是一个小于等于 L1 缓存大小的 obj 不能占用两个 CPU 缓存行。存储它要两个缓存行很浪费缓存行、降低执行效率。

另外，为了能够很好的调试 SLAB 分配器，这里提供了一个宏 CONFIG_DEBUG_SLAB 。如果你定义了它，编译器在编译内核期间根据此设置展开对应的代码段，宏 SLAB_RED_ZONE 和 SLAB_POISON 被启用

  • SLAB_RED_ZONE ：把 OBJ 的前后位置扩展 4 字节，并写入对应的魔数，在分配和释放期间检查此魔数是否被破坏，目的是检查使用该 OBJ 的进程是否对它越界访问。
  • SLAB_POISON ：把 OBJ 里面的空间填充魔数，在分配期间检查此魔数是否被破坏，目的是检查该 OBJ 是否被某个进程非法访问（指未执行正确的分配操作去使用它）。

    注意：SLAB_POISON 和上面讲的第二个 SLAB 分配器的特性冲突，某个 CACHE 有了它就不能启动 SLAB_POISON 。所以启动 CONFIG_DEBUG_SLAB 前要注意各模块对各自的 CACHE 的使用要求，再确定是否对自己的 CACHE 启用 SLAB_POISON。

概论

CACHE

描述符

CACHE 的描述符在 mm/slab.c 。由于此描述符过长，所以分段讲解。

  • slabs_full 、slabs_partail 、slabs_free ：用于存储 SLAB 描述符，且这三个字段的作用已在上文说明。
  • objsize ：单个 OBJECT 的大小，该字段是由使用该 CACHE 的模块决定的。
  • flags ：存储对 slab 和 object 的 flags ，其中包含 DEBUG 、OFF_SLAB、HWCACHE_ALIGN 等标签位。
  • num ：记录单个 SLAB 内 OBJ 的最大数量，内存页面块越大，OBJ 数量越多。不管 ON_SLAB 还是 OFF_SLAB ，SLAB 的内存页面块的页面数量还得根据系统的最大可用内存来决定的。一般来说，在低于等于 32MB 的系统，页面块的连续页面数是 2 ，否则就是 4 。
  • spinlock ：保护 CACHE 描述符的自旋锁。

  • gfporder ：分配 slab 时所需要的 2^gfporder 个页。

  • gfpflags ：分配 slab 时调用 kmem_getpages()时，即调用 __get_free_pages() 时所需的 GFP_MASK 。

  • colour ：当在一个页面块内确定 OBJ 的数量后，我们还需知晓剩余空间的大小（以字节为单位），然后根据此除以 offset ，得到的结果存储到 colour 。

    offset 一般是 32 。

  • colour_off ：一个 COLOUR_SPACE 的大小（以字节为单位）。

  • colour_next ：记录下一个 SLAB 的第一个 OBJ 在页面块内偏移的位置（以 COLOUR_SPACE 的个数为单位）

  • slabp_cache 当该 CACHE 启用了 OFF_SLAB ，我们需要在 size-N CACHE 获取适合的 OBJ 存储 SLAB 描述符。而这个字段存储该 CACHE 以免有新的 SLAB 描述符时再在遍历 size-N CACHE 寻找。

  • growing ：当 CACHE 新增一个 SLAB 时，该字段将被置为 1 。待此操作完成之后，该字段将被置为 0 。目的是防止和回收不活跃操作冲突。

  • dflags ：当 CACHE 新增一个 SLAB 时，该标签的 DFLGS_GROWN 位将被置位。目的和字段 growing 类似，但这个字段是给函数 kmem_cache_reap() 使用的。当执行此函数时，会遍历所有的 CACHE ，该字段会被检查。如果 DFLGS_GROWN 被置位，则把该位清除并跳过该 CACHE 。毕竟该字段和上一个字段被置位表示 CACHE 的 OBJ 的需求是非常大的。SLAB 分配器的收缩/回收 SLAB 函数应暂且“放过”这种类型的 CACHE 。

  • ctor ：当分配完 slab 确定 obj 数量之后，调用该函数初始化 obj 。

  • dtor ：但 slab 被摧毁时调用该函数操作 obj（不常用）。

  • name ：一个 CACHE 的名字。
  • list ：用于连接前一个和后一个 CACHE
  • stats ：记录 CACHE 的使用状态
  • num_active ：记录当前 CACHE 的 OBJ 的使用个数
  • num_allocations ：记录在 CACHE 内有多少个 OBJ 被分配了
  • high_mark ：记录当前 CACHE 内最多有多少个 OBJ 被使用过。

    我感觉这个可以把这个字段移出到 cache_stats ，添加到 CACHE ，或者为 CACHE 新增一个与它相似的字段。原因是我们是否可以根据 CACHE 的 OBJ 使用需求量来决定在回收不活跃页面期间（指执行 kmem_cache_reap() ），检查 CACHE 时，通过减少 pages ，释放较少量的 SLAB 对应的空闲页面块。如果使用需求量大，但是呢它用的快，释放的也快。这种情况周而复始，所以在执行回收不活跃页面期间，检查所有的 CACHE发现，也仅有它满足释放空闲页面块的要求，但应减少对该 CACHE 的释放空闲页面块。

  • grown ：分配一个新的 SLAB 后，此字段加 1 。
  • reaped ：函数 kmem_cache_reap() 对该 CACHE 进行一次回收操作时，此字段加 1 。

标签

SLAB 标签
  • SLAB_NO_REAP ：CACHE 对应所有的 SLAB 是不能回收的。
  • SLAB_CACHE_DMA ：CACHE 的 SLAB 是所属于 DMA 管理区的。
  • CFLGS_OFF_SLAB ：SLAB 的描述符存储在 size-N CACHE 。
OBJECT 标签
  • SLAB_HWCACHE_ALIGN ：表示所有的 OBJ 要向 CPU L1 缓存行对齐。
  • SLAB_MUST_HWCACHE_ALIGN ：表示所有的 OBJ 必须要向 CPU L1 缓存行对齐。就算定义了 SLAB_RED_ZONE ，该宏对设置了 SLAB_MUST_HWCACHE_ALIGN 的 CACHE 的 OBJ 失效，因为 SLAB_MUST_HWCACHE_ALIGN 的优先级比 SLAB_RED_ZONE 的高。
DEBUG 标签
  • SLAB_DEBUG_INITIAL ：该宏是与 ctor 搭配的，如果设置了它和 ctor ，那么每次 OBJ 释放之后都会调用 ctor 检查 OBJ （ ctor 必须要有 SLAB_CTOR_VERIFY）。
  • SLAB_RED_ZONE ：该宏已在上文讲解。
  • SLAB_POISON ：该宏已在上文讲解。
动态标签
  • DFLGS_GROWN ：该宏已在上文讲解。
页面分配标签
SLAB_NOFS 对标 GFP_NOFS
SLAB_NOIO 对标 GFP_NOIO
SLAB_NOHIGHIO 对标 GFP_NOHIGHIO
SLAB_ATOMIC 对标 GFP_ATOMIC
SLAB_USER 对标 GFP_USER
SLAB_KERNEL 对标 GFP_KERNEL
SLAB_NFS 对标 GFP_NFS
SLAB_DMA 对标 GFP_DMA

针对SLAB分配器的分配/释放页面函数

当我们需要一个新的 SLAB ，即内存页面块，则必须要使用伙伴位图分配器获取内存页面。针对SLAB分配器的分配/释放页面函数分别是：

创建

创建一个新的 CACHE 的函数是 kmem_cache_create() 。由于代码量较大，所以分批讲解

kmem_cache_create() 的参数：

  • name ：CACHE 的名字
  • size ：一个 OBJ 的大小（以字节为单位）
  • offset ：设置在 CACHE 的 SLAB 内的第一个 OBJ 的起始地址，默认为 0 。
  • flags ：对 CACHE 的标签
  • ctor ：初始化 OBJ 的函数
  • dtor ：摧毁 OBJ 的函数

这段代码就是对参数进行条件判断：

  1. 如果 name 为空或者字符串过长，则执行 BUG
  2. 如果该函数在执行中断处理程序期间被调用，则是错误的，因为接下来涉及到信号量。
    说到信号量，信号量的对象就是多任务（不管是单核还是多核都一视同仁）。当一个任务获取到信号量，那么想要获取该信号量的其他任务就只能等待（同步），直到该任务释放它，使其他任务又再次竞争。由此可知，它们之间又冲突了。假设在执行 TH/BH 期间调用 kmem_cache_create()，调用它期间发现信号量已被其他任务获取了，该任务只能强制睡眠等待，你觉得 TH/BH 还没有执行完就跳转到其他任务合理吗？这是不可取的。
    由于信号量的原因该函数不能在执行中断处理程序期间调用，所以顺势而然，接下来可能涉及 gfp()s 的操作，允许含有内存回收的操作（即 GFP_MASK 不是 GFP_ATOMIC）。内存回收包含 I/O 操作，可能会有中断操作的呢，所以内存回收不允许在执行中断处理程序期间使用。
  3. 如果 OBJ 的大小是小于 4 字节或者大于 2^5 页面，则错误。
  4. 如果仅有 dtor 但没有 ctor ，则错误。
  5. offset 必须要大于等于 0 且小于等于 OBJ 的大小，否则错误

  1. 如果 flags 包含 SLAB_DEBUG_INITIAL ，但是没有 ctor ，则把在 flags 的 SLAB_DEBUG_INITIAL 被清除。因为 SLAB_DEBUG_INITIAL 是用于在 OBJ 在释放期间调用 ctor 检查它是否有问题。
  2. 如果 flags 包含 SLAB_POISON ，但是有 ctor ，则把在 flags 的 SLAB_POISON 清除。因为两者都是预先对 OBJ 写入它们的数据，前者是填充魔数检查是否非法访问，后者是对 OBJ 预先初始化。
  3. 这里尝试强制开启 SLAB_RED_ZONE ，但是也有限制：如果 OBJ 的大小大于等于 512 字节，是不允许的。因为 OBJ 太大了，相应的在内存页面块内，它的数量也会减少，所以不应该在 OBJ 原有的大小上继续扩展，不然将造成较大的空间碎片，导致浪费内存，减少了可用 OBJ 的数量。还有，如果 flags 存在 SLAB_MUST_HWCACHE_ALIGN ，说明 OBJ 的大小必须遵循对 CPU L1 缓存行的大小对齐。毕竟启用 SLAB_RED_ZONE 就代表着要对 OBJ 扩展 8 字节（ OBJ 的前后各 4 字节）。如果这两个标签都存在，可能会浪费 CPU L1 缓存行。

回收

调用函数 kmem_cache_reap() ，遍历 CACHE 链表，找出满足要求的 CACHE ，并在它的 slabs_free 内释放适量的 SLAB 。

第一个条件判断是根据 gfp_mask 是否包含 __GFP_WAIT ，而决定要以什么方式去获取信号量。如果包含 __GFP_WAIT ，则允许当前进程可以睡眠，可被重新调度，获取信号量时，如获取失败，则睡眠；如果不包含 __GFP_WAIT ，如中断处理程序，它是不能被重新调度的，获取信号量时，如获取失败，则返回 0 。

*如果 flags 包含 SLAB_NO_REAP 、字段 growing 被置 1 或字段 dflags 包含 CFLGS_GROWN ，则跳过该 CACHE ，到下一个 CACHE 。

在当前 CACHE 的 slabs_free 内计算有多少个空闲 SLAB 。

通过上段代码可知，当所有空闲 SLAB 的页面数量大于 REAP_PERFECT 时，才能执行回收一部分空闲 SLAB 。但这个页面数量会根据一些原因减少，而减少 CACHE 满足该要求的概率：

  1. 如果 CACHE 有 ctor ，则 pages = (pages*4+1) / 5 ，大概减少 pages 的三分之一。原因是使用 ctor 初始化 obj 是相对费时间的，如果回收过多，模块有需要急需使用时，可能又要重新获取新的 SLAB ，再调用 ctor 初始化 SLAB 内的所有 OBJ ，这样又太浪费时间了。
  2. 如果 CACHE 的 SLAB 对应的内存页面块的页面数量大于 1 ，则 pages = (pages*4+1) / 5 ，大概减少 pages 的三分之一。原因是不容易获取较大的连续页。

所以，结合以上两点，回收空闲 SLAB 的优先级是：（从上到下，优先级越来越高）

  1. 有 ctor 且页面块的页面数量大于 1 的 CACHE
  2. 没有 ctor 和页面块的页面数量大于 1 的 CACHE
  3. 有 ctor 和页面块的页面数量为 1 的 CACHE （和第二个的优先级相等）
  4. 没有 ctor 和页面块的页面数量为 1 的 CACHE 。

如果没有一个 CACHE 满足 if (pages >= REAP_PERFECT) ，则找出最多空闲页面的 CACHE 并回收。
