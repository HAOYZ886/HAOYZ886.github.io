通过代码可以发现，为什么 I8259A 的所有 IRQ 对应的中断都把 IRQ号-256 压栈，然后跳转到 common_interrupt ，再调用 do_IRQ() 执行？
在解答这个疑问之前，这里有一个疑问：

  • 我们知道，中断处理程序的执行时间是非常紧凑的，一个接一个地接收中断并执行它，但如果有些中断处理程序的执行时间较长，可以开启中断后再执行它，但是不可避免的就是中断嵌套，这个怎么解决？

    这个疑问是真实存在的，但更多地出现在多核的环境下，所以以下解答可以以多核的角度出发。在单核环境下，在执行中断处理程序前（在 do_IRQ() 期间）会把对应的 IRQ 屏蔽，待执行完中断处理程序再开启 IRQ 。

如果某一个中断处理程序的执行时间很长，则允许执行它之前开启中断，执行完成后关闭中断。如果开启中断后执行中断处理程序期间，接收到来自另一个 IRQ 的中断号，那么可以执行它吗？可以，只要它不与之前的中断处理程序有冲突。那么如果接收同一个 IRQ 的中断信号呢？
则采用一种“先进先出”的方法。就是我们可以采用标签置位的方法禁止同一个 IRQ 的中断嵌套。比如 IRQ0 对应的中断处理程序正在执行中，我们可以在它执行之前置位标识它，表示“IRQ0 对应的中断处理程序执行中”；比如一个 CPU 接收来自 IRQ0 的中断后想要执行对应的中断处理程序，则标识“ CPU 想要执行 IRQ0 对应的中断处理程序”；比如 IRQ0 处于关闭状态中，则标识它“ IRQ0 关闭中”（前两个是在函数 do_IRQ() 操作）。
举个例子，在执行一个允许开启中断执行的中断处理程序期间（此前在 do_IRQ() 给 IRQ 对应的 flags 已标识“中断处理程序执行中”），CPU 接收了与当前正在处理的中断处理程序相同的中断号，在进入中断执行 do_IRQ() 检查对应的标签，发现它对应的中断处理程序正在执行中，为了避免因中断嵌套导致数据被覆盖，所以它标识了“CPU 想要执行 IRQX 对应的中断处理程序”，然后就灰溜溜地离开了。待之前的中断处理程序执行完成后，它会检测“CPU 想要执行 IRQX 对应的中断处理程序”这一标志位，如果被置位，它会把该位清除，接着再执行一次中断处理程序。
但在这里又有疑问了，在多核环境下，在这种情况下，在开中断期间大于等于2次接收了来自同一个 IRQ 的中断号，前一次中断把“想执行”的标志位置位了，那么剩下的中断也把它置位了，待之前的中断处理程序完成后却仅执行了一次，那么相当于剩下的中断信号浪费了，这是为什么？
在多核环境下，主要的是避免单个 CPU 在一直走“执行中断处理程序”的独木桥。中断信号虽然重要，但也别落下“执行其他任务”的任务呀。所以在这种情况下，同一个中断处理程序仅执行两次（指的是执行完中断处理程序检查“想执行”标志位，若存在则清除该位再执行一次）。虽然丢失了中断信号，但需要了解清楚的是这个中断信号真的来的这么频繁吗？

初始化中断-软中断

软中断，不是int x、int3、into指令的软中断，也不是进程间的软中断（信号），而是对时间没有硬性要求的中断（允许开启中断），而且使用代码模拟中断信号。

为什么需要软中断？

软中断的出现的原因之一是因为中断后半部分的缺点。中断后半部分是什么，为什么存在？
有一些中断是一被 CPU 接收就要立即执行，有很大的时间限制；有些中断对应的中断处理程序的执行时间是相对较长的，虽然可以开启中断再执行它，但从总体上讲，外部中断的处理还需尽快地执行完成，至少要把关键的操作执行完成。所以，针对这两个问题，可以把中断处理程序分为前半部分和后半部分，前半部分是针对有强烈的时间限制的，需尽快执行最关键的操作；后半部分是没有强烈的时间限制，开启中断允许其他前半部分的中断处理程序执行，后半部分可以稍微晚点执行。
中断后半部分类似于 PIC 芯片的硬件机制，中断后半部分的处理程序所对应的函数被存储在一个包含32个元素的数组 bh_base 内。

from softirq.c

还有两个全局变量，变量里面的每一个 bit 表示每一个中断后半部分。第一个变量 bh_active ， 表示“想要执行”；第二变量 bh_mask ，表示“禁止执行”。当某一个中断前部分想要执行它的中断后半部分时，在 bh_active 把它对应的位置位。当执行完成中断前半部分后，回到 do_IRQ() 检查是否有中断后半部分需要执行，如果有就开启中断后开始执行。
但在这里，发现了问题：

  1. 在单核环境下，如果在执行中断后半部分期间发生中断，执行完中断前部分后要求执行后半部分，但该中断后半部分和发生中断前执行的中断后半部分是一致的，如果执行则会导致中断后半部分的嵌套，影响原子操作。在多核环境下，这种情况更严重。
  2. 而且在执行中断前半部分期间，有些中断前半部分执行之前允许开启中断后执行。如果在期间接收来自 IRQ 的中断信号，该中断执行完它的中断前半部分后也顺势而然地执行中断后半部分，你觉得这情况下还没执行完上一次的中断前半部分就执行中断后半部分，这合适吗？
    针对这些问题，第一个可以使用自旋锁进行保护，第二个用信号量保护，在执行中断后半部分前做出这些操作足矣，就这么简单。

整个题外话，个人想法，可忽略
有没有想过还有没有另一种方法是否可取，如果新添加一个全局变量 bh_running ，该变量内的位对应一个中断后半部分的函数，如果要执行了，就在执行之前在该变量的位置位，表示该中断后半部分正在执行中。这个是否可取？
不可取。单只是从单核的角度来说，如果添加这一机制的话，会有一个最坏的情况。就是先执行的中断后半部分可能要过段很慢的时间才能执行完成。因为中断后半部分可以开启中断，虽然它被它自己的锁/信号量保护，禁止嵌套执行，但是其它的中断后半部分可以执行，如果数量一多起来，那么之前的中断后半部分可能就要过很久一段时间才能执行了，有点像栈的“后进先出”，虽然中断后半部分没有太高的时间限制但也因尽早执行完成，中断后半部分之间的执行要应做到“先进先出”的原则才能保证每个都能够完成执行。

那么如果用两个保护机制禁止中断后半部分的嵌套以及中断前半部分和中断后半部分之间的执行顺序，但这样从多核的角度来看也带来了一个问题。
其中，自选锁也是一个公共资源，多 CPU 可以去竞争获取。那么也就是获取锁之后仅有一个 CPU 执行中断后半部分。虽然这样也挺好，能够防止同时执行中断后半部分可能导致公共资源的损坏，也能够从单核过渡到多核。但是这也带来了一个问题，如果出现很多需要执行的中断后半部分，只有单个 CPU 去执行，让单个 CPU 去走“独木桥”，仅有一个 CPU 在一段时间周期内执行中断后半部分，这怎么行呢？在全局的角度来说如果是这种情况带来结果就是中断后半部分不能够快速地被执行。所以在中断后半部分的基础上又添加了一些新机制，叫做软中断。

为什么软中断能够解决该机制的缺点？
正是因为每一个 CPU 都有自己当前的中断状态、每一个 CPU 都有自己的 tasklet 调度队列（ CPU 之间没有相同的 tasklet）、每一个 tasklet 都有一个状态标签变量。现在一起来看看软中断是如何实现的。

软中断的实现

软中断是使用代码“模拟”硬件中断来实现，注意这个关键字 “模拟”，软中断是怎么模拟硬件中断的呢？
先从硬件中断出发，

  1. 硬件中断的 IRQ 的个数是有限制的，所以软中断的个数设置为最多 32 个。
  2. 当通过 PIC 芯片接收中断信号，比如 IRQ0 有中断信号，PIC 会发送来自 IRQ0 的中断信号给 CPU 执行。根据前一句所说，首先 CPU 和 PIC 是并行的，即自己处理自己的事务，属于异步状态。而对于软中断的触发，我们可以沾着“外部中断的光”去触发执行。换句话说，就是当 CPU 接收来自 IRQ 的外部中断信号，待中断处理程序执行完成后顺带执行软中断的中断处理程序。但是这样还不够，我们还需知道我们执行的哪一个软中断，就是执行前需要调度，类似于某个中断后半部分执行前，通过在 bh_active 置位知道以后要执行哪个中断后半部分。
  3. 有些 IRQ 是共享的，一个 IRQ 可被多个外部设备使用。虽然我们使用软件模拟，但还是需要有一些限制。就是一个软中断对应所有的中断后半部分，一个软中断对应若干个通用的 tasklet，一个软中断对应一个网卡的软中断的操作的集合。

    PS: 这里我更多地表示一个软中断应该对应一个系列的操作的集合，又或者说对操作有更多的时间要求的集合
