函数 setup_arch()

跳转到 start_kernel() ，第一件事就是执行函数 setup_arch() ，每一个平台都有它们自己的 setup_arch()，EleOS 内核仅支持 i386 ，现说明它的 setup_arch() 的执行流程。
目前 setup_arch() 首要的任务就是先解析 E820 图：

看函数 setup_memory_region() 之前，我们需要知道 E820 图从何而来。

E820 图从何而来

前言

如何知道内存的可用容量？
在 I386 平台，在实模式期间，我们可以使用 BIOS 中断来获取内存的可用容量。调用 INT 0x15 AH=E820/E801/88 即可获取可用内存的范围。这些系统调用更多的是由外设提供，这些系统调用被映射在内存地址的640KB～1MB范围，在执行 BIOS 期间它们会被检索，即检查是否可用。EleOS 内核是被内核启动器 GURB2 Multiboot 加载，获取可用内存的这类信息已被启动器获取并传输给内核，看 Multiboot information structure 。

正文

看 multiboot ，它的某一个结构体 Multiboot information structure

注意看字段 mem_* 以及 mmap_* ，它们被设为可用是在 Multiboot header 的 flags 字段，在该字段的第1位置位表示 mem_* 和 mmap_* 可用。在 EleOS 内核，该位是必须要设置的，且已在 head.S 设置

现在来看看 mem_* 字段：

  1. mem_lower ：记录 0～1MB 内存的可用范围，它记录的方式是：可用内存的结束地址/1024 。它可记录最大的内存范围是 640 ，即 640*1024=640KB ，表示 0～640KB 为内存的可用范围。
  2. mem_upper ：记录 1MB～4GB 的可用内存范围，记录方式和上一个类似：(可用内存的结束地址-1MB)/1024 。例如它需要记录 1MB～64MB 的内存范围，则存储：(64MB-1MB)/1024=63KB 。

再到 mmap_* 字段

  1. mmap_length ：记录由 multiboot 创建的 E820 图的总字节大小。注意！是整个 E820 图的以字节为单位的总大小，而不是 E820 图内的项的个数。

  2. mmap_addr ：这是指向 E820 图的指针，先来看看 E820 图内的个项结构体：

    • size 记录当前项的字节大小（固定为20，没什么作用）
    • base_addr 记录起始地址
    • length 记录范围大小（以字节为单位）
    • type 记录内存空间的类型

    Multiboot 就是使用该类型的结构体创建 E820 图，但这个结构体还是有不足之处。所以在把 Multiboot information structure 复制到 empty_zero_page 期间（看函数 handle_multiboot_info()），把该结构体转换为以下结构体：


    转换完成后，就把转换完成的 E820 图复制到 empty_zero_page，setup_arch() 就可使用该 E820 图去解析。

现在已知 E820 图从何而来，就算没有 E820 图，我们也可以根据字段 mem_* 自己创建 E820 图。如果 multiboot 调用 BIOS INT 0x15 AH=E820 不成功（主要原因是 BIOS 中断不支持该功能，没有相应的功能号），它也许通过调用 BIOS INT 0x15 AH=E801/88 获取内存可用信息创建 E820 图（至少字段 mem_* 是有的）。

内存管理区

接下来就是重头戏了，我们调用函数 setup_memory() 使用调整后的 E820 图找出最大的可用内存范围，便于划分内存管理区 ZONE_DMA 、ZONE_NORMAL 、ZONE_HIGHMEM 的范围，然后初始化引导内存分配器。

首先我们需要知道内存管理区是什么。内存管理区被分为三种，它们各自的作用是：

  1. 第一种管理区 ZONE_DMA 是用于满足 ISA 设备（Industry Standard Architecture devices），有些设备是不支持分页机制，多数是访问低端内存。所以划出 0～16MB 作为 ZONE_DMA 。
  2. 第二种管理区 ZONE_NORMAL 是直接映射到内核空间的区域，划出 16MB～896MB 作为 ZONE_NORMAL 。

    直接映射：物理地址 + 3GB

  3. 第三种管理区 ZONE_HIGHMEM 是包含 896MB 以后的内存区域。

为什么需要管理区 ZONE_HIGHMEM ？
在内核空间中，已有 0～896MB 被直接映射，剩下 128MB 是用于能够使用 896MB 以后的内存页面。

函数 setup_memory() 的执行步骤为：

  1. 调用函数 find_max_pfn() 找出可用内存的结束地址（以页为单位）。
  2. 调用函数 find_max_low_pfn() 划分出高端内存（如果有，且定义了 CONFIG_HIGHMEM）。
  3. 为引导内存分配器分配位图（一位对应一个页），并填充不可用页对应的位。

引导内存分配器

现在到初始化引导内存分配器，引导内存分配器就是位图内存位图分配器，以位图当中的一位当作一个页，按位分配内存页面。
它的优势有：

  1. 初始化位图分配器简单且快速，这对于内核初始化阶段的执行效率要求很重要。而且过渡到伙伴分配器前需要很多的内存空间，这些内存空间用于 struct page 和伙伴分配器描述符，而且这些内存空间也需要使用其他内存分配器分配。
  2. 便于设置保留的内存页（外设在内存地址的映射，在内存的内核映像，内核页表，位图等）。
  3. 在使用引导内存分配器期间，因为分配到的页面是不会释放的（极少概率），那么产生内存碎片的概率也极少。

什么是内存节点？

解释什么是内存节点之前，先需要知道什么是 UMA ，什么是 NUMA 。UMA 全称是 Uniform Memory Access ，译为一致内存访问。它的意思是在多核连接着一条通向内存的数据总线和一条地址总线，每个处理器核心共享相同的内存地址空间。但随着 CPU 内核数量的增加，有很多缺陷：

  1. 由于仅有一条总线（地址/数据），有多个内核在同一时间周期访问内存会导致拥堵。
  2. 如果多个内核同时访问同一个内存地址，一般需要前缀指令 LOCK 确保数据一致性。一旦执行 LOCK 指令，CPU 内核会发出一个 #LOCK 信号，使得其他 CPU 内核总线接口被锁住，仅有执行发出该信号的 CPU 独享内存。但由于在 UMA ，总线的吞吐量巨大，所以因为 #LOCK 信号也会导致其他内核 CPU 的效率降低。

带着这个问题，NUMA 出现了。NUMA 的全称是 Non-Uniform Memory Access ，译为非一致性内存访问。即一整块可用内存被分为多个簇，一个簇对应一个内存节点，一个内存节点对应一个或多个 CPU 内核。通俗点来说，就是主板的一个/多个内存条插槽对应一个内存节点。有了这个，缓解了 UMA 的问题。
有了 NUMA ，线程更多地是在执行它的 CPU 内核的所在的内存节点分配/释放内存页面。如果本地内存节点没有足够的空闲页面让线程去获取，但是其他内存节点有足够的空闲页面，如果单靠把不活跃的页面换出到硬盘当中是不行，所以可以在其他内存节点去获取内存。这里访问本地内存节点的策略叫做 Local-Access ，跨内存节点访问的策略叫做 Remote-Access ，执行效率前者比后者快。

所以由此可知，内存节点代表着一个内存空间，有一组 CPU 内核属于同一个内存节点，也有另一组 CPU 内核属于另一个内存节点。

linux 2.4 的引导内存分配器的改进

已知由于NUMA的机制，内存被分为多个簇，一个簇对应一个内存节点。简单点来说，主板的一条内存插槽对应一个内存节点。一个内存节点对应一个或多个 CPU 内核。在linux 2.4，每一个内存节点描述符都有一个存储引导内存分配器描述符的字段。我们知道，内存分配器指向的内存范围是前 896MB 。如果一个内存节点对应 64MB 内存空间，那么内存节点的引导内存分配器描述符字段都是利用得当的；如果对应 1GB ，那么该字段是很浪费的，应该把这个字段移出到内存节点描述符。根据单个内存节点的内存容量以及系统的最大可用内存，创建引导内存分配器描述符数组。
已知内存被分为三大管理区，其中管理区 zone_dma 和 zone_normal 内的内存页面是引导内存分配器的被分配对象。已知内存节点最多包含三个管理区，在 NUMA ，一个节点可能就仅有一到两个管理区，那么是有 zone_dma 和 zone_normal ，那么就有 zone_normal 和 zone_highmem ，又或者说就仅有三个管理区当中的其中一个（多数是 zone_highmem）。而且从分配内存的使用习惯来说，想要在哪里获取到内存页面更多的是在以0为起始以及以16MB为起始地址。那么从内存管理区的角度来说，这不就是以管理区 zone_dma 或 zone_normal 为起始获取页面吗？那么可以根据内存节点以及管理区把引导内存分配器描述符分成多个，与上一段所说的引导内存分配器描述符数组结合。然后根据上面说的第三点，把引导内存分配描述符中记录最近分配页面的位置的字段的功能扩展起来，即获取页面时，就在它所记录的位置的下一个页面获取。这样就不用在以0或16MB等为起始，在位图从头检索为0的位去获取对应的内存页面，从而加快分配效率。
比如内存节点0记录0-512MB的内存空间，存在两个管理区，分别是zone_dma和zone_normal；内存节点1记录512-1024MB的内存空间，存在两个管理区，分别是zone_normal和zone_highmem。那么引导内存分配器将存在三个元素，前两个元素是属于内存节点0的，即第一个元素是内存节点0内的 zone_dma 的，第二个是它的 zone_normal 的。而第三个元素是属于内存节点1的 zone_normal 。
那么在某个内存节点分配页面时如何在该数组获得对应的元素？
根据公式：zone编号+nid，比如获取在nid为0的内存节点的 zone_dma 的页面，则对应元素索引就是0；如果是获取同一内存节点的 zone_normal 的页面，则是1；如果是获取nid为1的内存节点，则是2。
