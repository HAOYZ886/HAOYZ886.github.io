伙伴位图分配器

什么是伙伴位图分配器

伙伴位图分配器是 Linux 内核从位图分配器过渡的第二款物理页面分配器。“伙伴”是指它把 2^order 个连续且相邻的内存页面作为一个块（ order >=0 & order < MAXORDER ），且把两个相邻的，页面数相同的块互相作为伙伴（一个新的整体）。而“位图”指的是一个位代表着一个伙伴（两个块）。当伙伴全为空闲或被分配时，它对应的位就被清除；当伙伴当中的其中一个块被分配，它对应的位就被置位。把 order 当做层数，每一层都代表着块是由 2^当前层层数 个连续内存页面组成。比如 order=0 ，表示第0层，表示一个块对应的连续页面数是 2^0=1 ；比如 order=1，表示第一层，表示一个块对应的连续页面数是 2^1=2 ；以此类推，直到 order=MAXORDER-1 停止。每一层都有它们自己的空闲链表还有伙伴位图，其中，空闲链表存储的是一个块，它的另一伴已被分配而“苦苦地等待它回来”的块（该块对应的位被置1）。当伙伴里面块都是空闲时，可以“往上一层楼”。意思是指把两个相邻且连续的块组成一个块，作为在上一层与它连续相邻的块作为伙伴。当我们需要在某一层获取一个块时，是优先从空闲链表获取。如果在当前层的空闲链表为空，我们就请求从上层的空闲链表获取更大的块，在高层获取到更大的块之后，把这个块按下层对应连续页面数拆分两个块，把两个块当中的右边块往下传给下一层（对应的伙伴位被置位），左边放在所处层的空闲链表，按此方式一直到在我们想要获取块的层的层数。把从上一层获取到的块被拆分成两个小块，一块放在所处层的空闲链表，另一个给需要获取该层的块的进程（对应的伙伴位被置位）。

SLAB 分配器为什么存在

由于内核有很多模块，每个模块都有它们自己的描述符（结构体），那么怎么存储它们的描述符是一个问题。为了能够很好的利用内存，我想应该把这些描述符都集中存储到一个页面或者是多个连续的页面，每个模块都有它们自己的内存页面去分配/释放描述符。但是随着模块的增加，有了很多缺点：

  1. 过于分散的管理，即让每一个模块管理自己的分配/释放的描述符内存页面，不利于以后的更新扩展。
  2. 由于每个模块都有基于自己描述符的内存管理器，内存回收机制需要到每个模块的页面去回收，影响效率。
  3. 代码重合度高，过于臃肿。
