Linux - PCI总线框架（内核版本 2.4.22，32位x86）

Posted on 2023-05-15 Edited on 2023-10-06 In 原理及实现 Word count in article: 30k Reading time ≈ 1:50

前言

从本篇文档开始，从底层出发，复现 Linux 2.4 的设备驱动，其中最主要的驱动是：PCI 驱动、IDE 硬盘驱动和块设备驱动。然后，往上复现，即复现虚拟文件系统、EXT2 文件系统、设备文件系统（如果有充裕的时间的话），接着再到用户进程、进程间通信、内存回收、内存交换、OOM。现实中，硬件设备众多，工业级别的操作系统都会有相应的驱动去识别并管理连接到电脑的外部设备。比如现在将要说明的 PCI 驱动，只要连接到 PCI 总线的外部设备，在 PCI 总线遍历搜索，它们统统都会被识别出来，然后分配内存空间去描述它们。下面开始说明 PCI 驱动。

关于 PCI 的说明

什么是 PCI 总线？

PCI 总线，是一条外设总线，是主流的标准总线（目前主流的是 PCIe 总线，仍延续使用 PCI 的标准，自下往上兼容）。它也是一条扩展性强、灵活性强的外设主线。在 EleOS 复现的 PCI 驱动是基于 PCI 32bit 标准的总线，因为 PCIe 总线的内容繁多，不太适合用于学习。PCI 驱动很适合学习如何识别并管理外设。至于其它被 PCI 总线淘汰的外设总线，如：ISA、EISA 等等，感兴趣的，请自寻查阅。

PCI 总线连接在哪里？

PCI 总线的接口是如何连接外设的？

目前，PCI 总线已成为一种标准的外设总线，现已有很多硬件厂商使用该总线标准去设计它们自己的硬件设备。PCI 总线的每一个连接外设的接口的作用都是一致的，而且一条 PCI 总线的接口是有限的。那么有一个疑问：

如果外设的数量超过 PCI 总线的接口数量，如何扩展接口数量？

我们需要搞一些概念，其实 PCI 设备也是分类型的，一是普通的 PCI 设备，二是桥接总线设备。普通的 PCI 设备无非就是 IDE 硬盘、键盘等等，而桥接总线设备也是外设总线，这种总线称之为 “PCI-to-PCI” 桥。把桥接设备接上上层的 PCI 总线，这就意味着可连接 PCI 设备的接口又多好几个。但是，在下层的 PCI 设备的传输速率可能会有所下降（层次问题），这种就像房子那样，分一环、二环、三环······ 见下图：

在 PCI 总线搜索外设期间，外设又是如何被识别出来的，又或者说是如何知道接口有外设连接上？

这里不得不提一个概念了，PCI 总线的接口内配有一组寄存器，称之为 “配置寄存器组” 。顾名思义，该寄存器组就是记录外设的配置信息，比如：厂商 ID 、设备 ID 、设备类型和初始化信息等等，而且这些信息是由 BIOS 加载到配置寄存器组内。在 EleOS 的 PCI 驱动中，我们是通过识别厂商 ID 是否存在来确定接口是否被外设接上。
配置寄存器组内的寄存器一般是以 4 字节为单位的，读出与写入必须是以 4 字节为单位，这是 PCI 总线标准的要求。早期的 PCI 总线，就是 EleOS 的 PCI 驱动对应的总线，配置寄存器组的大小是 256 字节（当然，目前主流的 PCIe 的配置寄存器组更大）。其中，前 64 字节的配置寄存器组的用途和格式是标准的，被称为配置寄存器组的 “头部” 。至于后面的配置寄存器组，它是可变长的，由硬件厂商设置的。在配置寄存器组头部内，前 16 字节的寄存器组的用途和格式是一致的；后 48 字节的寄存器组，因 PCI 设备类型的不同（桥接设备与常规设备）而不同。其中，常规设备的头部表示为：type 0；桥接设备的头部表示为：type 1。前 16 字节的寄存器组存储的是厂商 ID 、设备 ID 、头部类型、设备类型等等······ 我们通过宏定义来看前 16 字节的寄存器组的布局：

PCI_CLASS_DEVICE 用于表示设备类型，比如 legacy 模式的 IDE 硬盘，它对应的设备类型是 0x0101 。其中，高 8 位表示设备类型是什么（硬盘、网卡、键盘等等），低 8 位表示设备当前默认的模式是什么，又或者说是继续细化设备的类型，比如说 IDE ，IDE 不只是支持磁盘、还支持软盘，SCSI 磁盘等等，这些存储设备也有对应的类型编号。而 PCI_CLASS_PROG 表示设备操作的方式是什么，这个是根据设备类型去决定的。

知道每个 PCI 接口都有一个配置寄存器后，我们该知道如何访问 PCI 设备的配置寄存器组了。在 i386 平台，I/O 端口地址的 0x3F8 ~ 0x3FF 是提供给 PCI 总线使用的。其中，以 0x3F8 为起始，范围为 4 字节，是用作为地址寄存器，0x3FC-0x3FF 是数据寄存器。先来看看地址寄存器的布局：

地址寄存器的低 8 位是作为在配置寄存器组的索引。接着到位 8 - 15 ，这里要引出一个概念，一个 PCI 接口对应 8 个功能号，这就表示我们需要对某个硬件设备操作时，需要细化到它的功能。硬件内有多个功能是很常见的，尤其是那些集成设备。在后面讲解的 PCI 驱动中，我们是把硬件的功能看作是一个逻辑设备，目的是便于管理，便于使用。看位 11 - 15 ，设备号表示 PCI 接口，表示着一个物理设备。所以 设备号 + 功能号 是用于定位逻辑设备。然后看位 16 - 23 ，这个是用于定位 PCI 总线，第一个主 PCI 总线（也是我们第一个识别到的）的编号必须为 0 。每个桥接设备都有一个自己独立的 PCI 总线编号，假设在遍历主总线时发现一个编号为 1 的桥接设备，我们可使用该编号去定位该总线，然后去遍历搜索连接它的设备。

CPU 如何访问 PCI 设备，它又是怎么发中断信号给 CPU ？（逻辑设备）

CPU 如何访问 PCI 设备？

我们详细地讲述下图的硬件系统结构，已知 CPU 是通过系统总线访问内存的，那么 CPU 又是如何访问 PCI 设备呢？注意看上图的虚线，这条虚线的一端连接着系统总线，另一端连接着主 PCI 总线，CPU 可以通过该虚线去访问 PCI 设备。那为什么是用虚线连接表示，而不是用实线？其实这里有它的含义，且听我慢慢道来。

已知硬件设备众多，而且 PCI 总线已是一种标准化的外设总线了，这必然会有众多的硬件厂商去使用这套标准去设计它们的硬件设备。而且在众多 CPU 平台中，比如 i386 平台，我们可以使用 16 位的 I/O 端口地址，通过 in 和 out 指令去访问外部设备。但是，由于 16 位 I/O 端口地址过于小了，再加上外部硬件众多，16 位 I/O 端口地址已经不够用了。所以，根据这种趋势，目前主流的是通过内存地址映射，CPU 可直接使用访问内存的指令，如 mov 指令，去访问外部设备的寄存器组空间。

目前 PCI 总线仍然兼容 I/O 端口地址，允许 CPU 使用 I/O 端口地址访问 PCI 设备。

那么 PCI 设备的地址映射如何开启？

在配置寄存器组内有一个叫作 command 的寄存器，内有开启地址映射的位，起初它是默认关闭的（被清除为 0 ），当我们需要使用设备的地址映射时，把它置位即可。所以回顾上图，就发现为什么连接着 PCI 总线和系统总线的线是虚线了吧？当某一个 PCI 设备开启内存地址映射，该虚线为 “实” ，关闭时，该虚线为 “空” 。

题外话：其实从整个硬件系统来说，我倾向把这里的“内存地址”称为“总线地址”，因为外设的地址映射未必就和内存条的内存地址重合，它们都是独立的个体，所以经过系统总线的地址，我更倾向于称它为“总线地址”。

那么又来了一个问题， PCI 设备的地址映射是从哪来的？

PCI 设备的配置寄存器组内存在一个存储地址映射的寄存器组，它可存储地址的类型为：总线地址、I/O 端口地址和可预取地址（这一段都统称它们为地址）。地址寄存器组内的地址是硬件厂商在设计硬件时烧录的，当 PCI 设备插入 PCI 总线时，在 BIOS 运行时，会把该烧录的地址写入到配置寄存器组内的地址寄存器组内。在 PCI 驱动初始化阶段，在 PCI 总线遍历搜索 PCI 设备时，把 PCI 设备的地址映射从配置寄存器组的地址寄存器组读出。

可预取地址在后面讲解。

那么这里引出一个问题，如果 PCI 设备的地址映射与其它的 PCI 设备的冲突了怎么办？冲突又是如何被检测出来的？

其实 PCI 设计者也考虑到了这一点，毕竟硬件众多，在连接众多个设备时，设备与设备之间的地址映射冲突是不可避免的。所以，添加了一个特性：地址寄存器组可写。当地址寄存器的数值更新后，PCI 总线会根据该数据更改对应的 PCI 设备的地址映射。但为了确认 PCI 总线是否真的把该数值写入到寄存器，我们还需要手动读一次该寄存器确保是否已更新。如果发现读出来的数值和之前的一样，则表示该寄存器是不可写的，该地址映射不能被更改，这种情况操作系统应该反馈相应的错误信息，而且还需根据实际情况去解决该问题。

注意：地址寄存器是否可写也是由硬件厂商来决定的，又或者因为硬件设计的缺陷导致无法写入，但总得来说地址寄存器可写还得通过 PCI 总线以及 PCI 设备的支持才行。

另外，结合上文所说，其实可以把地址映射冲突的设备关闭地址映射，这样就不会有冲突。但是我认为这是一种很极端的方式，毕竟每个设备都是平等的（除非存在某一个领域需要更具实际情况对设备排序优先级）。在代码的实现中，EleOS 都是默认把所有的 PCI 设备的地址映射读出，构成地址资源树，如果存在地址映射冲突，则尝试为它寻找空闲的地址映射。

PCI 设备是如何发中断给 CPU ？

在 PCI 总线内有 4 条中断请求线：INTA ~ INTD 。要求有中断请求的单功能设备（一个 PCI 接口仅有一个功能号可用）必须连接 INTA；多功能设备在连接 INTA 之余，可以连接其它的中断请求线。在中断请求线的另一端，连接着中断请求路径互连器，这是关键部件，它的另一端连接着可编程中断控制器 PIC 。
在 PCI 设备的配置寄存器组内记录着该设备的 IRQ 号和中断请求线，但是可能因为一些原因，比如 PIC 的型号，PIC 是旧式的，等等，导致该 IRQ 号不可用。有些中断请求互连器是可编程的，即可在互连器为该 PCI 设备设置可用的 IRQ 号，也可以让互连器为该设备获得一个可用的 IRQ 号。这些功能的设置过程因互连器的不同而不同。当为设备设置好 IRQ 号后，就可以使用了。当设备发出中断信号后，经途中断请求线到互连器，互连器会根据是谁发出的信号，把信号转发到在 PIC 对应的 IRQ ，最后，PIC 会把从 IRQ 来的中断信号发送给 CPU 。

中断请求互连器最好和 APIC 搭配，因为 APIC 支持更多的 IRQ 号。

此处的 PCI 设备指的是 物理设备 。

那么在 PCI 驱动初始化时，如何知道设备连接在哪条中断请求线？
在逻辑设备的配置寄存器组（头部类型为 0 ）内有一个寄存器，存储着中断请求线号，表示它连接在哪一条中断请求线上。
而且，在 BIOS-ROM 的地址映射内，即 0xF0000 ~ 0xFFFFF 内，存在一个中断路径请求表。在表内，以逻辑设备的个数为单位，存储着逻辑设备的信息，其中在表项内存储着以中断请求线号为索引的表，表内存储 IRQ 、MARK FOR IRQ 信息。在以后的讲解中，会详细说明中断路径请求表。

由于核数的增加，那么对系统总线的竞争（包括主 PCI 总线）也愈加激烈。在 PCI 总线的竞争无非就是多个 CPU 内核同一时段去访问 PCI 设备，为了消除这一竞争，PCI 的设计者为 PCI 增加了一个仲裁器，把主动发出读写请求的设备称之为 “主设备” ，且 “主设备” 只能有一个，其余设备则称为 “从设备” 。比如，内存、IDE 硬盘等等都是从设备，而可以变为主设备的有 CPU 、DMA 等等。在 SMP 系统中，如果多个设备（不局限于 CPU ）同时竞争同一个 PCI 设备，通过 HOST-BUS 桥时，会通过仲裁器选择这些设备其中之一作为 “主设备”，然后再允许主设备去访问从设备。但是至于仲裁器选择主设备的策略是什么，由于本人学识短浅，还尚未清楚，还需查看 PCI 手册了解。

如果连接 PCI 总线的多个设备读写操作频繁可能会造成总线竞争频繁，除了仲裁器，PCI 有哪些特性可以缓解这一问题？

我们对设备的读写操作下手，PCI 总线增加了两个特性。当发生在 PCI 总线的总线竞争而触发仲裁器时，仲裁器选择出 “主设备” 之后，其余待成为 “主设备” 的设备该怎么办？难道要苦苦地死等吗？这是不现实的。PCI 总线提供了一个缓冲区，可以把待写入的数据写入到缓冲区内，达到 N+1 流水线的操作。至于读操作的话，PCI 总线的缓冲区不给读操作的使用的，但是允许 PCI 设备拥有 “可预读” 空间。可预读地址记录在配置寄存器组的地址寄存器组内，而且必须在总线地址内。可在预读地址内提前获取数据，从而达到 N+1 流水线的操作。但是预取数据需要注意：获取数据后寄存器内数据保持不变。比如读取设备的寄存器后，寄存器内的值被初始化为 0 ，或者是队列寄存器，当读取数据后，寄存器内的值被更新为下一个节点的数值。这类寄存器不适宜预取，当然设计硬件的设计者也知道这点，不会把这类寄存器被设置为 “可预期” 。

PCI 总线描述符：

  • node: 用于连接在同一层次的 PCI 总线。
  • parent: 指向上一层的 PCI 总线。
  • children: 链接下一层的 PCI 总线。
  • devices: 链接在该总线的 PCI 设备（ pci_dev 描述符）
  • self: 指向该总线对应的桥接设备描述符 pci_dev ，主总线不需要。
  • resource: 注意，它是指针数组，它用于记录 PCI 总线的过滤区间（又称为可用区间，在讲解对读取过滤区间的操作时再详细说明）。
  • sysdata: 该字段给特定平台使用。
  • number: 当前 PCI 总线的编号
  • primary: 当前 PCI 总线的上一层总线编号。如果当前 pci_bus 描述的是主总线，则该字段、字段 number、字段 secondary 一致。
  • secondary: 如果 pci_bus 描述的是 PCI-to-PCI 桥，该字段有效，用于记录当前总线的编号，而且该字段和字段 number 一致。
  • subordinate: 记录当前 PCI 总线的子 PCI 树的最大总线编号，即当前 PCI 总线的下层总线的最大编号。
  • name: 总线的名字
  • vender: 总线（桥接设备）的厂商 ID
  • device: 总线（桥接设备）的设备 ID

PCI 设备描述符：

  • global_list: 用于连接 pci_devices 链表，该链表的作用是便于遍历所有的 PCI 设备。

  • bus_list: 把当前 PCI 逻辑设备链接在它所属的 PCI 总线（链表）。

  • bus: 指向当前 PCI 逻辑设备所属的 PCI 总线

  • subordinate: 如果当前 PCI 逻辑设备是桥接设备，则该字段指向它对应的 pci_bus 描述符

  • sysdata: 该字段给特定的平台使用

  • devfn: 记录当前逻辑设备在它所属总线的逻辑设备号（接口号+功能号）

  • vendor: 当前设备的厂商 ID

  • device: 当前设备的设备 ID

  • subsystem_vendor: 记录当前设备所属的硬件子系统（集成设备）的厂商 ID

  • subsystem_device: 记录当前设备所属的硬件子系统（集成设备）的设备 ID

  • class: 记录当前设备的种类

  • hdr_type: 记录当前设备的 PCI 配置寄存器组的头部类型

  • rom_base_reg: 记录存储该设备的 ROM 信息的寄存器在配置寄存器组内的哪里

  • driver: 记录该设备的驱动（操作函数集，包含：初始化、删除、修改···）

  • driver_data: 给字段 driver 使用，就是给该设备的操作函数集的额外参数

  • dma_mask: DMA 掩码，表示当前设备支持在该 DMA 掩码内，使用 DMA 通过内存（总线）地址映射访问该设备的可访问寄存器组。

    DMA 也必须支持在该掩码对应的字长内访问地址，即双方都必须支持在该掩码内访问。

  • irq: 该设备的 IRQ 号

  • resource: 内有 12 个元素，其中第 0 ~ 6 个用于存储该设备的地址映射；第 7 ~ 10 个，如果当前设备是桥接设备，则用于存储 PCI-to-PCI 桥的过滤区间（和 pci_bus 的字段 resource 对应，pci_bus 的存储它的元素地址），常规设备用不上；第 11 个仅用于填充，字节对齐。

  • dma_resource 和 irq_resource: 略

  • name: 设备名称

  • slot_name: 该设备在它所属的总线的编号，格式：总线编号:接口编号.功能号 ，例子：0:1.1

  • transparent: 如果该设备是桥接设备，该字段有效，表示继用（ “继承” 的 “继” ）上层总线的过滤区间

pci_dev 描述符的多数字段都是使用设备的配置寄存器组内的数据填充，在下面的代码的讲解中，会注重说明如何把设备的配置寄存器组内的数据读出，并在系统内记录这些信息。

对 PCI 接口的配置寄存器组的访问

访问 PCI 接口的配置寄存器组大致分为两种方式：

  1. 使用 BIOS 提供的接口。这个不推荐，扩展性低，而且不能直观地理解如何访问寄存器组。

在地址寄存器内获取地址映射的标准是：

  1. 获取地址映射的起始地址
  2. 发送 0xFFFFFFFF 给存储地址映射的寄存器，使该寄存器的值变换为地址映射的大小
  3. 从该寄存器读取地址映射的大小
  4. 把这寄存器变回为之前的数值（地址映射的起始地址）

进行完以上操作后，我们需要对地址映射的大小进行调整，为什么需要调整？因为从寄存器读出的地址映射的大小，并不是以单纯的数值去表示的，这个数值有比较多的含义。比如我们从寄存器读出的地址映射的大小是 0xFFFF0000 ，它并不表示该地址映射的大小是 0xFFFF0000 ，其实正确的是最小位的 1 ，即 0x10000 才是地址映射的大小。那么，剩余的高位的作用是什么？它的作用更多的是表示地址对齐方式。

在多数情况下，以最低位开始往上的所有位都被置位的。
