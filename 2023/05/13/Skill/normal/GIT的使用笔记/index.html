打补丁

先检查patch文件：git apply –stat newpatch.patch
检查能否应用成功：git apply –check newpatch.patch
打补丁：git am –signoff < newpatch.patch

git stash

把工作区的内容暂存到另外的空间，可以通过 git stash list 查看，并且可以对该暂存空间进行描述，需要添加可选参数 save 。通过添加额外参数 pop 把该暂存空间回迁到当前工作区。
使用场景：比如说软件分为发行版和测试版，各为一个分支，而且测试版的分支总是最新的。如果在测试版本分支添加新功能，但是此时发行版本有bug，则需要从测试版本分支切换到发行版本修bug。但在切换之前需要把测试版本的工作区内容上传到暂存区，但是这些代码是不完善的，直接提交是不合理的，所以使用给git stash 额外划分一个暂存区存储工作区，然后在切换到发布版本分支修bug。但修改完成后，回到测试版本分支，然后可以通过 git stash list 查看暂存区空间，再执行 git stash pop 回迁。

git 分支的练习

先说结论：
图中的各分支“各尽其职”，且每到一个阶段就合并，保证软件开发流上的完整性。后说过程：先从分支种类说起：master 分支为主干；hotfixes分支为紧急修复；release分支为发行版分支；develop为针对于开发者的开发分支；feature分支有两个子分支，分别是开发下版本功能的分支和未来版本功能的分支。
然后从时间线上说起：
在master分支中，发布标签 0.1，表示这个软件的初始版本。从 master 分支 fork 出一个 develop 分支维护功能。然后从 develop 分支中 fork 出 feature 分支，开发新功能。如果出现 bug ，则把 master 分支分裂出 hotfixes 分支，用于查找bug并修复。修复完成后，再把 hotfixes 的内容合并到 master分支和develop分支。当下个版本的功能开发完成后，把 next release分支与 develop 分支合并，并产生新的分支 release ，表示发布新版本。如果在 release 分支发现 bug，则修复并与 develop 分支合并。而且待 release 分支完善后，与 master 分支合并。
