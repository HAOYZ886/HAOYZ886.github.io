文件是什么

文件是一个记录信息的载体，是有次序的、有组织的。它可以被写，也可以被读。人把大脑的想法写到文件去，人亦可阅读文件的信息输入到大脑内。对于的计算机的文件，应从狭义和广义上说。从狭义上说，电脑文件指的是存储在磁盘内的文件。从广义上说，世间万物都是文件，这是因为凡是满足文件的特性，就是能存储信息，且可被读写的事物。比如说，网卡是一个设备文件，它可以“产出”信息，即接收来自其它终端的数据，亦可暂存信息，也可“消耗”信息，即把数据输出到相应进程。所以说它是一个文件也不为过。我们把它从一个设备抽象成一个文件，能够更方便地使用它，能够有组织、有次序的访问它里面的数据（这个体现在文件系统）。

文件管理

上面说了文件是有次序有组织的，那么管理多个文件也是有次序、有组织的，需要使用文件夹去整理文件。但一个文件夹是不够的，需要多个文件夹，但多个文件夹都怎么管理？这就需要通过一对多的方式，多个文件夹联系起来。然后，文件夹也是文件，需要存储文件夹内的文件基本信息，如：文件名、文件类型、inode 号等，这些把它们构成一个结构体，名为目录项。那么在磁盘内的文件系统，这些文件（包括文件夹），都是这样存储的：

知道文件是如何存储在磁盘内的文件系统，那么把它们读出到内存有该怎么管理？如果只是单纯地把超级块、inode 和数据块读出是不行的，因为它们仅是存储的形式，把文件读出到内存更多的是注重对文件的操作，就是需要把磁盘内的数据结构转换成便于给内核使用和管理的数据结构。在某个文件操作中，在此之前必须根据给定的文件路径定位该文件并打开它。举个例子，如：“dir1/dir2/file1” ，从左到右逐步定位到达目的地 “file1” ，这些操作和目录文件的目录项离不开关系，不如我们直接把目录文件内的目录项读出组成一个整个系统可用的目录树？见下图：

此图是由上图的结构转换而来的

由图可知，此结构仍是一对多关系，即树型结构，所以 dentry 必须用字段 d_parent 、d_subdirs 、d_child 记录 dentry 之间的关系。还有，一般来说，每个文件都有它们自己的 inode ，所以还需要在 dentry 添加一个字段 d_inode ，用于指向 inode 结构体。
见上图发现我们好像是把所有的 dentry 读出到内存来，这只是便于理解而已。在现实中，一个目录内的 dentry 是等它被需要的时候才被读出并转换，最后才被接入到 dentry 树，这是因为内存容量总是比外存小。而且，当我们需要读出一个 dentry 时，并不是只从外存读出一个 dentry ，而是把该 dentry 所属的数据块从外存读出到内存，然后用链表等其它数据结构将它连接，最后从该数据块读出想要的 dentry 。往后，如果需要读出一个 dentry ，就在内存的数据块看看是否有，没有则再从外存读一个数据块到内存并连接，而这种操作是 “预读 dentry” 。这些已链接的数据块，我们把它当作是 dentry 的“中转站”。如果这些数据块不被使用了，它们就会被释放或把数据写回外存。
另外，如果一个 dentry 也不怎么被使用了，像数据块一样，也需要把它释放。在 dentry 内添加字段：d_lru ，用于链接 dentry lru 链表；d_count，记录当前 dentry 的引用次数，在使用它时需 +1 ，不被使用时 -1 。如果等于 0 ，则表示内核、用户进程近段时间都没使用它，则把它加入到 dentry lru 链表。

上面说的情况只是基于一个文件系统的，但是现在有很多类型的文件系统，那么一个系统又该如何管理多个文件系统？换句话说，如果另外接入了一个磁盘，磁盘内的分区的文件系统和当前内核的根设备的文件系统是不一致的该怎么办？想要访问该分区内的文件系统内的文件又该怎么做？
在 linux ，使用虚拟文件系统去管理多个文件系统，它有统一的标准和接口。它就像一个主板的一个插槽，规定了很多标准，而文件系统驱动就是一个接口，必须要求符合虚拟文件系统的制定的标准才能够接入使用。下图是大致的操作流程：

那么 vfs 提供的插槽有哪些？有什么规范？
在往后的会发现，有 dentry_operations 、inode_operations 、files_operations 、super_operations 等。其中，dentry_operations 是可选的，vfs 有默认对 dentry 的操作函数集；目录是必须要有 inode_operations；files_operations 是目录和常规文件必有的；

在 linux ，上图给出的 dentry 树是 vfs 的，但是 dentry 又是从下层文件系统读出并转换得来的。举个例子，比如说根设备的文件系统是 ext2 ，且把该文件系统的根目录作为整个系统的根目录，即 vfs 的 dentry 树的根 dentry 就是它。那么如果接入了一个磁盘，磁盘内的主分区1有一个文件系统 NTFS ，且想要访问该文件系统内的文件，该怎么办？在 linux 中的做法是在 VFS 支持 NTFS ，且 NTFS 驱动支持 VFS 标准的前提下，把 NTFS 的超级块读出，并把 NTFS 的根目录接上 dentry 树的某一个 dentry ，即挂载到一个空目录内，就像这样：

在内核初始化期间，存储内核映像的设备会被当作根设备，而根设备内的文件系统会被当作整个系统的下层文件系统，换句话说就是内核的 vfs 的操作必须靠底层的文件系统支撑，就像上文说的主板插槽和接口。然后，根设备的文件系统的根目录是作为整个系统的根目录。

图中的 mnt_mountpoint 是 NTFS 在 dentry 树的挂载点，而 mnt_root 是 NTFS 的根目录。mnt_mountpoint 和 mnt_root 是属于 vfs_mount 的字段。图中的发现它们连接在一起，其实不然，这仅是便于理解才这样画的。
vfs_mount 存在的原因是为了方便管理，目的就是把挂载这个功能和 dentry 树分开，把与挂载的相关函数和功能分来，达到高内聚低耦合。此外 vfs_mount 之间也用 mnt_parent 、mnt_child、mnt_mounts 联系在一起，构成一对多关系，即树形结构。vfs_mount 还有指向超级块的字段 mnt_sb，便于在对文件操作时使用（这点在使用 struct nameidata 和会提现到，还有 dentry 也会用到超级块 d_sb ）。另外， vfs_mount 还有一个字段 mnt_count 用于记录使用次数，和 dentry 的 d_count 类似。还有，dentry 还有一个字段 d_mounted ，如果大于 0 ，则表示它已作为某个下层文件系统的挂载点。

还有，需要知道在一个目录内根据文件名定位到文件。这个看起来很简单，不就是把目录项对应 dentry 的目录项都遍历不就好了。可以，但是效率极低，对应的时间复杂度为 O(n) 。虽说这仅仅是在目录内的一次遍历而已，但是用户给出的文件路径不单是一个目录，而是一至多个目录需要逐步定位，而且还没算上动态链接文件呢，所以说，遍历的次数一多起来，对应的时间复杂度就越高。打开一个文件，通常来说是从外存读出 inode 等相关数据，还有数据块等，这些 I/O 操作是很费时间的（虽然说有 buffer_head 这个中转站来缓解该情况）。而且，能否打开文件是取决于用户提供的文件路径进行定位，在定位的过程中有较大概率也进行 I/O 来读目录项。所以说，使用文件路径到目标节点是一个很消耗时间，且很频繁的操作。再从算法的角度去看，解决一个特定的问题所对应的算法，都有它的平均情况和最坏情况（预期最小的情况），那么对于“遍历目录项”的最坏情况其实并不好。所以在一个目录根据文件名定位一个目录项，使用 hash 表 + 链表的方式（目前最新的是 hash 表+红黑树）去定位。dentry hash 表是全局的，一个 dentry 树对应一个 dentry hash 表。那么一个 dentry 项是如何 hash() 加入到 dentry hash 表？是当前 dentry 的 d_parent 的内存地址 + 文件名进行 hash 。原因是在文件系统内，在一个目录内不能出现重名文件，但在不同的目录可以，而 hash 表是全局的，所以需要加上一个额外的，且极少可能重复的值来生成 hash 值，目的是减少 hash 冲突，而 d_parent 的内存地址是极好的 hash 原值。
总之，当 dentry 树的所有 dentry 都加入到 dentry hash 表后，在进行使用文件路径定位时，直接通过 hash() 找到对应的 dentry 。

虽说在定位过程中如果出现 hash 冲突还是需要在该 hash 表的对应元素的链表进行遍历，这是没办法避免的，现阶段是在初始化期间通过系统内存的大小适当地扩大 hash 表，还有链表换成红黑树去缓解，恰当地提高效率。

还有，dentry 内还应记录该对应文件名的 hash 值以及文件名长度，目的是防止在 hash 表的链表进行遍历时因检查不全面误把错的 dentry 当成想要的 dentry 。所以找出一个正确的 dentry ，需要检查 parent 、文件名对比、长度对比、hash 值对比才行。

dcache.h

在使用文件路径到目标节点时，还需要确认 dentry 是否是挂载点，上面说过了 dentry 的字段 d_mounted 来表示它是否是挂载点，但是挂载点和文件系统的根目录之间并不是直接连接一起的，而是分别作为 vfs_mount 的字段 mnt_pointmount 和 mnt_root 。那么现在的问题就是如何根据挂载点找出对应的 vfs_mount 。
已知系统有 vfs_mount 树，且 vfs_mount 内有相关的字段去表示 vfs_mount 之间的关系，但是找出相关的 vfs_mount 并不依靠这些字段。和 dentry 一样， vfs_mount 也有一个全局的 hash 表，生成 hash 值的原值是：mnt_parent 的内存地址 + 挂载点的 dentry 的内存地址。使用该 hash 值把 vfs_mount 加入到 hash 表，并找到它（根据上图去理解）。

从文件路径到目标节点

文件路径分绝对路径和相对路径，其中绝对路径是从根目录开始的。但并不是所有的进程都可以访问系统根目录，因为有些经途根目录的数据文件是很重要的，比如 /etc/passwd 。所以说，每个任务都有自己的，或者被系统设置为属于它们自己的根目录，比如说把用户任务 usr1 的根目录设置为 /home 。还有相对路径，是以用户进程的 pwd 为起始，如 /home/usr1 。所以，进程的 PCB 必须要记录这些字段（包括 root dentry 和 pwd dentry 所在的 vfs_mount ）。
当我们正在根据文件路径到目标节点时，我们是否需要一个类似于指针一样的东西去定位目前使用该路径已到什么地方？在 dentry 树内的哪一个 dentry ？答案是需要的，不止需要记录 dentry 和 vfs_mount ，还需要根据用户要求的条件去定位。

fs.h

  • LOOKUP_FOLLOW：表示如果目标节点是链接文件，则继续链接
  • LOOKUP_DIRECTORY：目标节点必须是目录
  • LOOKUP_POSITIVE：目标节点必须有 inode
  • LOOKUP_PARENT：目标节点的上层节点，即 d_parent 必须存在。如果 flags 包含该 flag ，则目标节点是不被定位的，止步到目标节点的 parent 节点就结束，并根据目标节点的名字确定目标节点是什么类型并返回，见下面讲解。
    • LAST_NORM：拥有常规名字的目标节点，即常规文件、目录文件、链接文件
    • LAST_ROOT：根目录文件，在定位前的初始化时默认设置的
    • LAST_DOT：对应的名字是 “.” ，即当前目录
    • LAST_DOTDOT：对应的名字是 “..” ，即上一层目录
